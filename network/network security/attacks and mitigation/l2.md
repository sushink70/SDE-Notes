# 1. Foundational Concepts

## 1.1 The OSI Model — Where is Layer 2?

The OSI (Open Systems Interconnection) model is a conceptual framework that describes how
data travels from one computer to another across a network. It has 7 layers, each with a
specific responsibility.

**Think of it like mailing a letter:**
- Layer 7 (Application): You write the letter (the content/data).
- Layer 4 (Transport): You put it in an envelope with a tracking number (TCP/UDP).
- Layer 3 (Network): You write a destination city and zip code (IP address).
- **Layer 2 (Data Link): You write the street address on the envelope (MAC address).**
- Layer 1 (Physical): The postal truck physically carries it (cables, signals).

```
+---------------------------+
|  Layer 7 - Application    |  HTTP, FTP, DNS, SMTP
+---------------------------+
|  Layer 6 - Presentation   |  TLS/SSL, Encoding
+---------------------------+
|  Layer 5 - Session        |  Session management
+---------------------------+
|  Layer 4 - Transport      |  TCP, UDP (ports)
+---------------------------+
|  Layer 3 - Network        |  IP, ICMP, routing
+---------------------------+
|  Layer 2 - Data Link      |  Ethernet, MAC, ARP, STP  <-- OUR FOCUS
+---------------------------+
|  Layer 1 - Physical       |  Cables, signals, voltage
+---------------------------+
```

**Layer 2 is responsible for:**
- Framing raw bits into structured packets (called "frames")
- Addressing devices on the SAME local network using MAC addresses
- Error detection within a single link
- Controlling access to the shared medium (CSMA/CD)
- Switching frames between devices on the same LAN segment

**Critical insight:** Layer 2 operates within a single broadcast domain. Layer 3 (routing)
is needed to cross between networks. This means L2 attacks are LOCAL — the attacker must
be on the same network segment (or gain access to one).

## 1.2 Ethernet Frame Anatomy

Before understanding attacks, you must know what is being attacked. An Ethernet frame is
the fundamental unit of data at Layer 2.

```
Ethernet II Frame Structure
============================================================
Bytes:  |  6  |  6  |  2  | 46 - 1500  |  4  |
        +-----+-----+-----+------------+-----+
        | DST | SRC | ETH |   PAYLOAD  | FCS |
        | MAC | MAC |TYPE |   (DATA)   | CRC |
        +-----+-----+-----+------------+-----+
          ^     ^     ^         ^          ^
          |     |     |         |          |
          |     |     |         |          +-- Frame Check Sequence
          |     |     |         |              (error detection, 4 bytes)
          |     |     |         +-- Actual data (IP packet lives here)
          |     |     +-- EtherType (0x0800=IPv4, 0x0806=ARP, 0x8100=VLAN)
          |     +-- Source MAC Address (who sent this)
          +-- Destination MAC Address (who should receive this)

Total frame size: 64 to 1518 bytes (without VLAN tag)
```

**Key fields for security:**

| Field | Size | Significance for Attacks |
|---|---|---|
| Destination MAC | 6 bytes | Can be spoofed — no authentication |
| Source MAC | 6 bytes | Can be spoofed — no authentication |
| EtherType | 2 bytes | 0x8100 = VLAN tag (used in VLAN attacks) |
| FCS/CRC | 4 bytes | Detects corruption, NOT tampering |

**No authentication exists at Layer 2 by default.** Any device can put any MAC address
in the source field. This is the root cause of most L2 attacks.

## 1.3 MAC Addresses Explained

**MAC = Media Access Control**

A MAC address is a 48-bit (6-byte) hardware identifier assigned to a Network Interface
Card (NIC). It is used to uniquely identify a device within a local network.

```
MAC Address Structure (48 bits = 6 bytes)
==========================================
  OUI (Organizationally Unique Identifier)     NIC-specific portion
  |___________________________|               |___________________________|
  |                           |               |                           |
  XX : XX : XX  :  XX : XX : XX
  
  Byte 1  Byte 2  Byte 3    Byte 4  Byte 5  Byte 6
  
  Example: 00:1A:2B:3C:4D:5E
           ^^^^^^^^^         ^^^^^^^^^^^^^^^^
           OUI = 00:1A:2B    Device-unique = 3C:4D:5E
           (assigned to      (assigned by manufacturer)
            manufacturer)

Special MAC Addresses:
  FF:FF:FF:FF:FF:FF = Broadcast (everyone receives this)
  01:xx:xx:xx:xx:xx = Multicast (starts with 01 in first byte)
  xx:xx:xx:xx:xx:xx where bit 1 of byte 1 is 0 = Unicast

Bit flags in the first byte:
  Bit 0 (LSB): Multicast bit (0=unicast, 1=multicast)
  Bit 1:       Locally administered bit (0=globally unique, 1=locally set)
```

**Why does this matter for security?**
- MAC addresses are **not authenticated** — any OS can set any MAC it wants.
- Manufacturers hardcode MACs, but software can override them (MAC spoofing).
- The locally administered bit (bit 1) can reveal if a MAC has been spoofed.

## 1.4 Switch Architecture Internals

A **switch** is the central device of a Layer 2 network. Unlike a hub (which broadcasts
everything to all ports), a switch intelligently forwards frames only to the correct port.

### The CAM Table (Content Addressable Memory)

The switch's brain is the **CAM table** (also called the MAC address table or forwarding table).

```
CAM Table Structure:
===========================================
| MAC Address       | Port  | VLAN | Age  |
|-------------------|-------|------|------|
| 00:1A:2B:3C:4D:5E | Gi0/1 |  10  | 120s |
| AA:BB:CC:DD:EE:FF | Gi0/2 |  10  |  45s |
| 11:22:33:44:55:66 | Gi0/3 |  20  | 300s |
===========================================
Default aging time: 300 seconds (5 minutes)
```

### How a Switch Learns and Forwards

```
SWITCH DECISION FLOW — For Every Incoming Frame
================================================

Frame arrives on Port X
        |
        v
+-------------------+
| Learn Source MAC  | --> Add/Update: {Source MAC -> Port X, VLAN, timestamp}
| in CAM table      |     in CAM table
+-------------------+
        |
        v
+-------------------------------+
| Look up Destination MAC       |
| in CAM table                  |
+-------------------------------+
        |
   +----+----+
   |         |
FOUND      NOT FOUND
   |         |
   v         v
Forward    FLOOD to all
to specific ports in same
port only   VLAN (except
           incoming port)
```

**"Flooding"** = sending a frame out ALL ports in the VLAN. This is the fallback when
the switch doesn't know where the destination is. Flooding is normal behavior but can
be weaponized (MAC flooding attack).

### Switch Port States

```
Port Access vs Trunk:
=====================
Access Port:                    Trunk Port:
  - Belongs to 1 VLAN             - Carries multiple VLANs
  - Strips VLAN tags              - Adds/reads 802.1Q tags
  - Connects to end devices       - Connects to other switches/routers
  
  [PC]---[Access Port Gi0/1]     [Switch]---[Trunk Port Gi0/24]---[Switch]
           VLAN 10 only                       VLANs 10, 20, 30, etc.
```

## 1.5 VLANs — Virtual Segmentation

**VLAN = Virtual Local Area Network**

A VLAN is a logical grouping of network devices that behaves as if they are on the same
physical switch, even if they are not. VLANs provide segmentation and isolation.

**Analogy:** Imagine one physical office building. VLANs are like invisible walls that
separate departments. HR can't talk to Engineering, even though they share the same
physical cables and switches.

```
Without VLANs:                    With VLANs:
======================            ======================
[PC1] [PC2] [PC3] [PC4]          [PC1] [PC2] [PC3] [PC4]
  |     |     |     |              |     |     |     |
  +--[SWITCH]--+                   +--[SWITCH]--+
  All devices can                  PC1,PC2 = VLAN 10 (Finance)
  communicate freely               PC3,PC4 = VLAN 20 (Engineering)
                                   Finance CANNOT reach Engineering
                                   without going through a router
```

### 802.1Q VLAN Tagging

When frames traverse a trunk link, a 4-byte tag is inserted into the Ethernet frame:

```
802.1Q Tagged Frame:
===============================================================
| DST MAC | SRC MAC | 0x8100 | TCI     | ETH TYPE | PAYLOAD |
|  6 bytes|  6 bytes| 2 bytes| 2 bytes | 2 bytes  |         |
===============================================================
                       ^          ^
                       |          |
              Tag Protocol    Tag Control Information:
              Identifier      Bits [0-2]:  Priority (PCP) - QoS
              0x8100 =        Bit  [3]:    Drop Eligible (DEI)
              "This is a      Bits [4-15]: VLAN ID (VID) 0-4095
              VLAN frame"     
              
VLAN ID range: 0-4095
  VLAN 0:    Reserved
  VLAN 1:    Default VLAN (native VLAN on Cisco)
  VLAN 4095: Reserved
  Usable:    VLANs 1-4094
```

**Native VLAN:** On a trunk port, frames belonging to the "native VLAN" are sent
**without** a VLAN tag. This is a critical concept for double-tagging attacks.

## 1.6 ARP — The Address Resolution Protocol

**ARP = Address Resolution Protocol (operates at L2/L3 boundary)**

**The problem ARP solves:** IP addresses are used for routing (Layer 3). But to actually
send a frame on Ethernet (Layer 2), you need a MAC address. ARP bridges this gap.

**Analogy:** You know someone's apartment number (IP = 192.168.1.10) but need to knock
on their physical door (MAC address). ARP is like asking "Who lives at apartment 10?"
and they shout back "I do! My name is 00:1A:2B:3C:4D:5E."

```
ARP Flow:
=========
PC-A wants to talk to PC-B (192.168.1.10)
PC-A doesn't know PC-B's MAC address

Step 1: ARP Request (Broadcast)
--------------------------------
PC-A broadcasts to FF:FF:FF:FF:FF:FF:
"Who has IP 192.168.1.10? Tell 192.168.1.1 (PC-A)"
ALL devices on the LAN receive this.

Step 2: ARP Reply (Unicast)
----------------------------
PC-B (who has IP 192.168.1.10) responds:
"I have 192.168.1.10. My MAC is AA:BB:CC:DD:EE:FF"
This is sent directly back to PC-A.

Step 3: PC-A caches the result
--------------------------------
PC-A stores: 192.168.1.10 -> AA:BB:CC:DD:EE:FF
in its ARP cache (typically 20 minutes)

ARP Packet Format:
==================
+------------------+---------+---------+-------+-------+
| Hardware Type    | Protocol| HW Len  |Prot   | Opcode|
| (1=Ethernet)     | (0x0800)| (6=MAC) |Len(4) | (1=Req|
|                  |         |         |       |  2=Rep)|
+------------------+---------+---------+-------+-------+
| Sender Hardware Address (MAC) | Sender Protocol Addr  |
| 6 bytes                       | (IP) 4 bytes          |
+------------------+-----------------------------------+
| Target Hardware Address (MAC) | Target Protocol Addr  |
| 6 bytes (zeroes in request)   | (IP) 4 bytes          |
+-------------------------------+-----------------------+

CRITICAL FLAW: ARP has NO authentication. 
Anyone can send ARP replies claiming any IP-to-MAC mapping.
The receiver will trust and cache whatever it receives.
```

### ARP Cache

Every operating system maintains an ARP cache — a temporary table of IP→MAC mappings.

```
Linux ARP Cache:
$ arp -n
Address          HWtype  HWaddress           Flags Iface
192.168.1.1      ether   00:1A:2B:3C:4D:5E   C     eth0
192.168.1.10     ether   AA:BB:CC:DD:EE:FF   C     eth0

Flags: C = Cached, M = Manually added, P = Published

ARP Cache Timing:
  Linux:   Stale after 60s, removed after ~180s
  Windows: Entry expires after 2 minutes (default)
  macOS:   Entry expires after 20 minutes (default)
```

## 1.7 STP — The Spanning Tree Protocol

**The Problem STP Solves:**

In redundant network topologies, you have multiple paths between switches. Without STP,
frames would loop forever, consuming all bandwidth (a broadcast storm).

```
Network With Redundant Links — Loop Problem:
=============================================
    [Switch A]
   /           \
  /             \
[Switch B]---[Switch C]

A broadcast frame from PC on Switch-B:
1. Goes to Switch A and Switch C (from B)
2. Switch A sends it to Switch C, Switch C sends it to Switch A
3. Switch C sends it back to Switch B, Switch A sends to Switch B
4. INFINITE LOOP — network collapses in seconds
```

**STP's solution:** Mathematically compute which links to block so that only ONE active
path exists between any two switches, eliminating loops.

### Key STP Concepts

**Bridge ID (BID):** Every switch has a unique ID = Priority (2 bytes) + MAC (6 bytes).
Default priority = 32768.

**Root Bridge:** The switch with the LOWEST Bridge ID becomes the "root" — the center
of the spanning tree topology. All other switches calculate their path to the root.

**Port States:**

```
STP Port State Machine:
========================
  DISABLED
      |
      v
  BLOCKING      <- Does NOT forward frames, DOES listen for BPDUs
      |             (BPDUs = Bridge Protocol Data Units, STP control messages)
      v
  LISTENING     <- Does NOT forward frames, participates in STP election
      |             (15 seconds default)
      v
  LEARNING      <- Does NOT forward frames, DOES learn MAC addresses
      |             (15 seconds default)
      v
  FORWARDING    <- Forwards frames normally
      
Total convergence time = 30-50 seconds (major security/availability risk)
```

**BPDU (Bridge Protocol Data Unit):** The control message STP uses to elect the root
bridge and communicate topology information.

```
BPDU Key Fields:
================
+------------------+------------------+------------------+
| Root Bridge ID   | Path Cost to Root| Sender Bridge ID |
| Priority+MAC     | (cumulative)     | Priority+MAC     |
+------------------+------------------+------------------+
| Port ID | Message Age | Max Age | Hello Time | Fwd Del|
+---------+-------------+---------+------------+--------+

Root election rule:
  Lowest Priority wins -> if tie, lowest MAC wins
  Default priority = 32768
  Attacker can send priority = 0 -> becomes root immediately
```

## 1.8 DHCP — Dynamic Host Configuration Protocol

**DHCP** automatically assigns IP addresses to devices on a network.

```
DHCP DORA Process (normal operation):
======================================
Client                              DHCP Server
  |                                      |
  |--[DISCOVER]-broadcast--------------> |  "Anyone, give me an IP!"
  |                                      |
  | <-----------[OFFER]-unicast/bcast-- |  "Take 192.168.1.100, here's config"
  |                                      |
  |--[REQUEST]-broadcast--------------> |  "Yes, I'll take 192.168.1.100"
  |                                      |
  | <----------[ACKNOWLEDGE]----------- |  "Confirmed. Lease time: 8 hours"
  |                                      |
  
DHCP Lease contains:
  - IP Address
  - Subnet Mask
  - Default Gateway     <- CRITICAL: tells client where to route traffic
  - DNS Server          <- CRITICAL: controls name resolution
  - Lease Duration
  
CRITICAL FLAW: DHCP has NO authentication.
Any device can respond to DISCOVER messages and claim to be a server.
The client accepts the FIRST offer it receives.
```

# 3. MAC-Based Attacks

## 3.1 MAC Flooding / CAM Table Overflow

### Concept

**The attack exploits the finite size of the CAM table.**

A switch's CAM table has limited memory. On many enterprise switches, this is typically
8,000 to 128,000 entries. When the CAM table fills up, the switch **can no longer learn
new MAC-to-port mappings.**

When this happens, the switch has no choice but to **flood all frames to all ports** —
it reverts to hub-like behavior. Now, an attacker on one port can capture all traffic
from all other ports on the same VLAN.

### Attack Flow

```
MAC FLOODING ATTACK - Step by Step
====================================

NORMAL STATE:
  CAM Table: [MAC-A -> Port 1], [MAC-B -> Port 2], [MAC-C -> Port 3]
  Frames between A and B: Switch forwards ONLY to Port 2 (unicast)
  
ATTACK BEGINS (Attacker on Port 4):
  Attacker sends millions of frames with RANDOM fake source MACs:
  
  Frame 1:  Source=AA:11:11:11:11:01 -> Port 4 (learned!)
  Frame 2:  Source=AA:11:11:11:11:02 -> Port 4 (learned!)
  Frame 3:  Source=AA:11:11:11:11:03 -> Port 4 (learned!)
  ...
  Frame N:  Source=AA:11:11:11:11:NN -> Port 4 (table FULL!)

AFTER TABLE OVERFLOW:
  CAM Table: FULL with fake MACs (all pointing to Port 4)
  Legitimate MACs A, B, C may be evicted (aged out or overwritten)
  
  Now when PC-A sends to PC-B:
  Switch looks up MAC-B -> NOT FOUND in CAM table
  Switch FLOODS to ALL ports including Port 4 (attacker!)
  
  Attacker captures:
  - Credentials
  - Session tokens
  - Sensitive data
  - All unencrypted traffic

IMPACT DIAGRAM:
===============
  [PC-A] ----> Switch floods ----> [PC-B] (intended)
               |                    [PC-C] (unintended)
               |                    [PC-D] (unintended)
               +------------------->[ATTACKER] CAPTURES TRAFFIC!

Tools Used:
  - macof (part of dsniff suite): generates 155,000 random MAC frames/minute
  - Scapy (Python): craft custom frames
  
  macof command: macof -i eth0
```

### CAM Table Overflow Decision Tree

```
CAM TABLE OVERFLOW DECISION TREE
==================================
Packet arrives at switch
         |
         v
   Is Source MAC        YES ->  Update existing entry
   already in table? ------->   timestamp/port, continue
         |
         NO
         v
   Is CAM table          YES -> DROP the new MAC entry
     full?          -------->   (or evict oldest entry, vendor-specific)
         |                      Switch enters FLOOD mode for unknown DST
         NO
         v
   Add Source MAC
   to CAM table
         |
         v
   Forward frame
   (unicast or flood
    per normal logic)
```

### Mitigation: Port Security

**Port Security** limits the number of MAC addresses that can be learned on a port.

```
PORT SECURITY CONFIGURATION (Cisco IOS):
=========================================

! Enable port security on an access port
interface GigabitEthernet0/1
  switchport mode access
  switchport access vlan 10
  switchport port-security                        ! Enable port security
  switchport port-security maximum 2              ! Max 2 MACs allowed
  switchport port-security mac-address sticky     ! Auto-learn and save MACs
  switchport port-security violation restrict     ! Action on violation

Violation Modes:
================
+----------------+------------------+------------------+------------------+
| Mode           | Drop Frame?      | Increment        | Send Syslog?     |
|                |                  | Violation Count? |                  |
+----------------+------------------+------------------+------------------+
| protect        | YES              | NO               | NO               |
| restrict       | YES              | YES              | YES              |
| shutdown       | YES (port down!) | YES              | YES + SNMP trap  |
+----------------+------------------+------------------+------------------+

"shutdown" is the MOST SECURE but causes downtime.
"restrict" is common in production — drops and logs without disrupting.

Sticky MAC: The switch learns the first N MACs and saves them to running config.
            Even after reboot, those MACs remain authorized.
            
Verification:
  show port-security interface GigabitEthernet0/1
  show port-security address
```

### Additional Mitigations

```
ADDITIONAL MAC FLOODING MITIGATIONS:
======================================

1. 802.1X Port Authentication
   - Devices must authenticate before ANY frames are processed
   - Unauthenticated ports only allow EAP/802.1X frames
   
2. VLAN Segmentation
   - Even if one VLAN is flooded, traffic from other VLANs is isolated
   - Attacker only sees traffic within their VLAN
   
3. Private VLANs (PVLANs)
   - Ports in the same VLAN cannot talk directly to each other
   - All traffic must go through a promiscuous port (e.g., router/firewall)
   
4. CAM Table Size Awareness
   - Understand your switch's CAM table capacity
   - Monitor SNMP for sudden CAM table growth
   
5. Network Monitoring / IDS
   - Detect unusually high rates of new MAC learning events
   - Alert on MAC learning rate thresholds
```

## 3.2 MAC Spoofing / MAC Cloning

### Concept

**MAC spoofing = changing your NIC's reported MAC address to impersonate another device.**

Since MAC addresses are stored in software-configurable registers (not burned-in hardware
as was once the case), any operating system can change its MAC address trivially.

**Why would an attacker spoof a MAC?**
1. Bypass MAC-based access control (port security with static MACs)
2. Impersonate a trusted device to receive its traffic
3. Evade MAC-based tracking / logging
4. Bypass network admission control (NAC)
5. Obtain an IP address assigned to another device (DHCP bypass)

### Attack Flow

```
MAC SPOOFING ATTACK - Scenario: Bypass Port Security
======================================================

SETUP:
  Switch Port Gi0/1: only allows MAC 00:11:22:33:44:55 (IT Admin laptop)
  Attacker's MAC:    AA:BB:CC:DD:EE:FF

ATTACK:
  Step 1: Reconnaissance
          Attacker sniffs network, finds allowed MAC: 00:11:22:33:44:55
          (Can use passive sniffing, or observe ARP traffic)
          
  Step 2: Spoof the MAC
          Linux: ip link set eth0 down
                 ip link set eth0 address 00:11:22:33:44:55
                 ip link set eth0 up
          
          macOS: sudo ifconfig en0 ether 00:11:22:33:44:55
          
          Windows: Device Manager -> NIC Properties -> Locally Administered MAC
          
  Step 3: Connect and gain access
          Switch sees MAC 00:11:22:33:44:55 -> allowed!
          Attacker gains network access

MAC SPOOFING FOR DHCP BYPASS:
==============================
  DHCP servers often track IP assignments by MAC address.
  If attacker spoofs a MAC of a known device:
  -> DHCP server may assign that device's reserved IP
  -> Attacker gets trusted IP and bypasses IP-based ACLs
  
DETECTION HINT:
  Look at the OUI (first 3 bytes of MAC).
  Spoofed MACs often have the "locally administered" bit set
  (bit 1 of byte 1 = 1, making first byte odd: 02:xx, 06:xx, 0A:xx, 0E:xx).
  
  Example: 02:11:22:33:44:55 <- locally administered bit is SET (suspicious!)
```

### Mitigation

```
MAC SPOOFING MITIGATIONS:
===========================

1. 802.1X with EAP (Extensible Authentication Protocol)
   - Authenticate the USER/DEVICE, not just the MAC
   - Even if attacker spoofs MAC, they cannot pass EAP authentication
   - MAC is irrelevant when strong auth is in place
   
2. Dynamic ARP Inspection (DAI) [Details in ARP section]
   - DAI validates IP-MAC bindings using DHCP snooping database
   - If a device claims to be a MAC that doesn't match its expected IP, DROP
   
3. DHCP Snooping [Details in DHCP section]
   - Builds a trusted table of {MAC, IP, Port, VLAN}
   - Rogue DHCP responses are blocked
   
4. Endpoint Detection and Response (EDR)
   - Detect when a workstation changes its MAC address
   
5. NAC (Network Admission Control)
   - Combine MAC + certificate + user credentials for full identity
   
6. Monitor for "locally administered" MACs
   - Alert when a device with locally-administered MAC bit set appears
   - This is a strong indicator of spoofing
   
7. Sticky port security with MAC violation shutdown
   - If a port suddenly sees a new MAC after learning the sticky one, shut it
```

## 3.3 Port Stealing

### Concept

Port stealing is a more targeted attack than MAC flooding. Instead of flooding the entire
CAM table, the attacker targets a **specific victim's MAC address entry** in the CAM table.

**The goal:** Make the switch believe the victim's MAC address is reachable through the
attacker's port, so the switch sends traffic destined for the victim to the attacker instead.

```
PORT STEALING ATTACK - Step by Step
=====================================

INITIAL STATE:
  CAM Table:
  +--------------------+-------+
  | 00:11:22:33:44:55  | Gi0/1 |  <- Victim's PC
  | AA:BB:CC:DD:EE:FF  | Gi0/2 |  <- Attacker's PC
  +--------------------+-------+

STEP 1: Attacker sends frames with VICTIM's MAC as source, from Gi0/2:
  Frame: Source=00:11:22:33:44:55 (VICTIM's MAC!) -> arrives on Gi0/2
  
STEP 2: Switch updates CAM table:
  CAM Table:
  +--------------------+-------+
  | 00:11:22:33:44:55  | Gi0/2 |  <- NOW POINTS TO ATTACKER'S PORT!
  | AA:BB:CC:DD:EE:FF  | Gi0/2 |
  +--------------------+-------+

STEP 3: Any traffic destined for the victim (00:11:22:33:44:55) is now
        forwarded to Gi0/2 (the attacker's port)!

STEP 4: Attacker captures the traffic, then (optionally) forwards it to
        the real victim on Gi0/1 to avoid detection (MITM mode).

WHAT MAKES THIS STEALTHY:
  - Only ONE specific entry is poisoned (not the whole table)
  - Can be timed to coincide with the victim's ARP cache refresh
  - Attack can be intermittent to avoid detection

TOOLS:
  ettercap --port-stealing
  arpspoof (can be used in conjunction)
```

### Mitigation

Port stealing is mitigated by the same controls as MAC flooding:
- Port Security (maximum MAC per port = 1, violation = shutdown)
- 802.1X (authentication prevents unauthorized frame injection)
- Dynamic ARP Inspection (validates IP/MAC consistency)

# 4. ARP-Based Attacks

## 4.1 ARP Spoofing / ARP Poisoning

### Concept

**ARP Spoofing** exploits ARP's fundamental design flaw: **there is no authentication.**
Any device can send an ARP reply to anyone, and the receiver will update its ARP cache
without question.

An attacker sends fake ARP replies telling other devices:
- "I am the default gateway — send all internet-bound traffic to me"
- "I am PC-B — send PC-B's traffic to me"
- "I am the DNS server — send all DNS queries to me"

**Key insight:** ARP is "stateless" — a device will update its ARP cache even if it
never sent an ARP request. This is called a **Gratuitous ARP** or **unsolicited ARP reply.**

### ARP Spoofing MITM Attack Flow

```
ARP SPOOFING - Man-in-the-Middle Attack
=========================================

NORMAL COMMUNICATION:
  PC-A                   Router (Gateway)
  IP: 192.168.1.10       IP: 192.168.1.1
  MAC: AA:AA:AA:AA:AA:AA MAC: GG:GG:GG:GG:GG:GG
  
  PC-A's ARP cache:
  192.168.1.1 -> GG:GG:GG:GG:GG:GG (correct)
  
AFTER ARP POISONING ATTACK:
  Attacker (IP: 192.168.1.99, MAC: EE:EE:EE:EE:EE:EE)
  sends two sets of fake ARP replies:
  
  To PC-A:    "192.168.1.1 is at EE:EE:EE:EE:EE:EE"  (attacker's MAC)
  To Router:  "192.168.1.10 is at EE:EE:EE:EE:EE:EE"  (attacker's MAC)

  Now:
  PC-A's ARP cache:
  192.168.1.1 -> EE:EE:EE:EE:EE:EE (POISONED! points to attacker)
  
  Router's ARP cache:
  192.168.1.10 -> EE:EE:EE:EE:EE:EE (POISONED! points to attacker)
  
TRAFFIC FLOW AFTER ATTACK:
  PC-A --> [ATTACKER] --> Router (attacker forwards, makes it transparent)
  Router --> [ATTACKER] --> PC-A
  
  Attacker is now in the MIDDLE of every conversation!

TIMELINE:
==========
t=0:  Attacker starts sending fake ARP replies every ~10 seconds
      (must keep poisoning because ARP cache has timeout)
t=5s: PC-A's ARP cache poisoned
t=5s: Router's ARP cache poisoned
t=5s: All PC-A traffic flows through attacker (MITM established)
t=ongoing: Attacker must keep sending fake ARPs to maintain position
           (because ARP caches timeout and re-resolve)

ARP REPLY PACKET (Fake):
=========================
Opcode:               2 (Reply)
Sender MAC:           EE:EE:EE:EE:EE:EE (attacker's real MAC)
Sender IP:            192.168.1.1 (VICTIM GATEWAY'S IP — SPOOFED!)
Target MAC:           AA:AA:AA:AA:AA:AA (PC-A's MAC)
Target IP:            192.168.1.10 (PC-A's IP)

Ethernet Header:
  Source MAC:         EE:EE:EE:EE:EE:EE (attacker)
  Destination MAC:    AA:AA:AA:AA:AA:AA (PC-A — targeted, not broadcast!)
```

### What an Attacker Can Do After MITM

```
POST-MITM CAPABILITIES:
========================
1. PASSIVE SNIFFING
   - Capture all HTTP traffic (credentials, cookies, forms)
   - Capture DNS queries (see which sites are visited)
   - Capture email (SMTP/IMAP if unencrypted)

2. SSL STRIPPING
   - Intercept HTTPS redirects
   - Serve HTTP instead of HTTPS to victim (using tools like sslstrip)
   - Victim sees http:// instead of https://

3. SESSION HIJACKING
   - Steal session cookies from HTTP traffic
   - Replay cookies to server to impersonate victim

4. DNS SPOOFING
   - See DNS queries, respond with fake IP addresses
   - Redirect victim to malicious websites

5. CREDENTIAL HARVESTING
   - Capture plaintext passwords from FTP, Telnet, HTTP forms
   
6. MALWARE INJECTION
   - Insert malicious JavaScript into HTTP responses
   - Inject iframes pointing to exploit kits

Tools Used:
  arpspoof: arpspoof -i eth0 -t 192.168.1.10 192.168.1.1
  ettercap: ettercap -T -M arp:remote /192.168.1.1// /192.168.1.10//
  Bettercap: bettercap -eval "arp.spoof on"
```

### Mitigation: Dynamic ARP Inspection (DAI)

**Dynamic ARP Inspection (DAI)** is the primary defense against ARP spoofing.

```
DAI ARCHITECTURE:
==================
DAI relies on the DHCP Snooping Binding Table:
  {MAC Address, IP Address, VLAN, Switch Port, Lease Time}
  
  Example binding table entry:
  MAC=AA:AA:AA:AA:AA:AA, IP=192.168.1.10, VLAN=10, Port=Gi0/1

DAI DECISION FLOW:
==================
ARP packet arrives on port Gi0/1
              |
              v
     Is this port      YES (Trusted: uplinks,   
     DAI-trusted? -------> router ports)         -> Allow ARP, forward
              |
              NO (Untrusted: all access ports)
              |
              v
     Extract: Sender MAC, Sender IP from ARP packet
              |
              v
     Look up in DHCP      FOUND: Check if
     Snooping Table -----> Sender IP matches -----> MATCH -> Forward ARP
                           binding table entry         |
                                                  NO MATCH -> DROP + Log
                           NOT FOUND -> DROP + Log
                           
DAI CONFIGURATION (Cisco):
============================
! Enable DHCP Snooping (prerequisite for DAI)
ip dhcp snooping
ip dhcp snooping vlan 10,20
no ip dhcp snooping information option

! Enable Dynamic ARP Inspection
ip arp inspection vlan 10,20

! Configure trusted ports (uplinks to router/distribution switch)
interface GigabitEthernet0/24
  ip arp inspection trust
  ip dhcp snooping trust

! Optional: Rate limit ARP packets on untrusted ports
interface GigabitEthernet0/1
  ip arp inspection limit rate 100 burst interval 1
  
! Verify
show ip arp inspection
show ip arp inspection vlan 10
show ip arp inspection statistics
```

### Mitigation: Static ARP Entries

For critical infrastructure (default gateways, servers), use static ARP entries:

```
STATIC ARP ENTRIES:
====================
Linux:
  arp -s 192.168.1.1 GG:GG:GG:GG:GG:GG
  # Or persistent:
  echo "192.168.1.1 GG:GG:GG:GG:GG:GG" >> /etc/ethers

Windows:
  netsh interface ipv4 add neighbors "Ethernet" 192.168.1.1 GG-GG-GG-GG-GG-GG

LIMITATION: Does not scale. Useful for servers and network devices,
            not practical for dynamic end-user environments.
            
ARP RATE LIMITING:
==================
Most modern switches can rate-limit ARP packets:
  - Detect burst of ARP replies from a single port
  - Alert or block if ARP rate exceeds threshold (e.g., 100 ARPs/second)
  
ARPON (Linux daemon):
  ArpON = ARP handler inspection
  Inspects ARP traffic, blocks unsolicited ARP replies
  Suitable for Linux endpoints
```

## 4.2 Gratuitous ARP Abuse

### Concept

**Gratuitous ARP:** An ARP reply sent by a device that was NOT preceded by an ARP request.
The device is essentially announcing: "Hey everyone, my MAC is X and my IP is Y."

**Legitimate uses of Gratuitous ARP:**
- Device notifying the network of its new IP address
- Failover (e.g., VRRP/HSRP hot standby router taking over)
- Duplicate IP detection

**Malicious use:**
- Attacker sends gratuitous ARP to poison caches without waiting for a request
- Much faster and more reliable than waiting for legitimate ARP traffic

```
GRATUITOUS ARP ABUSE:
======================
Normal Gratuitous ARP (e.g., PC just got IP from DHCP):
  Source: AA:AA:AA:AA:AA:AA (PC-A's MAC)
  Target: FF:FF:FF:FF:FF:FF (broadcast)
  Content: "192.168.1.10 is at AA:AA:AA:AA:AA:AA"
  
Malicious Gratuitous ARP:
  Source: EE:EE:EE:EE:EE:EE (attacker's MAC)
  Target: FF:FF:FF:FF:FF:FF (broadcast)
  Content: "192.168.1.1 is at EE:EE:EE:EE:EE:EE" <- SPOOFED gateway IP!
  
  ALL devices receive this and update their ARP cache for 192.168.1.1
  -> Mass MITM established with a single packet!
  
ARP POISONING SPEED COMPARISON:
================================
Targeted ARP Reply:  Must wait for victim to send ARP request (slow, targeted)
Gratuitous ARP:      Immediate, affects ALL devices in broadcast domain (fast, wide)
```

# 5. VLAN-Based Attacks

## 5.1 VLAN Hopping: Switch Spoofing

### Concept

**The Goal:** Gain access to a VLAN that you are not authorized to be in.

**Background — DTP (Dynamic Trunking Protocol):**
DTP is a Cisco proprietary protocol that allows switches to automatically negotiate
whether a port should become a trunk link. By default, many Cisco switch ports are in
**"dynamic desirable"** or **"dynamic auto"** mode.

**Dynamic Desirable:** "I actively want to be a trunk. Convince me."
**Dynamic Auto:** "I'll become a trunk if you ask me to."

If an attacker's device sends DTP negotiation frames, the switch port might **upgrade to
a trunk port** — giving the attacker access to **ALL VLANs** on that switch.

```
SWITCH SPOOFING ATTACK - VLAN HOPPING
=======================================

INITIAL STATE:
  Attacker's PC is plugged into an access port:
  Gi0/5 is in "dynamic desirable" mode (Cisco default on older switches)
  
ATTACK:
  Step 1: Attacker's device (e.g., running Yersinia or crafted DTP frames)
          sends DTP negotiation frames pretending to be a switch:
          
          DTP Frame: "I want to form a trunk! Mode: Desirable"
          
  Step 2: Switch port Gi0/5 responds and UPGRADES to trunk mode:
          Port Gi0/5 now carries ALL VLANs (10, 20, 30, 40...)
          
  Step 3: Attacker can now:
          - Send and receive 802.1Q tagged frames
          - Tag outgoing frames with VLAN 20 (Finance) tag
          - Access resources in VLAN 20 that they were blocked from
          - Access ANY VLAN configured on the switch

DTP NEGOTIATION FLOW:
=====================
Attacker                         Switch Gi0/5
   |                                 |
   |--DTP Frame (Mode: Desirable)--> |
   |                                 | Port mode: dynamic desirable
   |                                 | "They want a trunk, I will too"
   | <---DTP Frame (Mode: Trunk)--   |
   |                                 |
   |   Port Gi0/5 is now a TRUNK!   |
   |   All VLANs now accessible!     |

Tool: yersinia -G (graphical)
      yersinia -I (interactive)
      
      Or directly: send crafted 802.3 frames with LLC SNAP header
                   containing DTP Subtype=0x01 (Trunk)
```

### Mitigation

```
SWITCH SPOOFING MITIGATION:
============================

1. DISABLE DTP on all ACCESS ports (MOST IMPORTANT):
   interface GigabitEthernet0/1
     switchport mode access          <- Explicitly set as access port
     switchport nonegotiate          <- Disable DTP entirely on this port
     switchport access vlan 10
     
2. For LEGITIMATE TRUNK ports, also disable DTP auto-negotiation:
   interface GigabitEthernet0/24
     switchport mode trunk           <- Explicitly set as trunk
     switchport nonegotiate          <- Don't auto-negotiate, just be a trunk
     
3. Never leave ports in "dynamic desirable" or "dynamic auto" mode:
   show interfaces status | include connected
   show interfaces trunk
   
   Any access port that appears in "show interfaces trunk" is a red flag!
   
4. Shut down unused ports and assign them to an unused "parking" VLAN:
   interface range GigabitEthernet0/10-20
     shutdown
     switchport access vlan 999      <- Unused/parking VLAN
     switchport mode access
```

## 5.2 VLAN Hopping: Double Tagging

### Concept

Double tagging is a more subtle VLAN hopping attack that works **even on ports correctly
configured as access ports.** It exploits the behavior of the **native VLAN.**

**Prerequisite:** The attacker must be in the switch's **native VLAN** (typically VLAN 1
on Cisco switches). This attack is **one-directional** — the attacker can send traffic
to the target VLAN, but cannot receive replies directly.

### How It Works — Step by Step

```
DOUBLE TAGGING ATTACK
=====================

SETUP:
  Attacker: VLAN 1 (native VLAN) on access port Gi0/1
  Target:   VLAN 20 (Finance server) on different port
  
  Native VLAN on trunk between switches: VLAN 1
  
CONCEPT — Two VLAN Tags:
  The attacker constructs a frame with TWO 802.1Q VLAN tags:
  
  Outer Tag: VLAN 1 (native VLAN)
  Inner Tag: VLAN 20 (target VLAN)
  
  Normal frame:  [DST MAC][SRC MAC][ETH TYPE][PAYLOAD]
  Tagged frame:  [DST MAC][SRC MAC][0x8100][VLAN 20][ETH TYPE][PAYLOAD]
  Double-tagged: [DST MAC][SRC MAC][0x8100][VLAN 1][0x8100][VLAN 20][PAYLOAD]
                                    ^^^^^^^^^^^^^^^^ ^^^^^^^^^^^^^^^^
                                    OUTER TAG        INNER TAG
                                    (native VLAN=1)  (target VLAN=20)

STEP-BY-STEP:
=============
  
  SWITCH 1 (Access Switch):
  +--------------------------+
  | Port Gi0/1: access VLAN 1|
  | Port Gi0/24: trunk       |
  +--------------------------+
  
  Step 1: Attacker sends double-tagged frame out Gi0/1
  
  Step 2: Switch 1 receives frame on Gi0/1 (access port, VLAN 1)
          Switch 1 reads OUTER tag = VLAN 1
          Since VLAN 1 is the NATIVE VLAN, Switch 1 STRIPS the outer tag
          (native VLAN frames are sent untagged on trunk links)
          
          Frame now has only ONE tag: VLAN 20 (the inner tag)
          
  Step 3: Switch 1 forwards frame on trunk link Gi0/24 (to Switch 2)
          Frame only has VLAN 20 tag now
          
  SWITCH 2 (Distribution Switch):
  +--------------------------+
  | Trunk from Switch 1      |
  | Port Gi0/5: Finance VLAN |
  +--------------------------+
  
  Step 4: Switch 2 receives frame on trunk
          Reads tag = VLAN 20
          Forwards to Finance VLAN port!
          
  Step 5: Finance server receives frame from "nowhere" (attacker's IP/MAC)
          Finance server can't reply back to attacker directly
          (replies go through normal routing, not through the attack path)

VISUAL REPRESENTATION:
========================
Attacker                Switch 1               Switch 2         Finance Server
[VLAN 1 port]           [Trunk Gi0/24]         [VLAN 20 port]
     |                       |                      |                 |
     |--[VLAN1][VLAN20]----> |                      |                 |
     |                       | Strip outer VLAN 1   |                 |
     |                       | (native VLAN)        |                 |
     |                       |---[VLAN20]---------> |                 |
     |                       |                      |--[no tag]-----> |
     |                       |                      |                 |
     |    Attacker CANNOT receive replies via this path!              |
```

### Mitigation

```
DOUBLE TAGGING MITIGATION:
===========================

1. CHANGE THE NATIVE VLAN to an UNUSED VLAN (MOST CRITICAL):
   ! Do this on ALL trunk ports:
   interface GigabitEthernet0/24
     switchport trunk native vlan 999    <- Use VLAN 999 (unused/dedicated)
     switchport mode trunk
     switchport nonegotiate
   
   ! Ensure VLAN 999 has no hosts. It's a "dead end" VLAN.
   ! Now attacker in VLAN 1 cannot exploit native VLAN stripping.

2. EXPLICITLY TAG the native VLAN (some platforms support this):
   vlan dot1q tag native                 <- Global command (Cisco IOS)
   ! This causes native VLAN frames to be tagged too
   ! Eliminates the untagged native VLAN behavior entirely

3. NEVER have end-user hosts in VLAN 1:
   ! VLAN 1 is the default and native VLAN — it's dangerous
   ! Move all users to VLAN 10, 20, 30... leaving VLAN 1 empty
   
4. PRUNE VLANs on trunk links (allow only needed VLANs):
   interface GigabitEthernet0/24
     switchport trunk allowed vlan 10,20,30    <- Whitelist approach
   ! This way even if double-tagging succeeds, VLANs not on whitelist
   ! cannot be reached

5. VLAN Access Control Lists (VACLs):
   ! Apply ACLs at the VLAN level to restrict inter-VLAN traffic
   vlan access-map FINANCE_FILTER 10
     match ip address ALLOWED_SOURCES
     action forward
   vlan access-map FINANCE_FILTER 20
     action drop
   vlan filter FINANCE_FILTER vlan-list 20
```

## 5.3 Private VLAN Attacks

### Concept

**Private VLANs (PVLANs)** add a second level of segmentation within a VLAN. The use
case: multiple tenants in the same VLAN (e.g., a datacenter) shouldn't talk to each other.

```
PVLAN Architecture:
===================
Primary VLAN (e.g., VLAN 100)
  |
  +-- Community VLAN (VLAN 101): devices CAN talk to each other
  |     [Server A] <-> [Server B]  (same community)
  |
  +-- Isolated VLAN (VLAN 102): devices CANNOT talk to each other
  |     [Server C] X  [Server D]  (cannot communicate)
  |
  +-- Promiscuous Port: can talk to ALL ports (e.g., the router/firewall)

Port Types:
  Promiscuous: Router/gateway — talks to everyone
  Community:   Servers in same group — talk to each other + promiscuous
  Isolated:    Strict isolation — can only talk to promiscuous port

PVLAN ATTACK - Proxy Routing:
==============================
If the router/gateway at the promiscuous port does NOT have ingress filtering:
  Attacker (isolated port) ---sends frame with--> Router's MAC as destination
  Frame contains:
    Destination MAC: Router's MAC (promiscuous port)
    Destination IP:  Another isolated server's IP (e.g., Server D)
  
  Router receives frame, looks up routing table, forwards to Server D
  -> PVLAN isolation bypassed via the router!
  
MITIGATION:
  Enable "Private VLAN Edge" (Protected Ports) on access switches
  Configure router with anti-proxy-arp
  Apply strict ACLs on the router/gateway interface
  Use RFC 2267 ingress filtering on the gateway
```

# 6. Spanning Tree Protocol Attacks

## 6.1 Rogue Root Bridge Attack

### Concept

**Recall:** The Root Bridge is the "center" of the spanning tree. All switches compute
their best path to the root, and ports are blocked/forwarded accordingly.

**The attack:** If an attacker can become the Root Bridge, they can:
1. Control the entire Layer 2 topology
2. Force traffic to flow through their device (MITM)
3. Cause topology reconvergence (DoS — 30-50 seconds of downtime per change)
4. Redirect traffic away from legitimate paths

**How easy is this?** Extremely easy. Send ONE BPDU with Priority=0 and the switch
immediately considers you the best candidate for root bridge.

```
ROGUE ROOT BRIDGE ATTACK
=========================

NORMAL TOPOLOGY:
  Priority 4096              Priority 32768          Priority 32768
  [Core Switch - ROOT]  --   [Switch A]          --  [Switch B]
  BID: 4096 + MAC-C          BID: 32768 + MAC-A      BID: 32768 + MAC-B

  Root bridge = Core Switch (lowest priority)
  All traffic flows through Core Switch (correct path)

ATTACK - Step 1: Attacker sends BPDU with Priority=0:
  Priority 0!
  [Attacker Device]    --   [Switch B (access port)]
  BPDU: Priority=0, BID= 0 + AA:BB:CC:DD:EE:FF
  
  Attacker's BID = 0 + AA:BB:CC:DD:EE:FF
  Core Switch BID = 4096 + MAC-C
  
  0 < 4096 -> Attacker wins Root Bridge election!

ATTACK - Step 2: STP Reconvergence begins:
  All switches recalculate paths (30-50 seconds of potential disruption)
  All switches now route traffic toward the attacker's device!

ATTACK - Step 3: MITM position achieved:
  Traffic flows: [PC-A] -> [Switch A] -> [ATTACKER] -> [Core Switch] -> Internet
  Attacker sees ALL traffic passing through them

BPDU CRAFTED BY ATTACKER:
==========================
Protocol:          0x0000 (STP)
Version:           0x00
Type:              0x00 (Configuration BPDU)
Root ID Priority:  0          <- Set to 0 (LOWEST POSSIBLE)
Root ID MAC:       AA:BB:CC:DD:EE:FF (attacker's MAC)
Root Path Cost:    0
Bridge Priority:   0
Bridge MAC:        AA:BB:CC:DD:EE:FF
Port ID:           0x8001
Message Age:       0
Max Age:           20
Hello Time:        2
Forward Delay:     15

Tools: yersinia -attack 2 (STP root bridge attack)
       scapy: craft custom STP BPDUs
```

### Mitigation: BPDU Guard and Root Guard

```
SPANNING TREE PROTECTIONS
===========================

1. BPDU GUARD (for access ports - protects PortFast ports)
   =========================================================
   Purpose: If ANY BPDU is received on a PortFast-enabled port, SHUT IT DOWN.
   Reasoning: End devices (PCs, servers) should NEVER send BPDUs.
              If BPDUs arrive, it means a switch or rogue device is connected.
   
   ! Enable globally (applies to all PortFast-enabled ports):
   spanning-tree portfast bpduguard default
   
   ! Or per-interface:
   interface GigabitEthernet0/1
     spanning-tree portfast
     spanning-tree bpduguard enable
   
   BPDU GUARD FLOW:
   ==================
   BPDU received on Gi0/1
          |
          v
   Is BPDU Guard     YES -> Port enters err-disabled state IMMEDIATELY
   enabled? -------->        Interface shuts down
          |                  Log/SNMP alert generated
          NO
          |
          v
   Process BPDU normally

2. ROOT GUARD (for ports that should never become root ports)
   ===========================================================
   Purpose: If a port receives a SUPERIOR BPDU (one that would make
            the sender the new root), the port is placed in INCONSISTENT
            state (blocked) instead of reconverging.
   
   Use on: Ports connecting to access layer or external networks.
           Ports where legitimate superior BPDUs should never arrive.
   
   ! Configure on distribution/core ports facing access layer:
   interface GigabitEthernet0/24
     spanning-tree guard root
     
   ROOT GUARD FLOW:
   ==================
   BPDU received on Gi0/24 (root guard enabled)
          |
          v
   Is this a Superior   YES -> Port enters ROOT-INCONSISTENT state
   BPDU? (better than ------->  Port BLOCKED (no traffic forwarded)
   current root?) |              Log generated
          |                     When superior BPDUs stop, port recovers
          NO
          v
   Process BPDU normally, no change

3. BPDU FILTER (use with extreme caution!)
   ==========================================
   Purpose: Suppress all BPDUs in/out of a port.
   WARNING: Can CREATE loops if misconfigured! Use only on edge ports.
   
   spanning-tree portfast bpdufilter default
   
   Note: When globally applied, only suppresses BPDUs on PortFast ports.
         If BPDUs ARE received, the port loses PortFast and operates normally.

4. PORTFAST (reduces convergence time for access ports)
   =======================================================
   Purpose: Skip LISTENING and LEARNING states (30-second wait).
            Immediately move to FORWARDING.
   Use on: Access ports connected to end devices ONLY.
   
   interface GigabitEthernet0/1
     spanning-tree portfast
     
   WARNING: NEVER enable PortFast on ports connected to switches!
            This would bypass loop prevention!

SUMMARY TABLE:
==============
+-------------+-------------------+---------------------------+
| Feature     | Protects Against  | Applied On                |
+-------------+-------------------+---------------------------+
| BPDU Guard  | Rogue switch/BPDU | Access/PortFast ports     |
| Root Guard  | Rogue root bridge | Distribution/uplink ports |
| BPDU Filter | Suppresses BPDUs  | Edge ports ONLY (careful) |
| PortFast    | Slow convergence  | Access ports (end devices)|
+-------------+-------------------+---------------------------+
```

## 6.2 STP Topology Change Attacks (TCN Flooding)

### Concept

**TCN = Topology Change Notification.** When a switch detects a topology change (port
going up or down), it sends a TCN BPDU. The Root Bridge then floods a TC (Topology Change)
BPDU, causing all switches to **reduce their CAM table aging timer from 300 seconds to
15 seconds.**

**Why does this matter?** With a 15-second aging timer, the CAM table is constantly being
flushed. Old entries are removed, and the switch must flood frames to rediscover where
devices are. This creates:
1. Increased flooding (like a partial MAC flood attack)
2. Network performance degradation
3. Opportunity for traffic interception during re-learning phase

```
TCN FLOODING ATTACK:
=====================
Attacker continuously sends TCN BPDUs:

  [Attacker]   ->   TCN BPDU   ->   [Switch]   ->   TCN propagated to Root
  
  Root sends TC BPDU back to all switches:
  "Flush your CAM tables fast! Age = 15 seconds!"
  
  All switches constantly flushing CAM tables:
  -> Constant flooding
  -> Constant re-learning
  -> Network performance severely degraded
  -> Opportunity to intercept traffic during flooding phases

MITIGATION:
  1. BPDU Guard on access ports (stops TCNs from untrusted ports)
  2. BPDU Filter (suppresses TCNs from edge ports)
  3. Rapid STP (802.1w/802.1s) reduces TC impact significantly
  4. Loop Guard on uplinks prevents port state manipulation
  
! Enable Loop Guard globally:
spanning-tree loopguard default

Loop Guard: If a port stops receiving BPDUs (when it should be),
            instead of transitioning to forwarding (potentially creating a loop),
            the port enters "loop-inconsistent" state and blocks traffic.
```

# 7. DHCP-Based Attacks

## 7.1 DHCP Starvation

### Concept

**DHCP Starvation** exhausts the DHCP server's IP address pool, preventing legitimate
devices from obtaining IP addresses (Denial of Service).

**The DHCP pool** is a range of IP addresses the server can assign. A typical small
office might have 192.168.1.100 - 192.168.1.200 (100 addresses). An attacker with
a script can request all 100 of these in seconds.

```
DHCP STARVATION ATTACK:
========================
Attacker sends many DHCP DISCOVER requests, each with a different
spoofed source MAC address:

DHCP Request 1: Source MAC = AA:AA:AA:AA:AA:01, wants IP
DHCP Request 2: Source MAC = AA:AA:AA:AA:AA:02, wants IP
DHCP Request 3: Source MAC = AA:AA:AA:AA:AA:03, wants IP
...
DHCP Request N: Source MAC = AA:AA:AA:AA:AA:NN, wants IP

DHCP Server:
  "01 needs an IP - give them 192.168.1.100"
  "02 needs an IP - give them 192.168.1.101"
  ...
  "NN needs an IP - POOL EXHAUSTED! No more IPs available!"

AFTER ATTACK:
  New legitimate PC plugs in, sends DHCP DISCOVER
  DHCP Server: "No IP addresses available!"
  PC cannot get an IP -> cannot use the network (DoS!)

Combined attack (Starvation + Rogue DHCP):
  1. Starve the legitimate DHCP server (pool exhausted)
  2. Start your own rogue DHCP server
  3. Answer legitimate clients' DISCOVER messages
  4. Assign them your malicious gateway, DNS server
  5. All their traffic flows through you!

Tool: dhcpstarv, yersinia, gobbler
  yersinia dhcp -attack 1  (DHCP exhaustion)
```

### Mitigation: DHCP Snooping

**DHCP Snooping** is the primary defense. It acts as a "traffic cop" for DHCP messages.

```
DHCP SNOOPING ARCHITECTURE:
============================

Concept: Ports are classified as TRUSTED or UNTRUSTED.
  Trusted:   Only DHCP server ports and uplinks
             Can send DHCP OFFER, DHCP ACK, DHCP NAK
  Untrusted: All client/access ports
             Can ONLY send DHCP DISCOVER, DHCP REQUEST
             DHCP OFFER/ACK from untrusted ports -> DROPPED

SNOOPING BINDING TABLE:
  DHCP Snooping builds a binding table from legitimate DHCP exchanges:
  {Client MAC, Assigned IP, Port, VLAN, Lease Time}
  
  This table is then used by DAI (ARP Inspection) and IP Source Guard.

DHCP MESSAGE FILTERING RULES:
================================
Untrusted Port Receives:              Action:
DHCP DISCOVER (client request)    ->  Forward to trusted port (OK)
DHCP REQUEST  (client confirming) ->  Forward to trusted port (OK)
DHCP OFFER    (server response)   ->  DROP! (Rogue server!)
DHCP ACK      (server confirm)    ->  DROP! (Rogue server!)
DHCP NAK      (server deny)       ->  DROP! (Rogue server!)

DHCP STARVATION DEFENSE:
  DHCP Snooping rate-limits DHCP packets per port:
  interface GigabitEthernet0/1
    ip dhcp snooping limit rate 15   <- Max 15 DHCP packets/second
    
  If port exceeds 15 DHCP/sec -> port is err-disabled (blocked)
  This stops starvation attacks cold.

DHCP SNOOPING CONFIGURATION (Cisco):
======================================
! Enable globally
ip dhcp snooping
ip dhcp snooping vlan 10,20,30

! Trust uplinks and DHCP server ports ONLY
interface GigabitEthernet0/24        <- Uplink to DHCP server/core
  ip dhcp snooping trust

! Rate limit untrusted access ports
interface range GigabitEthernet0/1-20
  ip dhcp snooping limit rate 15

! Verify
show ip dhcp snooping
show ip dhcp snooping binding
show ip dhcp snooping statistics
```

## 7.2 Rogue DHCP Server

```
ROGUE DHCP SERVER ATTACK:
===========================
Attacker plugs in their own DHCP server (or uses software like dnsmasq):

Timeline:
  t=0: Legitimate DHCP server: 192.168.1.1, pool: .100-.200
  t=1: Attacker starts rogue DHCP server
  t=2: PC-X plugs in, sends DHCP DISCOVER (broadcast)
  t=3: BOTH servers receive DISCOVER
  t=4: BOTH servers send DHCP OFFER
  t=5: PC-X accepts FIRST offer received (often attacker's if closer)
  
Attacker's malicious DHCP offer:
  IP Address:       192.168.1.150    (seems legitimate)
  Subnet Mask:      255.255.255.0    (correct)
  Default Gateway:  192.168.1.99     (ATTACKER'S IP! <- Evil)
  DNS Server:       8.8.8.8 or 192.168.1.99 (can control DNS too)
  Lease Time:       Infinite         (victim stuck with bad config)
  
Now:
  PC-X sends ALL traffic to 192.168.1.99 (attacker's gateway)
  Attacker can forward to real gateway (transparent MITM)
  OR drop all traffic (DoS)
  OR redirect DNS queries to malicious servers

MITIGATION: DHCP Snooping (described above)
  - All DHCP server responses from untrusted ports are DROPPED
  - Rogue server's OFFER and ACK messages never reach clients
```

## 7.3 IP Source Guard

**IP Source Guard (IPSG)** extends DHCP Snooping protection to prevent IP spoofing.

```
IP SOURCE GUARD:
================
Uses the DHCP Snooping binding table to validate IP packets.

For each packet arriving on an untrusted port:
  Source IP and Source MAC are checked against binding table.
  
  If MATCH found: forward the packet
  If NO MATCH:    DROP the packet

This prevents:
  - IP address spoofing (attacker claiming to be a different IP)
  - Bypassing ACLs that are based on source IP

CONFIGURATION:
==============
! Enable IPSG on access ports
interface GigabitEthernet0/1
  ip verify source          <- Validates source IP only
  ! OR:
  ip verify source port-security   <- Validates source IP AND MAC

! Prerequisite: DHCP Snooping must be enabled and binding table populated

MANUAL BINDING (for hosts with static IPs):
  ip source binding AA:BB:CC:DD:EE:FF vlan 10 192.168.1.50 interface Gi0/1

show ip verify source
show ip source binding
```

# 8. CDP / LLDP Attacks

## 8.1 CDP — Cisco Discovery Protocol

**CDP** is a Cisco proprietary protocol that runs on Layer 2. It allows Cisco devices to
discover neighboring Cisco devices and exchange information about themselves.

**CDP is UNAUTHENTICATED and runs on all ports by default.**

```
WHAT CDP REVEALS (information disclosure):
==========================================
CDP Advertisement contains:
  - Device hostname
  - IP addresses of ALL interfaces
  - IOS software version (exact version!)
  - Hardware platform (model number)
  - Port capabilities
  - Native VLAN
  - Duplex settings
  - Power over Ethernet (PoE) info
  - VTP (VLAN Trunking Protocol) domain
  
This is essentially a complete network inventory handed to any attacker
on the LAN segment!

RISK ASSESSMENT:
  An attacker knows: "This is a Cisco Catalyst 3850 running IOS 16.9.4"
  -> Can look up known CVEs for that exact version
  -> Can plan targeted exploitation

CDP ATTACKS:
============
1. Information Gathering:
   wireshark: filter "cdp"
   cdp-tool, cdpsnarf
   
2. CDP Flooding (DoS):
   Attacker sends thousands of fake CDP advertisements
   -> Some older switches crash or have CPU overload
   -> CDP neighbor table overflows
   
   yersinia cdp -attack 1  (CDP flood)
   
3. VoIP VLAN Discovery:
   CDP reveals the Voice VLAN ID
   -> Attacker tags their traffic with Voice VLAN ID
   -> Bypasses data/voice VLAN segregation
   -> Can intercept VoIP calls, launch QoS bypass attacks

4. STP/DTP Interaction:
   CDP reveals switch capabilities, aiding other attack planning
```

### Mitigation

```
CDP MITIGATION:
===============
1. Disable CDP GLOBALLY (if no Cisco-to-Cisco management needed):
   no cdp run
   
2. Disable CDP per-interface on ALL access/external-facing ports:
   interface GigabitEthernet0/1
     no cdp enable
   
   ! Keep CDP only on switch-to-switch and switch-to-router trunk links
   ! where device discovery is needed for management

3. Use LLDP instead (if needed for multi-vendor):
   no cdp run           <- Disable CDP
   lldp run             <- Enable LLDP
   ! LLDP also needs to be secured (see below)

CDP SECURITY BEST PRACTICE:
  - Disable CDP on all ports connected to end-user devices
  - Disable CDP on all external-facing or untrusted ports
  - Disable CDP on WAN/internet-facing interfaces
  - Keep CDP only on device-to-device management links
  - Monitor CDP with IDS for unexpected CDP sources
```

## 8.2 LLDP — Link Layer Discovery Protocol

**LLDP** is the IEEE standard equivalent of CDP (802.1AB). It's vendor-neutral and used
in multi-vendor environments.

```
LLDP ATTACKS AND MITIGATION:
==============================
Same attack categories as CDP:
  - Information disclosure (device type, software version, IP addresses)
  - LLDP flooding (DoS)
  - LLDP-MED (Media Endpoint Discovery) abuse for VoIP attacks
  
LLDP-specific concern:
  LLDP-MED allows VoIP phones to automatically configure themselves.
  An attacker can send LLDP-MED frames advertising a fake Voice VLAN.
  VoIP phones may accept the fake configuration.

MITIGATION:
  interface GigabitEthernet0/1
    no lldp transmit        <- Don't send LLDP on this port
    no lldp receive         <- Don't process LLDP received on this port
  
  ! Global disable:
  no lldp run
  
  LLDP Rate Limiting:
  lldp timer 30            <- Reduce LLDP transmission frequency
  lldp holdtime 120        <- Reduce LLDP holdtime
```

# 9. 802.1X and Authentication Bypass

## 9.1 IEEE 802.1X — Port-Based Network Access Control

**802.1X** is an IEEE standard that provides **authentication** at Layer 2, before any
network access is granted. It's the strongest L2 access control mechanism available.

```
802.1X COMPONENTS:
==================
1. Supplicant:     The end device (PC, laptop) requesting access
2. Authenticator:  The switch (enforces access control)  
3. Authentication Server: RADIUS server (validates credentials)

802.1X TERMINOLOGY:
  EAP: Extensible Authentication Protocol (the auth framework)
  EAPOL: EAP over LAN (how EAP is carried at Layer 2)
  RADIUS: Remote Authentication Dial-In User Service (backend server)

802.1X PORT STATES:
  Unauthorized: Only 802.1X/EAP traffic allowed. No IP, no data.
  Authorized:   Full network access granted.

802.1X AUTHENTICATION FLOW:
============================
Supplicant            Authenticator           RADIUS Server
(PC)                  (Switch)                (Auth Server)
  |                       |                       |
  |<--EAPOL-Request----   |                       |  Switch triggers auth
  |                       |                       |
  |---EAPOL-Response-->   |                       |  PC sends credentials
  |                       |---RADIUS Access-Req-->|  Switch forwards to RADIUS
  |                       |                       |  RADIUS validates
  |                       |<--RADIUS Access-Acc---|  RADIUS approves
  |                       |                       |
  |<--EAPOL-Success---    |                       |  Switch notifies PC
  |                       |                       |
  | PORT IS NOW AUTHORIZED|                       |
  | Full network access!  |                       |

EAP METHODS (order of security, most secure first):
  EAP-TLS:     Mutual cert authentication (MOST SECURE — no passwords!)
  EAP-TTLS:    TLS tunnel + inner auth (password inside encrypted tunnel)
  PEAP:        MS-CHAPv2 inside TLS tunnel (widely used in enterprise)
  EAP-MD5:     Weak, obsolete (password hashed but vulnerable)
  EAP-FAST:    Cisco proprietary, uses PAC files
```

### 802.1X Bypass Attacks

```
802.1X BYPASS ATTACKS:
=======================

1. NON-SUPPLICANT DEVICES (No 802.1X Support):
   Problem: Printers, IP phones, IoT devices often can't run 802.1X
   Attack:  Attacker plugs in a non-supplicant device
   Switch:  Waits for EAPOL. Times out. What happens?
   
   Solution options:
     a) MAC Authentication Bypass (MAB): If no 802.1X after timeout,
        authenticate by MAC address. Weak (MAC is spoofable) but 
        necessary for legacy devices.
     b) Guest VLAN: Unauthenticated devices go to a restricted VLAN
     c) Auth-Fail VLAN: Failed auth goes to limited-access VLAN

2. IDENTITY SPOOFING (Credential Theft):
   If using PEAP/EAP-TTLS with password auth:
   - Rogue AP with fake RADIUS can capture MSCHAPv2 hashes
   - Offline brute-force cracks password (hashcat, asleap)
   MITIGATION: Use EAP-TLS (certificate-based, no password to steal)

3. EAPOL LOGOFF ATTACK (Session Hijacking):
   Attack: Attacker sends spoofed EAPOL-Logoff frame with victim's MAC
   Switch: Thinks victim logged off, unauthorizes the port
   Victim: Loses network access (DoS)
   MITIGATION: Lock port to authenticated MAC after auth (sticky MAC)

4. HUB-IN-THE-MIDDLE (802.1X Bypass via Hub):
   Scenario:
   [Authorized Printer] -- [Hub] -- [Switch with 802.1X]
                            |
                       [ATTACKER]
   
   Printer authenticates. Switch sees printer's MAC -> authorizes port.
   Hub connects attacker to the now-authorized port.
   Attacker gets network access by piggybacking on printer's auth!
   
   Mitigation:
   - Limit to ONE MAC per port (port security)
   - Enable 802.1X multi-auth mode (each device must authenticate)
   - Use 802.1X with MACsec (Layer 2 encryption)

5. EAP-MD5 ATTACK:
   If switch allows EAP-MD5:
   - Attacker captures challenge-response during auth
   - MD5 is crackable offline
   MITIGATION: Disable EAP-MD5. Require EAP-TLS or PEAP minimum.

CONFIGURATION (Cisco):
=======================
! Enable 802.1X globally
aaa new-model
aaa authentication dot1x default group radius
dot1x system-auth-control

! Configure RADIUS server
radius server RADIUS-1
  address ipv4 192.168.10.1 auth-port 1812 acct-port 1813
  key STRONG-SECRET-KEY

! Enable 802.1X on access ports
interface GigabitEthernet0/1
  dot1x pae authenticator
  dot1x port-control auto          <- Must authenticate to get access
  authentication event fail action authorize vlan 999   <- Fail VLAN
  authentication event no-response action authorize vlan 888  <- Guest VLAN
  mab                              <- MAC Auth Bypass as fallback
  spanning-tree portfast
```

# 10. Broadcast / Loop Attacks

## 10.1 Broadcast Storms

```
BROADCAST STORM:
================
A broadcast storm occurs when broadcast frames circulate endlessly
in a network with loops. Every device echoes every broadcast, causing
exponential growth in traffic until the network collapses.

CAUSES:
  1. Physical loop created (two cables between same switches)
  2. STP disabled or misconfigured
  3. STP topology change attack causing improper port states
  4. Attacker intentionally creates a loop

ATTACK MECHANISM:
  Step 1: Attacker connects two ports of a switch together (physical loop)
          OR manipulates STP to unblock a redundant link
  Step 2: A single broadcast (ARP, DHCP, etc.) enters the loop
  Step 3: Each switch forwards it out all ports, creating copies
  Step 4: Each copy creates more copies: 1 -> 2 -> 4 -> 8 -> 16...
  Step 5: Network saturated within seconds. All devices CPU-maxed.
          Nothing else can transit the network.

BROADCAST STORM PROPAGATION:
==============================
  t=0: 1 broadcast frame in loop
  t=1ms: 2 frames
  t=2ms: 4 frames
  t=3ms: 8 frames
  t=10ms: 1024 frames
  t=20ms: 1,048,576 frames -- NETWORK DEAD

MITIGATIONS:
============
1. Spanning Tree Protocol (enables by default — don't disable it!)
   spanning-tree mode rapid-pvst   <- Use RSTP for faster convergence

2. Storm Control:
   Limits the rate of broadcast/multicast/unknown-unicast frames.
   When threshold exceeded, port is err-disabled or traffic dropped.
   
   interface GigabitEthernet0/1
     storm-control broadcast level 20 10
     ! Level 20% = rising threshold (start limiting)
     ! Level 10% = falling threshold (stop limiting)
     storm-control action shutdown   <- Shut port if storm detected
     storm-control action trap       <- Send SNMP alert
   
   show storm-control GigabitEthernet0/1
   
3. Loop Guard (STP):
   Prevents ports from transitioning to forwarding when they stop
   receiving BPDUs (unidirectional link failure can cause loops).
   spanning-tree loopguard default

4. UniDirectional Link Detection (UDLD):
   Detects fiber or copper links where one direction has failed.
   Physical one-way links can cause STP to miscalculate.
   udld enable
   udld aggressive     <- More aggressive detection
```

# 11. Rogue Device Attacks

## 11.1 Rogue Access Points

```
ROGUE ACCESS POINT:
===================
An unauthorized wireless access point connected to the wired network.

Types:
  1. Employee-deployed "convenience AP": Employee plugs in personal AP.
     No malicious intent, but bypasses network security controls.
     
  2. Attacker-deployed rogue AP: Attacker physically or remotely installs

  3. Evil Twin: Attacker creates an AP with the same SSID as the
     legitimate network. Clients connect to attacker's AP instead.

ROGUE AP ATTACK FLOW:
=====================
  [Internet]
       |
   [Firewall]
       |
   [Core Switch]
       |         <- Attacker physically connects rogue AP here
   [Access Switch]
       |
  [Rogue AP]      <- Broadcasts legitimate-looking SSID
       |
  [Victim devices connecting via WiFi]
  
All wireless traffic now bypasses:
  - Network Access Control
  - VLAN segmentation
  - 802.1X authentication
  - IDS/IPS inline inspection

DETECTION TECHNIQUES:
  1. Wireless IDS/IPS: Scan for unauthorized SSIDs and BSSIDs
  2. Monitor switch ports for unexpected new devices (by MAC OUI)
  3. 802.1X on ALL wired ports: Rogue AP can't connect without auth
  4. RF monitoring: Detect unusual RF signals
  5. DHCP tracking: Unexpected device types in DHCP binding table
  6. Network scanning: Regular Nmap scans for unexpected devices

MITIGATION:
  1. 802.1X on ALL wired ports (even "empty" ones)
  2. Physical port security — lock unused ports
  3. Wireless IDS/IPS deployment
  4. Disable unused switch ports, assign to parking VLAN
  5. Network inventory management + alerting for new MAC addresses
  6. NAC (Network Admission Control) solutions
```

## 11.2 Rogue Switches

```
ROGUE SWITCH ATTACK:
====================
Attacker plugs in a managed switch to gain:
  - VTP (VLAN Trunking Protocol) access
  - STP influence (see Root Bridge attack)
  - Ability to create trunk links (see switch spoofing)
  - Network topology visibility via CDP/LLDP

VTP (VLAN TRUNKING PROTOCOL) ATTACK:
======================================
VTP synchronizes VLAN databases across switches automatically.
VTP modes: Server, Client, Transparent.

  VTP Server: Creates/modifies/deletes VLANs. Sends to all.
  VTP Client: Receives VLAN updates. Cannot modify locally.
  VTP Transparent: Passes VTP but uses own VLAN config.

ATTACK:
  Attacker connects a switch with:
  - Same VTP domain name (discovered via CDP)
  - Higher VTP revision number (incremented to trigger update)
  
  VTP Update sent:
  "I am VTP domain CORPORATE, revision 9999, here is my VLAN database"
  
  All Client switches accept this (higher revision number wins!)
  -> ALL VLANs on ALL switches replaced with attacker's VLAN config
  -> VLANs deleted -> ALL hosts lose network connectivity (catastrophic DoS!)

VTP MITIGATION:
  1. Use VTP version 3 (supports MD5 authentication, better security)
  2. Set all access ports to VTP Transparent mode:
     vtp mode transparent
  3. Set VTP password:
     vtp password STRONGPASSWORD hidden
  4. Disable VTP entirely (use manual VLAN config):
     vtp mode off
  5. 802.1X on all ports prevents rogue switch connection
```

# 12. Advanced L2 Attack Chains

## 12.1 The Complete MITM Chain

Real-world attackers combine multiple techniques:

```
ADVANCED L2 ATTACK CHAIN — Complete Scenario
=============================================

TARGET: Steal credentials of corporate users
ATTACKER: Connected to VLAN 10 (corporate LAN)

PHASE 1: RECONNAISSANCE (Passive)
===================================
  [Action 1]: Listen to CDP/LLDP broadcasts
              -> Learn: switch model, IOS version, VTP domain, native VLAN
  
  [Action 2]: Passively sniff ARP traffic
              -> Learn: IP range, MAC addresses of gateway and key hosts
              -> Learn: Gateway MAC = GG:GG:GG:GG:GG:GG, IP = 192.168.1.1
  
  [Action 3]: Observe DHCP traffic
              -> Learn: DHCP server IP, lease times, DNS servers
              
PHASE 2: POSITIONING
=====================
  [Action 4]: ARP Poisoning
              -> Poison gateway: "192.168.1.10 (victim) = MY MAC"
              -> Poison victim: "192.168.1.1 (gateway) = MY MAC"
              -> Enable IP forwarding: sysctl -w net.ipv4.ip_forward=1
              -> Attacker is now MITM!
              
PHASE 3: EXPLOITATION
======================
  [Action 5]: SSL Stripping
              -> Tool: sslstrip
              -> Intercept HTTPS redirects, serve HTTP
              -> Victim sends credentials over HTTP -> captured!
              
  [Action 6]: DNS Spoofing
              -> Respond to DNS queries with attacker-controlled IPs
              -> Redirect banking.example.com to attacker's clone site
              
  [Action 7]: Session Hijacking
              -> Extract session cookies from HTTP traffic
              -> Inject cookies into attacker's browser -> full access
              
PHASE 4: PERSISTENCE
=====================
  [Action 8]: Modify DHCP (optional — requires rogue DHCP setup)
              -> Exhaust legit DHCP pool
              -> Serve malicious gateway permanently
              -> Survive reboots and reconnections

DETECTION POINTS IN THIS CHAIN:
================================
  Phase 1: CDP disabled -> No recon
  Phase 2: DAI enabled -> ARP poisoning blocked
  Phase 2: 802.1X -> Attacker can't connect at all
  Phase 3: HTTPS everywhere (HSTS) -> SSL stripping fails
  Phase 3: DNSSEC -> DNS spoofing fails
  Phase 3: Certificate pinning -> MITM TLS interception fails
  Phase 4: DHCP Snooping -> Rogue DHCP blocked
```

## 13.2 MACsec — IEEE 802.1AE Layer 2 Encryption

**MACsec** is the ultimate L2 defense. It encrypts and authenticates every Ethernet frame,
making all the above attacks impossible (or easily detectable).

```
MACsec (IEEE 802.1AE):
=======================
Provides:
  - Hop-by-hop Layer 2 encryption (AES-GCM-128/256)
  - Frame integrity verification (GCM authentication tag)
  - Replay protection
  - Confidentiality (sniffing gives only encrypted bytes)

MACsec FRAME FORMAT:
====================
+----------+----------+--------+---------+---------+---------+-------+
| DST MAC  | SRC MAC  | EtherT | SecTAG  | Payload | ICV     | FCS   |
| 6 bytes  | 6 bytes  | 2 bytes| 8 bytes | Encrypt | 16 bytes| 4 bytes|
+----------+----------+--------+---------+---------+---------+-------+
                                  ^           ^         ^
                                  |           |         |
                              Security    Encrypted  Integrity
                              Tag (VLAN, counter,   Check Value
                              key ID)    auth       (AES-GCM MAC)

MACsec vs Other L2 Security:
  Port Security: Limits MACs, but doesn't encrypt or authenticate frames
  DAI:           Validates ARP, but frames themselves are in plaintext
  802.1X:        Authenticates devices, but traffic unencrypted after auth
  MACsec:        Encrypts and authenticates EVERY FRAME — ARP, IP, all!

IMPACT ON ATTACKS:
  ARP Spoofing:   Impossible (frames are authenticated — can't fake them)
  MAC Spoofing:   Useless (attacker has no encryption key, frames rejected)
  MAC Flooding:   Still possible conceptually, but frames without valid key
                  are dropped at hardware level
  Traffic sniff:  Impossible (all payload encrypted)

MACsec Configuration (Cisco IOS-XE):
=======================================
! Define MACsec key
key chain MKA-KEY macsec
  key 01
    key-string STRONGHEXKEY cryptographic-algorithm aes-128-cmac

! Apply to interface
interface GigabitEthernet0/1
  macsec
  mka policy MKA-POLICY
  mka pre-shared-key key-chain MKA-KEY

LIMITATION: Both ends must support MACsec. Not all devices do.
            More CPU overhead. Works hop-by-hop (not end-to-end).
```

```
COMPLETE L2 HARDENING CHECKLIST
=================================

SWITCH GLOBAL SETTINGS:
  [ ] Disable unused services (HTTP server, Telnet)
  [ ] Use SSH v2 only for management
```

## 1. Foundations of Layer 2

### 1.1 What is Layer 2?

The OSI model's **Data Link Layer (Layer 2)** is responsible for node-to-node communication within a local network segment. It sits directly above the Physical Layer (Layer 1) and below the Network Layer (Layer 3). Layer 2 is responsible for:

- **Framing:** Packaging raw bits into structured frames.
- **Physical Addressing:** Using MAC (Media Access Control) addresses to identify devices on a local segment.
- **Error Detection:** Using checksums (CRC — Cyclic Redundancy Check) to detect frame corruption.
- **Flow Control:** Managing the rate of data transmission between devices.
- **Media Access Control:** Governing how devices share access to the physical medium (e.g., CSMA/CD for Ethernet).

Layer 2 is further divided into two sublayers:

- **LLC (Logical Link Control — IEEE 802.2):** Handles error checking, flow control, and multiplexing of network protocols above L2.
- **MAC (Media Access Control):** Handles physical addressing and channel access control.

### 1.2 Ethernet Frame Structure

Understanding Layer 2 security begins with understanding the Ethernet frame:

```
+----------+---------+------+--------+------+---------+------+
| Preamble | SFD     | Dst  | Src    | Type | Payload | FCS  |
| 7 bytes  | 1 byte  | MAC  | MAC    | /Len |         |      |
|          |         |6B    |6B      | 2B   | 46-1500B| 4B   |
+----------+---------+------+--------+------+---------+------+
```

- **Preamble (7 bytes):** Alternating 1s and 0s used for clock synchronization.
- **SFD (Start Frame Delimiter, 1 byte):** Marks the beginning of the frame.
- **Destination MAC (6 bytes):** Target hardware address.
- **Source MAC (6 bytes):** Sender's hardware address.
- **EtherType/Length (2 bytes):** Identifies the encapsulated protocol (e.g., 0x0800 = IPv4, 0x0806 = ARP, 0x8100 = 802.1Q VLAN).
- **Payload (46–1500 bytes):** Upper-layer data.
- **FCS (Frame Check Sequence, 4 bytes):** CRC-32 for error detection.

### 1.3 MAC Addresses

A MAC address is a 48-bit (6-byte) hardware identifier, typically represented in colon-hexadecimal notation (e.g., `00:1A:2B:3C:4D:5E`):

- **OUI (Organizationally Unique Identifier):** First 3 bytes, assigned by IEEE to manufacturers.
- **NIC Specific:** Last 3 bytes, assigned by the manufacturer to uniquely identify the device.

**Special MAC Addresses:**
- `FF:FF:FF:FF:FF:FF` — Broadcast (delivered to all hosts on the segment).
- `01:XX:XX:XX:XX:XX` — Multicast (first bit of first byte set to 1).
- `00:XX:XX:XX:XX:XX` — Unicast (first bit of first byte set to 0).
- `01:00:5E:XX:XX:XX` — IPv4 multicast range.
- `33:33:XX:XX:XX:XX` — IPv6 multicast range.

### 1.4 Switches and the CAM Table

Modern Ethernet networks use **switches** instead of hubs to provide dedicated bandwidth and collision domain isolation. Switches maintain a **CAM (Content Addressable Memory) table** (also called the MAC address table or forwarding table):

- When a frame arrives, the switch reads the source MAC and records it alongside the ingress port and VLAN into the CAM table.
- When forwarding, the switch looks up the destination MAC. If found, it forwards the frame only out the matching port (unicast). If not found, it **floods** the frame out all ports in that VLAN except the ingress port.
- CAM entries expire after a configurable aging timer (default: 300 seconds on most Cisco switches).

This flooding behavior on unknown unicast frames is a critical design characteristic that is exploited in several Layer 2 attacks.

### 1.5 VLANs

A **Virtual LAN (VLAN)** is a logical partitioning of a physical switch into multiple isolated broadcast domains. VLANs:

- Separate broadcast traffic between groups.
- Provide logical segmentation without requiring physical separation.
- Are implemented using **IEEE 802.1Q** tagging.

**802.1Q VLAN Tag (4 bytes, inserted into Ethernet frame after Src MAC):**

```
+------+-----+---+---+-------+
| TPID | PCP | DEI | VID    |
| 0x8100 | 3b | 1b | 12 bits|
+------+-----+---+---+-------+
```

- **TPID:** Tag Protocol Identifier (0x8100 for 802.1Q).
- **PCP:** Priority Code Point (QoS).
- **DEI:** Drop Eligible Indicator.
- **VID:** VLAN ID (1–4094).

**Port Types:**
- **Access Port:** Carries traffic for a single VLAN. Frames are untagged when sent to end devices.
- **Trunk Port:** Carries traffic for multiple VLANs simultaneously using 802.1Q tagging.
- **Native VLAN:** The VLAN whose frames are sent untagged on a trunk port. Default is VLAN 1 on Cisco devices.

### 1.6 Why Layer 2 is Inherently Insecure

Layer 2 protocols were designed in an era where physical security was assumed and network participants were trusted. Several design decisions create fundamental security weaknesses:

1. **No Authentication:** Ethernet has no built-in mechanism to verify the identity of a sender.
2. **MAC Address is Software-Configurable:** MACs can be trivially changed in software, breaking any security assumption that relies on MAC identity.
3. **Broadcast-Heavy Protocols:** ARP, DHCP, and STP rely on broadcasts that all hosts on the segment receive.
4. **Implicit Trust:** Switches trust the CAM table updates they receive from any connected port.
5. **Limited Visibility:** Network security tools typically focus on Layer 3 and above. L2 attacks may go undetected.
6. **Protocol Plaintext:** Most L2 control protocols (ARP, CDP, STP BPDUs) carry no cryptographic protection.

## 2. The Layer 2 Threat Landscape

### 2.1 Attack Categories

Layer 2 attacks can be grouped into the following categories:

| Category | Goal | Primary Protocols Targeted |
|---|---|---|
| **Eavesdropping** | Intercept traffic not destined for attacker | Ethernet switching, ARP |
| **Impersonation / Spoofing** | Pretend to be another device | ARP, MAC, DHCP, STP |
| **Man-in-the-Middle (MitM)** | Intercept and possibly modify traffic | ARP, DHCP, STP |
| **Denial of Service (DoS)** | Disrupt network availability | CAM, STP, DHCP |
| **VLAN Hopping** | Access traffic in unauthorized VLANs | 802.1Q, DTP, VTP |
| **Reconnaissance** | Gather information about the network | CDP, LLDP, ARP |
| **Privilege Escalation** | Gain elevated access to network resources | 802.1X, STP |

### 2.2 Attacker Positioning

Layer 2 attacks require the attacker to be on the same Layer 2 segment (broadcast domain) as the target. This can be achieved through:

- **Physical access:** Plugging into a network port.
- **Wireless access:** Connecting to a Wi-Fi network.
- **Compromised host:** A malware-infected machine on the segment acts as a pivot.
- **Social engineering:** Tricking an employee into connecting to a rogue device.
- **Supply chain compromise:** Malicious hardware inserted into the network.
- **Insider threat:** A malicious employee or contractor with legitimate access.

### 2.3 The Insider Threat Dimension

Layer 2 attacks are particularly relevant in insider threat scenarios because corporate networks often have minimal east-west (lateral) L2 security controls. An employee with legitimate network access can:

- Conduct ARP poisoning to intercept colleagues' traffic.
- Perform DHCP starvation to deny service.
- Execute VLAN hopping to access restricted segments.
- Eavesdrop on unencrypted legacy protocols.

## 3. MAC Address Attacks

### 3.1 MAC Flooding (CAM Table Overflow)

#### How It Works

The CAM table in a switch has a fixed memory capacity — typically capable of storing between 8,000 and 128,000 MAC address entries depending on the hardware. A **MAC flooding attack** exploits this limitation by injecting enormous volumes of Ethernet frames, each with a different (spoofed) source MAC address.

**Attack mechanics:**

1. The attacker generates frames with randomized source MAC addresses at high speed.
2. The switch learns each fake MAC and stores it in the CAM table.
3. The CAM table fills to capacity (overflow state).
4. New frames with unknown destination MACs can no longer be learned correctly.
5. The switch enters **"fail-open" mode**: it floods all frames out all ports in the VLAN, effectively behaving like a hub.
6. The attacker's NIC, now in promiscuous mode, captures all flooded unicast traffic.

**Traffic volume required:** A standard switch CAM table can be overflowed within seconds using tools like `macof` (part of `dsniff`), which can generate approximately 155,000 fake MAC entries per minute.

#### Impact

- **Confidentiality breach:** The attacker can sniff unicast traffic of any host on the flooded VLAN.
- **Performance degradation:** Flooding consumes significant CPU and bandwidth on the switch.
- **Cascading effect:** If the flooded switch is upstream, downstream switches may also be affected.

#### Detection Indicators

- Sudden spike in MAC address table entries.
- Increased broadcast/flooded traffic on switch ports.
- Switch CPU utilization spikes.
- Logs showing rapid MAC address learning events.

#### Mitigation: Port Security

**Port Security** is the primary defense against MAC flooding. It limits the number of MAC addresses allowed on a switch port.

**Cisco IOS configuration example:**

```
! Enable port security on an access port
interface FastEthernet0/1
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 3
 switchport port-security mac-address sticky
 switchport port-security violation restrict
```

**Violation Modes:**

| Mode | Action on Violation | Sends SNMP Trap | Increments Violation Counter | Disables Port |
|---|---|---|---|---|
| **Protect** | Drops violating frames silently | No | No | No |
| **Restrict** | Drops frames, logs, and alerts | Yes | Yes | No |
| **Shutdown** (default) | Err-disables the port | Yes | Yes | Yes |

**MAC Address Learning Modes:**
- **Static:** Manually configured MAC addresses.
- **Dynamic:** Learned automatically, lost on reboot.
- **Sticky:** Learned dynamically and written to running config (persistent).

**Best Practice Recommendations:**
- Set maximum to the minimum number of expected devices (typically 1 for end-user access ports).
- Use `violation restrict` or `violation shutdown` depending on policy.
- Use `sticky` learning to persist legitimate MAC addresses.
- Monitor violation counters via SNMP.
- Combine with 802.1X for identity-based authentication.

### 3.2 MAC Spoofing

#### How It Works

MAC addresses, while burned into NIC hardware, can be overridden in software on virtually every modern operating system. An attacker changes their NIC's MAC address to impersonate another device. This enables:

- **Bypassing MAC-based access controls:** Some networks use MAC filtering as a security mechanism. Spoofing a whitelisted MAC bypasses this.
- **Bypassing port security:** If the attacker knows a legitimate device's MAC, they can spoof it to gain access.
- **IP theft:** Claiming another host's MAC allows the attacker to intercept traffic destined for that host.
- **Evading detection:** Changing MAC address makes it harder to track an attacker's movements.
- **Bypassing 802.1X if MAC-based authentication fallback is enabled.**

**Changing MAC on Linux:**
```bash
ip link set dev eth0 down
ip link set dev eth0 address AA:BB:CC:DD:EE:FF
ip link set dev eth0 up
```

**Changing MAC on Windows:**
Can be done via Device Manager → Network Adapter Properties → Advanced → Network Address.

#### Impact

- Authentication bypass.
- Traffic interception.
- Accountability evasion.
- DHCP exhaustion if combined with automated MAC generation.

#### Mitigation

- **802.1X Port-Based Network Access Control (PNAC):** Provides cryptographic identity authentication independent of MAC address.
- **Dynamic ARP Inspection (DAI):** Validates ARP packets against a DHCP snooping binding table that maps IP to MAC to port.
- **DHCP Snooping:** Creates the binding table used by DAI and prevents rogue DHCP servers.
- **Port Security with Sticky MAC:** Limits which MACs can be used on a port.
- **Network Access Control (NAC):** Combines multiple authentication mechanisms.
- **Avoid relying solely on MAC filtering** as a security control — it is trivially defeated.

## 4. VLAN Attacks

### 4.1 VLAN Hopping — Double Tagging

#### How It Works

**Double tagging** is a technique that exploits the behavior of 802.1Q native VLANs to send frames into a VLAN the attacker is not authorized to access. This is a **one-way attack** — the attacker can send frames into the target VLAN, but cannot receive responses (making it useful for DoS or blind injection, not MitM).

**Attack Prerequisites:**
- The attacker must be connected to an access port in the **native VLAN** of a trunk link.
- The target VLAN must be allowed on that trunk.

**Step-by-step mechanics:**

1. The attacker crafts a frame with **two 802.1Q VLAN tags**:
   - **Outer tag:** The native VLAN of the trunk (e.g., VLAN 1).
   - **Inner tag:** The target VLAN the attacker wants to reach (e.g., VLAN 100).

2. The attacker sends this double-tagged frame to the first switch.

3. **First switch processing:** The switch receives the frame on an access port in the native VLAN. When it forwards the frame onto the trunk port, it strips the outer tag (native VLAN tag) because native VLAN frames are sent untagged. The inner tag remains.

4. **Second switch processing:** The next switch receives the frame and sees only the inner VLAN 100 tag. It forwards the frame into VLAN 100 as if it legitimately belongs there.

5. The frame arrives in VLAN 100 — a VLAN the attacker was never authorized to access.

**Why the attack is unidirectional:** Any response from the VLAN 100 target is addressed back to the attacker's MAC in VLAN 100. The target switch will try to send it back through the trunk on VLAN 100. The first switch will forward it to the VLAN 100 access port — but the attacker is not on VLAN 100, so the response never arrives.

#### Impact

- Sending malicious traffic (DoS packets, exploit payloads) into restricted VLANs.
- Amplifying attacks (e.g., sending a crafted packet to a broadcast address in a restricted VLAN to trigger responses from many hosts).

#### Mitigation

1. **Change the native VLAN to an unused, dedicated VLAN** (not used for any host traffic):
   ```
   interface GigabitEthernet0/1
    switchport trunk native vlan 999
   ```
   Ensure no access ports are assigned to VLAN 999.

2. **Explicitly tag the native VLAN on trunks** (Cisco proprietary):
   ```
   vlan dot1q tag native
   ```
   This causes the switch to send native VLAN frames with an 802.1Q tag, breaking the double-tagging exploit.

3. **Prune VLANs on trunk ports** to carry only the VLANs that are explicitly needed:
   ```
   switchport trunk allowed vlan 10,20,30
   ```

4. **Avoid using VLAN 1 as the native VLAN** — it is the default and thus the most targeted.

5. **Convert unused trunk ports to access ports**.

### 4.2 VLAN Hopping — DTP Exploitation (Switch Spoofing)

#### How It Works

**DTP (Dynamic Trunking Protocol)** is a Cisco proprietary Layer 2 protocol that allows switches to automatically negotiate trunk links. When a switch port is in `dynamic auto` or `dynamic desirable` mode, it will negotiate with the connected device to form a trunk.

An attacker exploits this by connecting a rogue device (or running software like `Yersinia`) that sends DTP negotiation frames, causing the switch port to transition from access mode to trunk mode. Once the attacker has established a trunk link, their device receives traffic from **all VLANs** allowed on the trunk.

**DTP Port Modes:**

| Mode | Behavior |
|---|---|
| `access` | Never trunks, ignores DTP frames |
| `trunk` | Always trunks, sends DTP frames |
| `dynamic auto` | Passively waits for DTP from peer — vulnerable |
| `dynamic desirable` | Actively sends DTP to negotiate trunk — vulnerable |
| `nonegotiate` | Trunks but does not send/respond to DTP frames |

**Attack flow:**
1. Attacker connects to a switch port in `dynamic auto` or `dynamic desirable` mode.
2. Attacker sends DTP frames claiming to be a switch desiring a trunk.
3. Switch port transitions to trunk mode.
4. Attacker now receives frames from all VLANs on the trunk.

**Tool:** Yersinia, a Layer 2 protocol attack framework, can automate this with `yersinia -G` (GUI) or `yersinia dtp -attack 1`.

#### Impact

- Full access to all VLAN traffic carried on the trunk.
- Effectively bypasses VLAN segmentation.

#### Mitigation

1. **Explicitly disable DTP and set ports to access mode** on all non-trunk ports:
   ```
   interface range FastEthernet0/1 - 24
    switchport mode access
    switchport nonegotiate
   ```

2. **For legitimate trunk ports, explicitly set trunk mode and disable negotiation:**
   ```
   interface GigabitEthernet0/1
    switchport mode trunk
    switchport nonegotiate
   ```

3. **Prune allowed VLANs** on trunk ports.

4. **Physically secure switch ports** — disable unused ports and place them in a quarantine VLAN.

5. **Use 802.1X on access ports** to authenticate devices before allowing any network access.

## 5. Spanning Tree Protocol (STP) Attacks

### 5.1 STP Overview

**IEEE 802.1D Spanning Tree Protocol (STP)** prevents Layer 2 loops in networks with redundant switch links. Loops in a switched network cause **broadcast storms** — packets loop endlessly, consuming all available bandwidth and crashing the network within seconds.

**STP Operation:**
1. Switches elect a **Root Bridge** — the switch with the lowest **Bridge ID** (priority + MAC address).
2. All other switches calculate the shortest path to the Root Bridge.
3. Ports are placed in one of four states: Blocking, Listening, Learning, Forwarding.
4. Redundant paths have their ports placed in **Blocking** state to prevent loops.
5. Switches communicate via **BPDUs (Bridge Protocol Data Units)**.

**BPDU Frame Contents:**
- Protocol ID, Version, BPDU Type
- Bridge ID (Priority + MAC)
- Root Bridge ID
- Root Path Cost
- Port ID
- Timers (Hello, Max Age, Forward Delay)

**STP Timers (default):**
- **Hello:** 2 seconds (frequency of BPDU transmission)
- **Max Age:** 20 seconds (time before BPDU is considered stale)
- **Forward Delay:** 15 seconds (time spent in Listening and Learning states)
- **Convergence:** Up to 50 seconds for STP to converge after a topology change

**STP Variants:**
- **STP (802.1D):** Original, slow convergence.
- **RSTP (802.1w — Rapid STP):** Much faster convergence (sub-second in ideal conditions).
- **MSTP (802.1s — Multiple STP):** Multiple spanning tree instances for groups of VLANs.
- **PVST+ (Cisco):** Per-VLAN STP, one STP instance per VLAN.
- **Rapid PVST+:** Cisco's per-VLAN implementation of RSTP.

### 5.2 Root Bridge Takeover

#### How It Works

Any device connected to the network can send BPDU frames claiming to be the Root Bridge. Because STP has no authentication mechanism, switches will accept and react to any received BPDU.

**Attack mechanics:**

1. The attacker connects to the network and sends forged BPDUs with:
   - **Bridge Priority:** Set to 0 (the minimum possible — guarantees election win).
   - **Bridge MAC:** Set to the attacker's device MAC.
   
2. Legitimate switches receive the BPDUs and determine that the attacker has a lower Bridge ID than the current Root Bridge.

3. An **STP re-election** occurs. Switches recalculate their paths, and the attacker's device becomes the new Root Bridge.

4. All traffic now flows through the attacker's device on the path to the Root Bridge, enabling passive **MitM eavesdropping** of all inter-VLAN traffic that passes through.

**Attack tool:** `yersinia -attack stp` can automate Root Bridge hijacking.

#### Impact

- **Man-in-the-Middle:** All traffic on the segment that routes through the attacker becomes visible.
- **Network Instability:** Constant BPDU injection causes continuous STP reconvergence, creating network disruption.
- **Broadcast Storm (intentional):** If the attacker disables their STP instance after taking Root Bridge role, loops can form.

### 5.3 STP DoS — BPDU Flooding

#### How It Works

An attacker continuously injects BPDUs claiming different Root Bridge identities. Each new BPDU triggers STP topology change processing on all switches. The constant reconvergence:

- Keeps switch CPU busy processing STP.
- Causes ports to cycle through states, dropping traffic.
- Can effectively cause a network-wide outage.

### 5.4 STP Manipulation for Traffic Redirection

Beyond simple Root Bridge takeover, a sophisticated attacker can:

- **Selectively manipulate path costs** to redirect specific traffic flows through attacker-controlled segments.
- **Send Topology Change Notifications (TCN)** to flush CAM tables on switches (causing flooding) without winning the Root Bridge election.

A **TCN attack** sends rapid BPDU Topology Change Notifications, causing switches to reduce their MAC address aging timers and flush CAM tables frequently, resulting in sustained unicast flooding.

### 5.5 STP Mitigations

#### PortFast

**PortFast** skips the Listening and Learning states for access ports, immediately placing them in Forwarding. It is **only for access ports** connected to end devices (not switches).

```
interface FastEthernet0/1
 spanning-tree portfast
```

Or globally for all access ports:
```
spanning-tree portfast default
```

#### BPDU Guard

**BPDU Guard** shuts down a PortFast-enabled port if it receives any BPDU. This prevents rogue switches or devices from participating in STP.

```
interface FastEthernet0/1
 spanning-tree portfast
 spanning-tree bpduguard enable
```

Or globally (applied to all PortFast-enabled ports):
```
spanning-tree portfast bpduguard default
```

When BPDU Guard triggers, the port goes into **err-disabled** state. To recover (after removing the threat):
```
interface FastEthernet0/1
 shutdown
 no shutdown
```

Or configure automatic recovery:
```
errdisable recovery cause bpduguard
errdisable recovery interval 300
```

#### BPDU Filter

**BPDU Filter** prevents a port from sending or receiving BPDUs. It effectively removes a port from STP participation. Use with caution — on non-PortFast ports, it can cause loops.

```
interface FastEthernet0/1
 spanning-tree portfast
 spanning-tree bpdufilter enable
```

**Important:** BPDU Guard is preferred over BPDU Filter. BPDU Filter on a non-PortFast port is dangerous because it suppresses BPDUs that would otherwise prevent loops.

#### Root Guard

**Root Guard** prevents a port from becoming a Root Port. If a superior BPDU is received on a Root Guard-enabled port, the port is placed in **root-inconsistent** state (blocking), protecting the current Root Bridge.

```
interface GigabitEthernet0/1
 spanning-tree guard root
```

Apply Root Guard to all ports that **should never receive BPDUs from a Root Bridge** — i.e., all downlink ports facing end users and access switches. Never apply to uplink ports facing the legitimate root.

#### Loop Guard

**Loop Guard** prevents **unidirectional link failures** from causing loops. If a port that should be receiving BPDUs stops receiving them (due to a unidirectional failure), Loop Guard places it in **loop-inconsistent** state rather than transitioning it to Forwarding.

```
interface GigabitEthernet0/1
 spanning-tree guard loop
```

Or globally:
```
spanning-tree loopguard default
```

**Note:** Root Guard and Loop Guard are mutually exclusive on the same port.

#### Storm Control

**Storm Control** limits the rate of broadcast, multicast, or unknown unicast traffic on a port. If traffic exceeds the threshold, the port either drops traffic or shuts down.

```
interface FastEthernet0/1
 storm-control broadcast level 20.00
 storm-control multicast level 10.00
 storm-control action shutdown
```

Thresholds are expressed as a percentage of available bandwidth. The `action shutdown` option err-disables the port if the threshold is exceeded.

#### Summary of STP Security Features

| Feature | Protects Against | Applied To |
|---|---|---|
| PortFast | Slow convergence (not security) | Access ports |
| BPDU Guard | Rogue switches, STP manipulation | PortFast ports (access) |
| BPDU Filter | BPDU leakage | Edge ports (use carefully) |
| Root Guard | Root Bridge hijacking | Ports facing access layer |
| Loop Guard | Unidirectional link failures | Uplinks and ring topologies |
| Storm Control | Broadcast storms, DoS | All ports |

## 6. ARP Attacks

### 6.1 ARP Protocol Overview

The **Address Resolution Protocol (ARP)** — defined in RFC 826 (1982) — resolves IPv4 addresses to MAC addresses within a Layer 2 segment. ARP is entirely stateless and has no authentication mechanism.

**ARP Message Types:**
- **ARP Request:** Broadcast frame asking "Who has IP 192.168.1.1? Tell 192.168.1.50."
- **ARP Reply:** Unicast response "I have 192.168.1.1. My MAC is AA:BB:CC:DD:EE:FF."

**The ARP Cache:**
All hosts maintain a local ARP cache that maps IP addresses to MAC addresses. Cache entries expire after a timeout (typically 60–300 seconds on different OSes).

**Critical Design Flaw:** ARP is **completely stateless and unauthenticated**. A host will accept an ARP Reply even if it never sent an ARP Request for that IP — this is called a **Gratuitous ARP (GARP)**. This enables ARP cache poisoning.

**Gratuitous ARP:** An unsolicited ARP Reply where the sender announces its own IP-to-MAC mapping. Used legitimately for:
- Detecting IP conflicts at boot.
- Updating caches after IP or MAC changes.
- NIC failover in high-availability configurations.

**Maliciously**, GARPs are used to poison ARP caches.

### 6.2 ARP Cache Poisoning (ARP Spoofing)

#### How It Works

**ARP poisoning** is one of the most powerful and commonly used Layer 2 attacks. By sending forged ARP Replies, an attacker can corrupt the ARP caches of target hosts, causing them to send their traffic to the attacker's MAC address instead of the intended destination.

**Man-in-the-Middle Attack via ARP Poisoning:**

**Scenario:** Attacker wants to intercept traffic between Host A (192.168.1.10) and the Gateway (192.168.1.1).

**Normal state:**
- Host A's ARP cache: `192.168.1.1 → AA:BB:CC:DD:EE:01` (Gateway's real MAC)
- Gateway's ARP cache: `192.168.1.10 → AA:BB:CC:DD:EE:10` (Host A's real MAC)

**Attack steps:**

1. Attacker sends a forged ARP Reply to **Host A**:
   - "192.168.1.1 is at AA:BB:CC:DD:EE:FF" (attacker's MAC)
   - Host A updates its cache: `192.168.1.1 → AA:BB:CC:DD:EE:FF`

2. Attacker sends a forged ARP Reply to the **Gateway**:
   - "192.168.1.10 is at AA:BB:CC:DD:EE:FF" (attacker's MAC)
   - Gateway updates its cache: `192.168.1.10 → AA:BB:CC:DD:EE:FF`

3. Now:
   - Host A sends traffic destined for 192.168.1.1 (Gateway) → goes to attacker.
   - Gateway sends traffic destined for 192.168.1.10 (Host A) → goes to attacker.

4. The attacker, with **IP forwarding enabled**, relays traffic between both parties. Neither host detects anything wrong — the attacker is transparent.

**ARP poisoning tools:**
- `arpspoof` (dsniff suite): `arpspoof -i eth0 -t 192.168.1.10 192.168.1.1`
- `ettercap`: Full-featured MitM framework with ARP poisoning.
- `bettercap`: Modern, modular network attack tool with ARP module.
- `Scapy`: Python library for crafting custom ARP frames.

**What an attacker can do once in-path:**
- **Passive sniffing:** Capture all unencrypted traffic (HTTP, FTP, Telnet, POP3, etc.).
- **SSL stripping:** Downgrade HTTPS connections to HTTP using tools like `sslstrip`.
- **Credential theft:** Capture usernames, passwords, session cookies.
- **Session hijacking:** Steal authenticated sessions.
- **DNS spoofing:** Modify DNS responses to redirect users to malicious sites.
- **Malware injection:** Inject malicious code into HTTP responses.
- **Traffic manipulation:** Modify data in transit (e.g., alter file downloads).

### 6.3 ARP DoS

Rather than performing MitM, an attacker can use ARP poisoning to create a **denial-of-service** condition:

- **Black-holing traffic:** Poison the gateway's ARP cache with a non-existent MAC. All traffic from the gateway to the victim is sent to the null MAC and dropped.
- **Misdirection:** Point victim traffic at a non-existent host.

### 6.4 ARP Scanning / Reconnaissance

ARP is also used for passive and active network reconnaissance:

- **ARP Sweep:** Send ARP Requests for every IP in a subnet to discover live hosts. ARP-based scanning is more reliable than ICMP ping because firewalls often block ICMP but cannot block ARP on the local segment.
- Tool: `arp-scan -l` (local network scan), `nmap -PR` (ARP ping scan).

### 6.5 ARP Mitigations

#### Dynamic ARP Inspection (DAI)

**DAI** is the primary hardware-level defense against ARP attacks. It validates ARP packets against a trusted **DHCP snooping binding table**.

**How DAI works:**

1. Ports are designated as **trusted** (uplinks to routers, other switches) or **untrusted** (user access ports).
2. ARP packets arriving on **untrusted ports** are intercepted and validated against the DHCP snooping binding table.
3. The binding table contains: IP address ↔ MAC address ↔ VLAN ↔ Switch Port (learned from DHCP transactions).
4. If an ARP packet's IP-to-MAC mapping **matches the binding table**, it is forwarded.
5. If it does **not match** (forged ARP), it is **dropped** and logged.

**Cisco IOS DAI configuration:**

```
! Enable DHCP snooping first (required for DAI binding table)
ip dhcp snooping
ip dhcp snooping vlan 10,20,30

! Enable Dynamic ARP Inspection
ip arp inspection vlan 10,20,30

! Configure trusted interfaces (uplinks to routers, DHCP servers)
interface GigabitEthernet0/24
 ip arp inspection trust
 ip dhcp snooping trust

! (Optional) Set ARP rate limiting on untrusted ports
ip arp inspection limit rate 100
```

**DAI ARP ACLs (for static IP hosts):**

Since DAI relies on DHCP snooping, statically configured hosts (servers, routers) may not have entries in the binding table. ARP ACLs provide static entries:

```
arp access-list STATIC_HOSTS
 permit ip host 192.168.1.1 mac host AA:BB:CC:DD:EE:01
 permit ip host 192.168.1.2 mac host AA:BB:CC:DD:EE:02

ip arp inspection filter STATIC_HOSTS vlan 10
```

#### Host-Based ARP Defenses

- **Static ARP Entries:** Permanently map critical IP addresses (gateway) to MAC addresses on hosts:
  ```
  arp -s 192.168.1.1 AA:BB:CC:DD:EE:01   # Linux/Windows
  ```
  Limitation: Labor-intensive to maintain; entries may be overwritten by OS in some cases.

- **ArpWatch / XArp:** Tools that monitor ARP activity and alert on suspicious changes.
- **Encrypted communications:** Even if ARP poisoning succeeds, traffic encrypted with TLS/SSH cannot be read or easily modified.

## 7. DHCP Attacks

### 7.1 DHCP Protocol Overview

The **Dynamic Host Configuration Protocol (DHCP)** — RFC 2131 — allows hosts to automatically obtain IP configuration from a server. The DHCP process (DORA):

1. **Discover:** Client broadcasts a DHCP Discover on the local segment.
2. **Offer:** Server responds with an IP address offer.
3. **Request:** Client broadcasts a DHCP Request accepting the offer.
4. **Acknowledge:** Server acknowledges and finalizes the lease.

**DHCP is entirely unauthenticated.** Any device can:
- Respond to DHCP Discover messages (rogue server).
- Exhaust the DHCP pool (starvation).
- Request fraudulent leases.

### 7.2 DHCP Starvation

#### How It Works

**DHCP starvation** exhausts a server's available IP address pool by sending enormous numbers of DHCP Discover and Request messages with randomized source MAC addresses, each claiming to be a different client.

**Attack steps:**

1. Attacker generates thousands of DHCP Discover frames, each with a unique random source MAC (in the CHADDR field).
2. The DHCP server responds with offers and allocates IP leases for each "client."
3. The DHCP pool becomes exhausted.
4. Legitimate clients can no longer obtain an IP address — **denial of service**.

**This is often a precursor to a rogue DHCP server attack:** after starving the legitimate server, the attacker starts their own rogue DHCP server to answer legitimate client requests with attacker-controlled configuration.

**Attack tool:** `dhcpstarv`, `Yersinia`, `gobbler`.

### 7.3 Rogue DHCP Server

#### How It Works

A **rogue DHCP server** responds to DHCP Discover broadcasts before the legitimate server, providing clients with malicious network configuration:

- **Attacker as Default Gateway:** The client uses the attacker's IP as its default gateway, routing all traffic through the attacker. Full MitM on all outbound traffic.
- **Malicious DNS Server:** Clients use the attacker's DNS server, enabling DNS spoofing.
- **Short Lease Times:** Causes frequent DHCP renewals, maintaining attacker control.
- **Wrong Subnet Mask:** Can cause routing failures or force traffic through unexpected paths.

**The rogue DHCP server wins the race condition** because:
- It is on the same local segment (no relay agent hop).
- It may respond faster than the legitimate server.
- After DHCP starvation of the legitimate server, the rogue is the only remaining option.

**Attack tools:** `dnsmasq` (configured as DHCP server), Metasploit DHCP module, `Yersinia`.

### 7.4 DHCP Mitigations

#### DHCP Snooping

**DHCP Snooping** is a switch security feature that acts as a DHCP firewall:

**How it works:**
- Ports are designated **trusted** (connected to legitimate DHCP servers or uplinks) or **untrusted** (connected to end users).
- DHCP **server messages** (OFFER, ACK, NAK) received on **untrusted ports are dropped** — preventing rogue DHCP servers.
- DHCP **client messages** on untrusted ports are validated (e.g., checking that the CHADDR field matches the source MAC).
- All successful DHCP exchanges build the **DHCP snooping binding table** (used by DAI and IP Source Guard).

**DHCP Snooping binding table entry format:**
```
MAC Address     | IP Address  | Lease (sec) | VLAN | Interface
AA:BB:CC:DD:EE:FF | 192.168.1.10 | 86400    | 10   | Fa0/5
```

**Cisco IOS DHCP Snooping configuration:**

```
! Enable DHCP snooping globally
ip dhcp snooping

! Enable per VLAN
ip dhcp snooping vlan 10,20,30

! Optional: Disable Option 82 insertion (can cause issues with some servers)
no ip dhcp snooping information option

! Trust legitimate uplinks and DHCP server ports
interface GigabitEthernet0/24
 ip dhcp snooping trust

! Optionally rate-limit DHCP on untrusted ports (mitigates starvation)
interface range FastEthernet0/1 - 23
 ip dhcp snooping limit rate 15
```

**Rate limiting DHCP** on untrusted ports (15 packets/second default) prevents starvation attacks. Exceeding the limit puts the port in err-disabled state.

#### IP Source Guard (IPSG)

**IP Source Guard** extends DHCP Snooping by validating **all IP traffic** (not just DHCP) against the binding table. It prevents:

- IP address spoofing.
- Using a statically configured IP different from the DHCP-assigned one.

**Configuration:**

```
interface FastEthernet0/1
 ip verify source
```

With IPSG enabled, the switch creates a **per-port VLAN ACL** allowing only the IP-to-MAC binding from the DHCP snooping table. All other IP traffic is dropped.

Optionally include MAC address validation:
```
ip verify source port-security
```

## 8. CDP / LLDP Attacks

### 8.1 CDP (Cisco Discovery Protocol)

**CDP** is a Cisco proprietary Layer 2 protocol that runs on all Cisco devices by default. It allows Cisco devices to discover directly connected Cisco neighbors and share information. CDP packets are sent every 60 seconds as multicasts to `01:00:0C:CC:CC:CC`.

**Information disclosed by CDP:**
- Device hostname
- Device type and model
- IOS version and software version
- IP addresses
- Interface names and types
- Native VLAN
- Duplex settings
- VTP domain name
- Power-over-Ethernet capability

This information is invaluable to an attacker for:
- **Network mapping:** Understanding the topology.
- **Targeted exploitation:** Knowing the exact IOS version enables targeted CVE exploitation.
- **Credential attacks:** Knowing the device model narrows default credential guessing.
- **VLAN information:** Helps with VLAN hopping planning.

### 8.2 LLDP (Link Layer Discovery Protocol)

**LLDP (IEEE 802.1AB)** is the vendor-neutral equivalent of CDP. It is supported by all major vendors and is increasingly enabled by default. LLDP sends its advertisements to multicast `01:80:C2:00:00:0E`.

LLDP extensions include:
- **LLDP-MED (Media Endpoint Discovery):** Shares VLAN assignments and PoE for IP phones.
- **IEEE 802.1 extensions:** VLAN names, port VLAN IDs, protocol VLANs.
- **IEEE 802.3 extensions:** MAC/PHY capabilities, link aggregation.

### 8.3 CDP/LLDP Attacks

#### Information Disclosure

An attacker connected to any switch port passively receives CDP/LLDP advertisements and collects detailed network topology information without sending a single packet, making it completely passive and undetectable.

**Tools:** `wireshark` (filter: `cdp` or `lldp`), `cdpsnarf`, `Yersinia`.

#### CDP Flooding

Using tools like Yersinia, an attacker can flood the switch with thousands of fake CDP neighbor advertisements. This can:
- Consume switch CPU (processing CDP updates).
- Exhaust CDP neighbor table memory.
- Potentially crash or destabilize older switch firmware.

#### VoIP VLAN Exploitation

CDP and LLDP-MED are used by IP phones to discover their assigned Voice VLAN (the VLAN designated for VoIP traffic). An attacker with software that emulates a Cisco IP phone can receive the Voice VLAN ID via CDP/LLDP-MED and switch their device into that VLAN, potentially accessing the voice network — which is often less secured than the data network and may have fewer security controls.

### 8.4 CDP/LLDP Mitigations

1. **Disable CDP globally** if not needed:
   ```
   no cdp run
   ```

2. **Disable CDP per interface** (more surgical — disable on user-facing ports):
   ```
   interface FastEthernet0/1
    no cdp enable
   ```

3. **Disable LLDP:**
   ```
   no lldp run
   
   interface FastEthernet0/1
    no lldp transmit
    no lldp receive
   ```

4. **Keep CDP/LLDP only on infrastructure links** (switch-to-switch, switch-to-router) where it provides genuine operational value.

5. **Implement 802.1X** on user ports to prevent unauthorized devices from connecting and receiving CDP/LLDP data.

6. **Separate Voice VLANs** with proper ACLs and security controls. Use 802.1X with RADIUS-based VLAN assignment instead of relying solely on CDP/LLDP for voice VLAN discovery.

## 9. 802.1X and Authentication Attacks

### 9.1 802.1X Overview

**IEEE 802.1X** (Port-Based Network Access Control — PNAC) provides a framework for authenticating devices before granting them network access at the Layer 2 level. It uses the **EAP (Extensible Authentication Protocol)** over LAN (EAPOL).

**Three-party model:**
- **Supplicant:** The client device requesting access (e.g., laptop).
- **Authenticator:** The network access device (switch or wireless AP) that controls port access.
- **Authentication Server:** A RADIUS server (e.g., FreeRADIUS, Cisco ISE, Microsoft NPS) that validates credentials.

**802.1X Authentication Flow:**

```
Supplicant          Authenticator (Switch)         Auth Server (RADIUS)
    |                       |                               |
    |<-- EAP-Request/ID ----|                               |
    |--- EAP-Response/ID -->|                               |
    |                       |---- RADIUS Access-Request --->|
    |                       |<--- RADIUS Access-Challenge --|
    |<-- EAP-Request -------|                               |
    |--- EAP-Response ----->|                               |
    |                       |---- RADIUS Access-Request --->|
    |                       |<--- RADIUS Access-Accept -----|
    |<-- EAP-Success -------|                               |
    |                       |                               |
    [Port moves to Authorized State]
```

**EAP Methods (common):**
- **EAP-TLS:** Certificate-based (both client and server present certificates). Strongest method.
- **PEAP (Protected EAP):** Tunneled EAP — server presents certificate, client authenticates inside TLS tunnel (typically MSCHAPv2).
- **EAP-TTLS:** Similar to PEAP, supports various inner authentication methods.
- **EAP-FAST:** Cisco proprietary, uses Protected Access Credentials (PAC).
- **MD5-Challenge:** Weak, no mutual authentication — should not be used.

### 9.2 802.1X Attacks

#### Identity Theft / Credential Harvesting

**PEAP-MSCHAPv2 attacks:**

1. Attacker sets up a **rogue 802.1X authentication server** (e.g., using `hostapd-wpe` or `FreeRADIUS-WPE`).
2. Client connects and begins PEAP authentication.
3. Client validates the server's TLS certificate — if the client does not strictly validate the server certificate (a common misconfiguration), it accepts the rogue server's certificate.
4. Inside the PEAP TLS tunnel, the client submits MSCHAPv2 credentials.
5. The rogue server captures the NTLMv2 challenge-response.
6. Attacker cracks the NTLMv2 hash offline (using `hashcat`, `John the Ripper`).

**Root cause:** Clients that do not validate the 802.1X server certificate are vulnerable to rogue authentication servers.

#### Identity Spoofing / MAC Bypass

**MAB (MAC Authentication Bypass)** is a fallback mechanism for devices that do not support 802.1X (printers, IP cameras, etc.). The switch uses the device's MAC address as its identity for RADIUS authentication.

**Attack:** An attacker discovers the MAC address of an authorized device (via passive sniffing or physical inspection) and spoofs it to gain network access without 802.1X.

#### Authentication Bypass — EAPOL Flooding

Some older 802.1X implementations can be destabilized by flooding EAPOL packets, causing the authenticator to fail open (granting access) or enter an error state.

#### VLAN Authorization Bypass

**RADIUS VLAN assignment** allows the RADIUS server to dynamically assign a client to a specific VLAN upon successful authentication. An attacker who compromises RADIUS or manipulates RADIUS Attributes (e.g., through a rogue RADIUS proxy) can be placed into a privileged VLAN.

#### Rogue Authentication Server

In environments that use 802.1X without proper server certificate validation:
1. Attacker deploys a rogue AP or rogue switch with an 802.1X authenticator.
2. Clients that roam or connect attempt 802.1X authentication.
3. Rogue server captures credentials.

### 9.3 802.1X Mitigations

1. **Use EAP-TLS with mutual certificate authentication** — eliminates credential harvesting because no passwords are transmitted.

2. **Enforce server certificate validation on all supplicants:**
   - Specify the trusted Root CA.
   - Validate server certificate CN/SAN.
   - Reject self-signed or unknown certificates.

3. **Protect MAB-enabled ports** with additional controls:
   - Restrict MAB to specific, non-sensitive VLANs.
   - Use DHCP snooping and DAI on MAB ports.
   - Implement device profiling (Cisco ISE profiling, Aruba ClearPass).

4. **Implement Network Access Control (NAC)** with posture assessment:
   - Check for OS patch level, antivirus status, firewall status before granting access.

5. **Segment networks by authentication method** — place fully authenticated 802.1X EAP-TLS devices in trusted VLANs, MAB devices in restricted VLANs with ACLs.

6. **Monitor for EAPOL anomalies** — unexpected EAPOL frames, authentication failures, unusual VLAN assignments.

## 10. Switch-Specific Attacks

### 10.1 Telnet / SSH Brute Force Against Switch Management

Network switches are typically managed via Telnet (insecure, plaintext) or SSH. Attacks include:

- **Default credentials:** Many switches ship with default credentials (`admin/admin`, `cisco/cisco`, etc.) that are never changed.
- **Brute force:** Automated attempts against SSH or Telnet using tools like `Hydra`, `Medusa`.
- **Credential stuffing:** Using previously breached credentials.

**Mitigations:**
- Disable Telnet; use SSH version 2 only.
- Use strong, unique passwords or certificate-based SSH authentication.
- Restrict management access to a dedicated management VLAN or out-of-band network.
- Implement AAA (Authentication, Authorization, Accounting) with RADIUS or TACACS+.
- Apply login rate limiting and lockout.
- Use ACLs to restrict management access to authorized IP ranges.

**Cisco IOS hardening for management:**
```
! Disable Telnet
line vty 0 15
 transport input ssh

! SSH version 2 only
ip ssh version 2

! Generate RSA key
crypto key generate rsa modulus 2048

! Login protection
login block-for 120 attempts 3 within 60
login quiet-mode access-class MGMT_ACL
```

### 10.2 SNMP Attacks

**SNMP (Simple Network Management Protocol)** is used to monitor and manage network devices. Vulnerabilities include:

- **SNMPv1/v2c Community String Exposure:** Community strings (`public`, `private`) are transmitted in plaintext. An attacker who sniffs a community string can query the device (read-only) or modify its configuration (read-write).
- **Default Community Strings:** Many devices use `public` and `private` as default community strings.
- **SNMPv3 Credential Attacks:** Brute forcing SNMPv3 authentication credentials.
- **Amplification Attacks:** SNMP can be used in reflection/amplification DDoS attacks.

**Mitigations:**
- Use **SNMPv3 with AuthPriv mode** (authentication + encryption).
- Disable SNMPv1 and SNMPv2c.
- Change default community strings.
- Apply ACLs to restrict SNMP access to authorized management hosts.
- Use read-only SNMP where write access is not required.

### 10.3 TFTP/FTP Configuration Theft

Switches often use TFTP to save and restore configurations. If TFTP is accessible, an attacker can:
- Download the switch configuration file (contains all settings, potentially including password hashes).
- Upload a modified configuration to take control.

**Mitigation:**
- Restrict TFTP access via ACLs.
- Use SCP instead of TFTP for configuration management.
- Enable configuration file encryption.
- Monitor and audit configuration changes.

### 10.4 Management Plane Attacks — CPU Exhaustion

Certain Layer 2 operations are processed by the switch CPU rather than the hardware ASIC:
- STP BPDU processing
- ARP processing
- DHCP processing
- CDP/LLDP processing
- 802.1X EAPOL processing

An attacker who floods any of these protocols can **exhaust switch CPU**, causing:
- Inability to process management traffic (SSH/Telnet sessions drop).
- STP convergence failures.
- DHCP assignment failures.
- Overall switch instability.

**Mitigation — Control Plane Policing (CoPP):**

CoPP limits the rate of traffic directed to the switch CPU, protecting the control plane from flood attacks:

```
! Classify ARP traffic
class-map match-all ARP
 match protocol arp

! Define policy
policy-map COPP_POLICY
 class ARP
  police rate 1000 pps burst 2000 pps
   conform-action transmit
   exceed-action drop

! Apply to control plane
control-plane
 service-policy input COPP_POLICY
```

## 11. Private VLAN Attacks

### 11.1 Private VLAN Overview

**Private VLANs (PVLANs)** — defined in RFC 5517 and IEEE 802.1Q — provide traffic isolation within a VLAN. They are used primarily in hosting/datacenter environments to prevent communication between hosts in the same VLAN (e.g., web servers in a DMZ should not be able to talk to each other directly).

**PVLAN Port Types:**

| Port Type | Description |
|---|---|
| **Promiscuous** | Communicates with all port types (router/firewall uplinks) |
| **Isolated** | Communicates only with promiscuous ports — never with other isolated or community ports |
| **Community** | Communicates with promiscuous ports and other ports in the same community |

### 11.2 Private VLAN Proxy Attack

#### How It Works

An **isolated port** cannot send traffic directly to another isolated port in the same VLAN. However, a **proxy attack** can bypass this:

1. Host A is on an isolated port. Host B is also on an isolated port.
2. Host A crafts a packet addressed to Host B's IP **but with the router (promiscuous port) MAC as the destination MAC**.
3. The switch sees the destination MAC as the router, so it forwards the frame to the router (promiscuous port — which is allowed).
4. The router has a Layer 3 entry for Host B and performs a routing lookup.
5. The router forwards the packet to Host B, using Host B's MAC.
6. Traffic flows from Host A to Host B via the router, bypassing the PVLAN isolation.

This attack relies on the router's L3 routing to forward what should have been isolated Layer 2 traffic.

### 11.3 Private VLAN Mitigation

- **Configure the router/firewall with ACLs** that explicitly deny traffic between hosts in the same PVLAN:
  ```
  ! Deny traffic from isolated VLAN hosts to other isolated VLAN hosts
  ip access-list extended PVLAN_ACL
   deny ip 192.168.50.0 0.0.0.255 192.168.50.0 0.0.0.255
   permit ip any any
  ```

- **Enable PVLAN Edge (Protected Ports)** on switches that do not support full PVLAN:
  ```
  interface FastEthernet0/1
   switchport protected
  ```
  Protected ports cannot send traffic to other protected ports on the same switch.

- **Apply routing policies** to prevent intra-subnet routing where isolation is required.

## 12. VTP (VLAN Trunking Protocol) Attacks

### 12.1 VTP Overview

**VTP (VLAN Trunking Protocol)** is a Cisco proprietary Layer 2 protocol that propagates VLAN database changes across all switches in a VTP domain. A VTP Server switch makes changes; VTP Client switches synchronize automatically.

**VTP Modes:**
- **Server:** Creates, modifies, deletes VLANs. Changes propagate across the domain.
- **Client:** Receives and applies VLAN changes. Cannot make local changes.
- **Transparent:** Passes VTP advertisements but does not participate. Manages its own local VLAN database.
- **Off (VTPv3):** Completely disables VTP.

**VTP Advertisements are sent on trunk links every 300 seconds (summary advertisement) or immediately upon change.**

### 12.2 VTP Attack — VLAN Database Wiping

#### How It Works

VTP uses a **configuration revision number** to determine which switch has the most current VLAN database. The switch with the **highest revision number** is considered authoritative, and all others synchronize to it.

**Attack:**

1. An attacker connects a device (or introduces a rogue switch) to the VTP domain.
2. The attacker configures their device as a VTP Server in the same VTP domain with a **very high revision number** (e.g., 65535).
3. Legitimate switches see the higher revision number and synchronize.
4. If the attacker's VLAN database is empty (or maliciously modified), all switches in the domain overwrite their VLAN databases with the attacker's version.
5. **All VLANs are wiped from the network** — complete network outage.

This attack is particularly dangerous because a misconfigured switch brought from another network can inadvertently cause this if it has a high revision number.

### 12.3 VTP Mitigations

1. **Use VTP version 3** — adds authentication (MD5 or SHA-1 based) and primary server concept, significantly improving security.

2. **Set a VTP password** on all switches:
   ```
   vtp password StrongVTPPassword
   ```

3. **Use VTP Transparent mode** or **VTP Off** on all switches except those that explicitly need to manage VLANs centrally:
   ```
   vtp mode transparent
   ```
   In transparent mode, the switch ignores incoming VTP updates and maintains its own VLAN database.

4. **Before connecting any switch** to the network, reset its VTP revision number:
   - Change VTP mode to Transparent.
   - Change VTP domain name to something temporary.
   - Change back to the original mode.
   - This resets the revision number to 0.

5. **Monitor VTP revision numbers** and alert on unexpected changes.

6. **Restrict trunk ports** — VTP only traverses trunk links. Proper access port configuration prevents unauthorized VTP participation.

## 13. IPv6 Layer 2 Attacks

### 13.1 IPv6 Neighbor Discovery Protocol (NDP)

IPv6 replaces ARP with the **Neighbor Discovery Protocol (NDP)** — RFC 4861. NDP uses ICMPv6 messages and operates at Layer 2/3. Key NDP messages:

| ICMPv6 Type | Message | Purpose |
|---|---|---|
| 133 | Router Solicitation (RS) | Host requests router info |
| 134 | Router Advertisement (RA) | Router announces itself and prefixes |
| 135 | Neighbor Solicitation (NS) | Equivalent of ARP Request |
| 136 | Neighbor Advertisement (NA) | Equivalent of ARP Reply |
| 137 | Redirect | Router redirects host to better next-hop |

NDP suffers from the **same fundamental unauthenticated design flaw as ARP**, enabling analogous attacks.

### 13.2 NDP Spoofing / Neighbor Cache Poisoning

**Identical in concept to ARP poisoning**, but using ICMPv6 Neighbor Advertisement messages. An attacker sends forged NA messages claiming that a target IPv6 address corresponds to the attacker's MAC address, poisoning the neighbor cache of other hosts.

**Attack tool:** `THC-IPv6 suite` (particularly `fake_router6`, `parasite6`), `scapy`.

### 13.3 Rogue Router Advertisement (RA) Attack

**Router Advertisements** are sent by routers to announce:
- Default gateway address.
- Network prefixes (for SLAAC — Stateless Address Autoconfiguration).
- MTU settings.
- DNS server addresses (RFC 8106).

Any host on the segment can send RA messages. An attacker can:

1. Send forged RA messages claiming to be the default IPv6 router.
2. Hosts update their default gateway to the attacker's link-local address.
3. All outbound IPv6 traffic is routed through the attacker — **full MitM**.

**Additional RA attack scenarios:**
- **SLAAC Attack:** Attacker's RA provides a malicious prefix, causing hosts to autoconfigure addresses from it.
- **RA Flooding (DoS):** Continuous RAs with different prefixes cause hosts to constantly add/remove prefixes, consuming CPU.
- **DNS Hijacking via RDNSS:** Attacker's RA advertises a malicious DNS server via RDNSS option (RFC 8106).

### 13.4 DHCPv6 Attacks

Similar to IPv4 DHCP attacks, **DHCPv6** is vulnerable to:
- **Rogue DHCPv6 servers:** Providing malicious DNS, prefixes.
- **DHCPv6 starvation:** Exhausting the address pool.

### 13.5 IPv6 Layer 2 Mitigations

#### RA Guard (IPv6 RA Guard — RFC 6105)

RA Guard prevents unauthorized Router Advertisement messages on untrusted ports. The switch validates RA packets on untrusted ports and drops those coming from non-router ports.

```
! Cisco IOS IPv6 RA Guard
ipv6 nd raguard policy RAGUARD_POLICY
 device-role host

interface FastEthernet0/1
 ipv6 nd raguard attach-policy RAGUARD_POLICY
```

#### IPv6 DHCP Guard

**DHCPv6 Guard** validates DHCPv6 server/relay messages on untrusted ports, blocking rogue DHCPv6 servers.

```
ipv6 dhcp guard policy DHCPV6_GUARD
 device-role client

interface FastEthernet0/1
 ipv6 dhcp guard attach-policy DHCPV6_GUARD
```

#### IPv6 Source Guard

```
interface FastEthernet0/1
 ipv6 source-guard attach-policy DEFAULT
```

Validates that IPv6 source addresses on untrusted ports are bound to the port via SLAAC or DHCPv6 snooping entries.

#### SEND (Secure Neighbor Discovery — RFC 3971)

**SEND** extends NDP with cryptographic protection using RSA-based Cryptographically Generated Addresses (CGA). It provides:
- Address ownership proof.
- Router authorization.
- Message integrity and replay protection.

**Limitation:** SEND has seen limited deployment due to implementation complexity and computational overhead. It is not widely supported on commercial switches.

## 14. Wireless Layer 2 Attacks

### 14.1 Wireless Layer 2 Frame Structure

IEEE 802.11 wireless uses its own Layer 2 frame format, distinct from Ethernet. The 802.11 frame has a more complex header including:
- Frame Control (type, subtype, flags)
- Duration/ID
- Up to four address fields (BSSID, SA, DA, RA)
- Sequence Control
- QoS Control
- Frame Body
- FCS

**802.11 Frame Types:**
- **Management Frames:** Beacons, Probe Requests/Responses, Authentication, Association, Disassociation, Deauthentication.
- **Control Frames:** ACK, RTS, CTS, Block ACK.
- **Data Frames:** Carry actual data payload.

### 14.2 Rogue Access Point

A **rogue AP** is an unauthorized wireless access point connected to the wired network. Types:

- **Malicious Rogue AP:** Deliberately installed by an attacker to provide a backdoor into the network.
- **Accidental Rogue AP:** An employee connects a personal AP to the corporate network.
- **Evil Twin AP:** An attacker sets up an AP with the same SSID/BSSID as a legitimate AP to lure clients.

### 14.3 Evil Twin / Deauthentication Attack

**802.11 Management Frame Attack:**

Original 802.11 management frames (Deauthentication, Disassociation) are **unauthenticated** — any device can send them spoofing the AP's BSSID.

**Attack flow:**

1. Attacker identifies target AP SSID and BSSID.
2. Attacker sends **Deauthentication frames** spoofed from the AP's BSSID to the client (or broadcasts to all clients).
3. Clients disconnect from the legitimate AP.
4. Attacker's **evil twin AP** (with same SSID) is present. Clients automatically reconnect to the stronger signal.
5. Attacker captures the WPA handshake (for cracking) or acts as a transparent MitM gateway.

**Tools:** `aireplay-ng`, `mdk4`, `wifiphisher`.

### 14.4 PMKID Attack

Modern WPA2/WPA3 networks can be attacked via the **PMKID** — a value derivable from the PMK (Pairwise Master Key) that is transmitted in the EAPOL frame during association. Unlike capturing the 4-way handshake (requires a client to connect), PMKID can be obtained without a client being present.

**Attack flow:**
1. Attacker captures PMKID from the AP's EAPOL response.
2. PMKID = HMAC-SHA1-128(PMK, "PMK Name" || AP_MAC || STA_MAC).
3. Attacker brute-forces the pre-shared key offline.

### 14.5 Wireless Layer 2 Mitigations

- **WPA3 with SAE (Simultaneous Authentication of Equals):** Replaces WPA2-PSK's vulnerable 4-way handshake with a zero-knowledge proof (Dragonfly key exchange), resistant to offline dictionary attacks.
- **802.11w (Protected Management Frames — PMF):** Cryptographically protects management frames, preventing deauthentication attacks. Now mandatory in WPA3.
- **Enterprise 802.1X for wireless (WPA2/WPA3-Enterprise):** EAP-TLS with certificates provides per-client authentication and mutual authentication.
- **Wireless Intrusion Prevention System (WIPS):** Detects and blocks rogue APs and attacks.
- **WIDS/WIPS:** Monitor for deauthentication floods, rogue APs, evil twin APs.
- **SSID isolation / Client isolation:** Prevents wireless clients from communicating directly with each other.

## 15. Hardware-Level and Physical Layer Attacks

### 15.1 Passive Optical Tapping

A **fiber tap** or **Ethernet tap** is a physical device inserted into a network link to passively copy traffic. High-quality taps are completely passive (no power required, transparent to the network).

**Types:**
- **Passive optical splitter:** Splits a fiber signal — attacker receives a copy of all traffic.
- **Active Ethernet tap:** Inserts into copper link, copies traffic to a monitoring port.
- **SPAN/port mirroring abuse:** If an attacker gains switch access, they can configure a SPAN (Switched Port Analyzer) session to mirror traffic to their port.

**Mitigation:**
- Physical security — secured network closets, patch panels, cable pathways.
- Fiber sealing with tamper-evident materials.
- Monitor physical switch configurations — alert on unauthorized SPAN configuration changes.
- Optical power monitoring — a passive tap reduces optical power levels (may be detectable).

### 15.2 Unauthorized SPAN / Port Mirroring

SPAN sessions configured by an authorized admin (or an attacker with switch access) mirror all traffic from one or more ports to a monitoring port. This is legitimate for network monitoring but becomes a serious attack vector if misconfigured or abused.

**Mitigation:**
- Audit SPAN configurations regularly.
- Use role-based access control (RBAC) to restrict who can configure SPAN.
- TACACS+ authorization for SPAN configuration commands.
- Alert on SPAN configuration changes via SYSLOG/SNMP.

### 15.3 Switch Port Stealing (Gratuitous ARP-Based)

An attacker sends Gratuitous ARP messages claiming their MAC for a target IP, causing the switch to associate their MAC with the correct port for that IP while also sending continuous GARPs. When the legitimate host sends traffic, the switch may temporarily flood or misdirect frames. This is an alternative method to achieve traffic interception.

### 15.4 USB / Physical Device Drop Attacks

An attacker introduces a rogue device (a small SBC like a Raspberry Pi, or a commercial product like a Hak5 LAN Turtle) by physically plugging it into an Ethernet port. The device may:
- Perform Layer 2 attacks silently.
- Provide remote access via cellular or Wi-Fi.
- Act as a persistent backdoor.

**Mitigation:**
- 802.1X authentication on all switch ports — unauthorized devices cannot access the network.
- Disable unused ports and place them in a blocked VLAN.
- Physical security: locked server rooms, port lockers, tamper-evident seals.
- Monitor for unauthorized MAC addresses and authentication failures.

### 16.2 VLAN Design Best Practices

**Principle: Segment by security zone and communication need.**

| VLAN | Description | Security Level |
|---|---|---|
| User Data VLANs | Regular workstation traffic | Medium |
| Server VLAN | Critical servers | High — restrict inbound |
| Management VLAN | Switch/router management | Highest — restricted to admins |
| Voice VLAN | VoIP traffic | High — separate from data |
| Quarantine VLAN | Failed 802.1X, posture check | Low — limited access |
| Guest VLAN | Visitors, BYOD | Isolated — internet only |
| Native VLAN | Untagged trunk traffic | Unused/dedicated |
| Black Hole VLAN | Disabled ports | No routing, no services |

**Rules:**
- Never use VLAN 1 for user traffic.
- Use a dedicated, non-routed native VLAN.
- Use ACLs or firewall at Layer 3 boundaries between VLANs.
- Prune VLANs on trunk ports — only carry VLANs that are needed.
- Implement private VLANs in hosting/DMZ environments.

### 16.4 Zero Trust Principles Applied to Layer 2

**Zero Trust** extends to Layer 2 through the principle of "never trust, always verify":

1. **Verify identity before access:** 802.1X for all wired and wireless connections. No device accesses the network without authentication.

2. **Least-privilege VLAN assignment:** Devices are placed in the minimum-access VLAN appropriate for their function, not a privileged VLAN.

3. **Microsegmentation:** Even within a VLAN, use private VLANs or ACLs to restrict lateral movement.

4. **Continuous validation:** Device posture is checked periodically, not just at connection time. Non-compliant devices are re-assigned to quarantine VLAN.

5. **Encrypt all traffic:** Even within a trusted segment, encrypt sensitive traffic. Layer 2 attacks succeed in reading data — encryption at Layer 4+ (TLS) or Layer 2 (MACsec) defeats eavesdropping.

### 16.5 MACsec (IEEE 802.1AE)

**MACsec** provides hop-by-hop authenticated encryption at the Layer 2 level. It protects all Ethernet frames between two directly connected MACsec-capable devices.

**Key properties:**
- **Encryption:** AES-128 or AES-256 (GCM mode).
- **Integrity:** Cryptographic message authentication (ICV — Integrity Check Value) appended to each frame.
- **Replay Protection:** Sequence numbers prevent replay attacks.
- **Per-hop:** Encryption is terminated at each switch; traffic is re-encrypted for the next hop.

**MACsec defeats:**
- Passive eavesdropping (traffic is encrypted).
- ARP poisoning MitM (attacker sees only encrypted frames).
- MAC flooding eavesdropping (flooded frames are encrypted).

**MACsec uses MKA (MACsec Key Agreement — IEEE 802.1X-2010)** to negotiate keys, leveraging the existing 802.1X infrastructure.

**Cisco MACsec configuration (switch-to-switch):**
```
! Configure MKA policy
mka policy MKA_POLICY
 macsec-cipher-suite gcm-aes-256

interface GigabitEthernet0/1
 mka pre-shared-key key-chain MKA_KEY_CHAIN
 mka policy MKA_POLICY
 macsec
```

### 17.1 Layer 2 Logging and Alerting

**Key events to log and alert on:**

| Event | Source | Priority |
|---|---|---|
| Port Security Violation | Switch SYSLOG | High |
| BPDU Guard Triggered | Switch SYSLOG | High |
| Root Guard Violation | Switch SYSLOG | High |
| DHCP Snooping Violation | Switch SYSLOG | High |
| DAI ARP Drop | Switch SYSLOG | High |
| Err-Disabled Port | Switch SYSLOG | Medium |
| VTP Revision Number Change | Switch SYSLOG | Critical |
| New MAC Address Learned | Switch SYSLOG/SNMP | Low (baseline) |
| MAC Address Flapping | Switch SYSLOG | High |
| 802.1X Auth Failure | RADIUS Logs | Medium |
| Storm Control Triggered | Switch SYSLOG | Medium |
| Unauthorized SPAN Config | Switch SYSLOG | Critical |
| Management Login Failure | Switch/AAA Logs | Medium |
| Configuration Change | Switch SYSLOG/TACACS+ | High |

### 17.2 MAC Address Flapping Detection

**MAC flapping** occurs when the switch sees the same MAC address on two different ports within a short time. This can indicate:
- A network loop.
- ARP poisoning (attacker claiming another host's MAC).
- A dual-connected device.

Cisco switches log MAC flapping:
```
%SW_MATM-4-MACFLAP_NOTIF: Host 00:11:22:33:44:55 in vlan 10 is flapping between port Gi0/1 and port Gi0/5
```

Set up SNMP traps and SYSLOG forwarding to a SIEM to alert on MAC flapping events.

### 17.3 ARP Anomaly Detection

Monitor ARP traffic for:
- **High-rate ARP Replies without corresponding Requests** — sign of ARP poisoning.
- **ARP for the gateway's IP from unexpected MACs** — gateway impersonation.
- **Rapid changes in IP-to-MAC mappings** — poisoning in progress.
- **ARP storms** — potential scanning or flood attack.

**Detection tools:**
- **ArpWatch:** Monitors ARP tables and alerts on changes. Available on Linux.
- **XArp:** Windows/Linux tool for ARP-based attack detection.
- **SIEM correlation rules:** Alert when a single IP address maps to multiple MACs in a short period.
- **DAI violation logs:** Cisco switches with DAI log every dropped ARP — forward to SIEM.

### 17.4 NetFlow for Layer 2/3 Correlation

While NetFlow operates at Layer 3, Layer 2 anomalies often manifest as Layer 3 anomalies:
- Sudden traffic volume changes.
- Traffic to unexpected destinations.
- Asymmetric flows (sign of MitM).

**Analyze NetFlow for:**
- Hosts communicating with the gateway at unusual volumes (DHCP flooding).
- Unexpected host-to-host traffic (VLAN hopping).
- Rapid scanning behavior (ARP scanning followed by targeted attacks).

### 17.5 Network TAP and Out-of-Band Analysis

Deploy **network TAPs** (rather than SPAN) for production traffic capture at strategic points:
- Core switch uplinks.
- Internet edge.
- Data center ingress.

TAPs provide 100% lossless capture without impacting production traffic, ideal for security analysis and incident response.

Forward captured traffic to an **IDS/IPS** (Snort, Suricata, Zeek):
- Write custom Layer 2 detection rules.
- Detect ARP anomalies, DHCP anomalies, STP manipulation.

## 18. Hardening Checklists

### 18.1 Layer 2 Security Audit Checklist

**VLAN Configuration:**
- [ ] VLAN 1 not used for user or management traffic
- [ ] Native VLAN set to unused, dedicated VLAN (not VLAN 1)
- [ ] Native VLAN tagged (`vlan dot1q tag native` or equivalent)
- [ ] VLANs pruned on all trunk links (only necessary VLANs allowed)
- [ ] VTP set to Transparent or Off mode
- [ ] VTP password configured if VTP Server/Client mode is used
- [ ] All trunk ports explicitly configured (`switchport mode trunk`)
- [ ] DTP disabled on all ports (`switchport nonegotiate`)

**Spanning Tree:**
- [ ] PortFast enabled on all access ports
- [ ] BPDU Guard enabled on all PortFast ports
- [ ] Root Guard enabled on downlink/access layer ports
- [ ] Loop Guard enabled on appropriate uplink ports
- [ ] Storm Control configured on all access ports
- [ ] Root Bridge placement is intentional and documented
- [ ] Appropriate STP variant in use (RSTP/Rapid PVST+ preferred)

**Port Security:**
- [ ] Port Security enabled on all access ports
- [ ] Maximum MAC addresses set appropriately (typically 1)
- [ ] Violation mode set to Restrict or Shutdown
- [ ] Sticky MAC learning enabled
- [ ] Unused ports shutdown and placed in black hole VLAN

**DHCP and ARP Security:**
- [ ] DHCP Snooping enabled on all access VLANs
- [ ] DHCP Snooping trust configured only on uplinks and DHCP server ports
- [ ] DHCP rate limiting configured on untrusted ports
- [ ] Dynamic ARP Inspection enabled on all access VLANs
- [ ] DAI trust configured on uplinks only
- [ ] ARP ACLs created for statically configured hosts
- [ ] IP Source Guard enabled on user access ports

**Discovery Protocols:**
- [ ] CDP disabled globally or limited to infrastructure links only
- [ ] LLDP disabled globally or limited to infrastructure links only
- [ ] CDP/LLDP disabled on all user-facing ports

**Management Plane:**
- [ ] Telnet disabled on all VTY lines
- [ ] SSH version 2 only
- [ ] RSA key 2048 bits or greater
- [ ] Login brute force protection configured
- [ ] Management access restricted via ACL
- [ ] SNMP v1/v2c community strings removed
- [ ] SNMPv3 with AuthPriv mode configured
- [ ] TACACS+/RADIUS for AAA
- [ ] Out-of-band management VLAN
- [ ] All unused services disabled
- [ ] Control Plane Policing (CoPP) configured

**Monitoring:**
- [ ] SYSLOG forwarding to centralized SIEM
- [ ] SNMP traps configured for security events
- [ ] NTP configured (accurate timestamps for logging)
- [ ] All security feature violations are logged
- [ ] Log retention policy defined and enforced

**Physical:**
- [ ] Unused switch ports physically secured (port locks)
- [ ] Switch in locked cabinet/rack
- [ ] Cable paths secured and documented
- [ ] No unauthorized devices connected
- [ ] Regular physical inspection schedule
