## 1. Foundational Concepts

### What is an Interrupt?

Normally, a CPU executes instructions **sequentially** — one after another in a program flow. But hardware devices (keyboard, disk, network card) operate at their own pace, independently of the CPU. The CPU cannot simply "wait" for them — that would waste billions of cycles.

The solution: **Interrupts**.

An interrupt is a **signal sent to the CPU** that says:
> "Stop what you are doing. Something important needs your attention right now. Handle it, then return to what you were doing."

Think of it like a surgeon performing an operation. A nurse (hardware device) suddenly says "emergency in room 5!" The surgeon pauses, handles the emergency, then returns to the original operation.

### Analogy: Polling vs Interrupts

```
POLLING (Inefficient):
  CPU: "Is keyboard ready?" → No
  CPU: "Is keyboard ready?" → No
  CPU: "Is keyboard ready?" → No
  CPU: "Is keyboard ready?" → Yes! Read key.
  (CPU wastes cycles asking)

INTERRUPT-DRIVEN (Efficient):
  CPU: doing useful work...
  Keyboard: "I have a key!" → sends IRQ signal
  CPU: pauses work, handles key, resumes.
  (CPU is never idle waiting)
```

### Types of Interrupts

There are THREE fundamental categories:

```
                    INTERRUPTS
                        |
        +---------------+---------------+
        |               |               |
   HARDWARE          SOFTWARE        EXCEPTIONS
  INTERRUPTS        INTERRUPTS      (CPU-Generated)
        |               |               |
   External        int instruction   Page Fault
   devices:         (syscalls)       Div by Zero
   - Keyboard        User mode →     GPF (General
   - NIC             Kernel mode     Protection)
   - Disk                            Breakpoint
   - Timer                           Invalid Opcode
```

**Hardware Interrupts (IRQs):** Signals from physical devices. These are ASYNCHRONOUS — they arrive at any time, unrelated to what the CPU is executing.

**Software Interrupts (Exceptions):** Generated by the CPU itself when executing instructions. These are SYNCHRONOUS — they occur at a specific point in the instruction stream.
- `int 0x80` — old Linux syscall mechanism
- `syscall` — modern x86-64 syscall instruction

**CPU Exceptions:** Generated by the CPU when something goes wrong (divide by zero, page fault, etc.).

### Key Terminology (Glossary)

| Term | Definition |
|------|-----------|
| **IRQ** | Interrupt ReQuest — a signal from hardware requesting CPU attention |
| **ISR** | Interrupt Service Routine — the function that handles the IRQ |
| **IDT** | Interrupt Descriptor Table — a lookup table of ISR addresses |
| **Vector** | A number (0–255 on x86) that identifies a specific interrupt type |
| **APIC** | Advanced Programmable Interrupt Controller — manages IRQ routing |
| **Interrupt Latency** | Time from IRQ signal to first ISR instruction executing |
| **IRQ Affinity** | Binding an IRQ to a specific CPU core |
| **Top Half** | The fast, immediate IRQ handler |
| **Bottom Half** | Deferred, slower processing after the IRQ |
| **Softirq** | Software interrupt — bottom-half mechanism |
| **Tasklet** | Single-CPU bottom-half mechanism built on softirqs |
| **Workqueue** | Process-context bottom-half mechanism |
| **Spurious IRQ** | A false interrupt with no actual device cause |
| **IRQ Storm** | Extremely high rate of IRQs that overwhelm the CPU |
| **NAPI** | Network API — adaptive polling for network cards |

## 2. Hardware Perspective

### How the CPU Processes an Interrupt

When hardware raises an IRQ signal, here is the EXACT sequence at the hardware level:

```
STEP-BY-STEP CPU INTERRUPT PROCESSING (x86-64)
================================================

1. Device asserts IRQ line (electrical signal)
         |
         v
2. Interrupt Controller (APIC) receives signal
         |
         v
3. APIC prioritizes (if multiple IRQs pending)
         |
         v
4. APIC sends interrupt vector number to CPU via APIC bus
         |
         v
5. CPU finishes CURRENT instruction (not current task!)
         |
         v
6. CPU saves state on KERNEL STACK:
        ┌─────────────────┐
        │   SS (old)      │  ← Stack Segment (if privilege change)
        │   RSP (old)     │  ← Old Stack Pointer
        │   RFLAGS        │  ← CPU Flags (contains IF - Interrupt Flag)
        │   CS            │  ← Code Segment
        │   RIP           │  ← Return Address (where to resume)
        └─────────────────┘
         |
         v
7. CPU clears IF flag (RFLAGS.IF = 0)
   → No further hardware interrupts can occur
         |
         v
8. CPU looks up vector number in IDT
         |
         v
9. CPU jumps to ISR address found in IDT
         |
         v
10. ISR executes (handles the interrupt)
         |
         v
11. ISR calls `iret` (Interrupt Return)
         |
         v
12. CPU restores saved state:
    - Pops RIP, CS, RFLAGS, RSP, SS from stack
    - Restoring RFLAGS restores IF flag
    - IF=1 → interrupts re-enabled
         |
         v
13. CPU resumes interrupted task at exact instruction
```

### The Interrupt Flag (IF)

The **IF (Interrupt Flag)** in the `RFLAGS` register controls whether the CPU accepts hardware interrupts:

```
RFLAGS Register (x86-64):
Bit 63                                    Bit 0
┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
│  │  │  │  │  │NT│IOPL│OF│DF│IF│TF│SF│ZF│  │PF│  │CF│
└──┴──┴──┴──┴──┴──┴────┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
                                  ↑
                        Interrupt Flag (bit 9)
                        IF=1: CPU accepts hardware IRQs
                        IF=0: CPU ignores hardware IRQs (CLI)

Instructions:
  CLI  → Clear IF → disable interrupts
  STI  → Set IF   → enable interrupts
  IRET → Restores RFLAGS (and therefore IF)
```

### Interrupt Priority

Hardware interrupts have priorities. If two IRQs arrive simultaneously, the higher-priority one is handled first. On x86:
- **NMI (Non-Maskable Interrupt)** — highest priority, cannot be disabled with CLI
- **External hardware IRQs** — normal priority
- **Software interrupts** — lowest priority

NMI is used for critical events like hardware failure, watchdog timers, and machine check exceptions.

## 3. Interrupt Controllers

### Historical Background: The 8259 PIC

Before modern systems, the **Intel 8259 Programmable Interrupt Controller (PIC)** was used. The PC/AT used TWO cascaded 8259 PICs (master + slave) providing 16 IRQ lines:

```
        ┌─────────────┐         ┌─────────────┐
IRQ0 ── │             │         │             │
IRQ1 ── │  MASTER PIC │         │  SLAVE PIC  │
IRQ2 ── │   (8259A)   │ ──IRQ2──│   (8259A)   │
IRQ3 ── │             │         │             │──IRQ8
IRQ4 ── │  I/O Port   │         │  I/O Port   │──IRQ9
IRQ5 ── │  0x20-0x21  │         │  0xA0-0xA1  │──IRQ10
IRQ6 ── │             │  INTR   │             │──IRQ11
IRQ7 ── │             │────────►│             │──IRQ12
        └─────────────┘  to CPU └─────────────┘──IRQ13
                                               ──IRQ14
                                               ──IRQ15

Classic IRQ Assignments:
  IRQ0  = System Timer (PIT)
  IRQ1  = Keyboard
  IRQ2  = Cascade (slave PIC)
  IRQ3  = COM2/COM4 (serial)
  IRQ4  = COM1/COM3 (serial)
  IRQ5  = LPT2 / Sound Card
  IRQ6  = Floppy Controller
  IRQ7  = LPT1 (parallel port)
  IRQ8  = RTC (Real Time Clock)
  IRQ9  = ACPI / Available
  IRQ10 = Available / NIC
  IRQ11 = Available / USB
  IRQ12 = PS/2 Mouse
  IRQ13 = FPU / Coprocessor
  IRQ14 = IDE Primary
  IRQ15 = IDE Secondary
```

The 8259 PIC is **obsolete** on modern systems but Linux still supports it for legacy hardware. Modern systems use APIC.

### The APIC — Advanced Programmable Interrupt Controller

Modern x86 systems use the **APIC architecture**, consisting of two components:

```
┌─────────────────────────────────────────────────────────┐
│                    SYSTEM ARCHITECTURE                   │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │  CPU 0   │  │  CPU 1   │  │  CPU 2   │  ...         │
│  │          │  │          │  │          │              │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │              │
│  │ │Local │ │  │ │Local │ │  │ │Local │ │              │
│  │ │ APIC │ │  │ │ APIC │ │  │ │ APIC │ │              │
│  │ └──┬───┘ │  │ └──┬───┘ │  │ └──┬───┘ │              │
│  └────┼─────┘  └────┼─────┘  └────┼─────┘              │
│       │             │             │                     │
│       └─────────────┴─────────────┘                     │
│                       │                                 │
│                  APIC Bus / System Bus                  │
│                       │                                 │
│               ┌───────┴──────┐                          │
│               │  I/O APIC    │                          │
│               │              │                          │
│               │ 24 IRQ lines │                          │
│               └──────┬───────┘                          │
│                      │                                  │
│         ┌────────────┼────────────┐                     │
│         │            │            │                     │
│       [NIC]       [Disk]      [USB]  ...                │
└─────────────────────────────────────────────────────────┘

LOCAL APIC (per CPU):
  - Receives interrupts directed to this CPU
  - Has a timer (LAPIC timer — used for scheduling)
  - Manages IPI (Inter-Processor Interrupts)
  - Has TPR (Task Priority Register) to control which IRQs to accept

I/O APIC:
  - Receives IRQs from external devices
  - Routes them to the appropriate Local APIC (and thus CPU)
  - Has a Redirection Table: maps IRQ line → vector + destination CPU
  - Supports 24 interrupt inputs (vs 16 for legacy PIC)
```

### The xAPIC and x2APIC

Modern CPUs use **x2APIC** (accessed via MSR registers instead of memory-mapped I/O), which supports more than 255 CPUs (old APIC limit).

### GIC — ARM Generic Interrupt Controller

For ARM systems (embedded Linux, servers, phones):

```
┌──────────────────────────────────────────────────────┐
│              ARM GIC Architecture (GICv3)            │
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │   CPU0   │  │   CPU1   │  │   CPU2   │           │
│  │          │  │          │  │          │           │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │           │
│  │ │  CPU │ │  │ │  CPU │ │  │ │  CPU │ │           │
│  │ │ I/F  │ │  │ │ I/F  │ │  │ │ I/F  │ │           │
│  │ └──┬───┘ │  │ └──┬───┘ │  │ └──┬───┘ │           │
│  └────┼─────┘  └────┼─────┘  └────┼─────┘           │
│       └─────────────┴─────────────┘                  │
│                       │                              │
│               ┌───────┴──────┐                       │
│               │ Redistributor│ (per CPU, GICv3)      │
│               └──────┬───────┘                       │
│                      │                               │
│               ┌───────┴──────┐                       │
│               │  Distributor │ (global)               │
│               └──────┬───────┘                       │
│                      │                               │
│  Interrupt Sources:                                  │
│  ┌──────────────────────────────────────┐            │
│  │ PPI (Private Peripheral Interrupt)   │ ← per CPU  │
│  │ SPI (Shared Peripheral Interrupt)    │ ← routable │
│  │ SGI (Software Generated Interrupt)   │ ← IPI      │
│  │ LPI (Locality-specific Peripheral)   │ ← MSI-like │
│  └──────────────────────────────────────┘            │
└──────────────────────────────────────────────────────┘

Interrupt Types in GIC:
  SGI  (0-15):   Software-generated, used for IPIs
  PPI (16-31):   Private per-CPU (e.g., local timer)
  SPI (32-1019): Shared, from external devices
  LPI (8192+):   Message-based (GICv3+)
```

## 4. Interrupt Descriptor Table (IDT)

### What is the IDT?

The **IDT (Interrupt Descriptor Table)** is a data structure that maps **interrupt vectors (0–255)** to their **handler functions**. It is the lookup table the CPU uses when an interrupt occurs.

```
IDT Structure (x86-64):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 Index  |  Interrupt Vector  |  Handler Address
        |                    |
  0     │  #DE Divide Error  │  → divide_error_handler
  1     │  #DB Debug         │  → debug_handler
  2     │  NMI               │  → nmi_handler
  3     │  #BP Breakpoint     │  → int3_handler
  4     │  #OF Overflow      │  → overflow_handler
  5     │  #BR Bound Range   │  → bounds_handler
  6     │  #UD Invalid Opcode│  → invalid_op_handler
  7     │  #NM Device N/A    │  → device_not_available
  8     │  #DF Double Fault  │  → double_fault_handler
  9     │  (reserved)        │
  10    │  #TS Invalid TSS   │  → invalid_tss_handler
  11    │  #NP Seg Not Pres. │  → segment_not_present
  12    │  #SS Stack Fault   │  → stack_segment_fault
  13    │  #GP General Prot. │  → general_protection_fault
  14    │  #PF Page Fault    │  → page_fault_handler
  15    │  (reserved)        │
  16    │  #MF FPU Error     │  → fpu_error_handler
  ...   │  ...               │  ...
  32–255│  External IRQs /   │  → hardware_irq_handlers
        │  Software Ints     │
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Each IDT entry (Gate Descriptor) is 16 bytes:
┌────────────────────────────────────────────────┐
│ Offset[63:32]  │ Reserved │ Offset[31:16]       │
│ Segment Selector (CS)     │ Offset[15:0]        │
│ Type/DPL/P    │ IST       │ (interrupt gate)    │
└────────────────────────────────────────────────┘

P   = Present (1 = valid entry)
DPL = Descriptor Privilege Level (0=kernel, 3=user)
IST = Interrupt Stack Table index
```

### IDT Registration in Linux

Linux sets up the IDT at boot time:

```c
// arch/x86/kernel/idt.c (simplified)
static const __initconst struct idt_data def_idts[] = {
    INTG(X86_TRAP_DE,      divide_error),      // vector 0
    INTG(X86_TRAP_NMI,     nmi),               // vector 2
    INTG(X86_TRAP_BP,      int3),              // vector 3
    INTG(X86_TRAP_PF,      page_fault),        // vector 14
    INTG(X86_TRAP_GP,      general_protection),// vector 13
    // ...
};
```

The `IDTR` register holds the **base address and limit** of the IDT:

```
IDTR Register:
┌──────────────────────────┬──────────┐
│     Base Address (64b)    │  Limit   │
└──────────────────────────┴──────────┘
        ↓
    Points to IDT in memory
    
Loaded with: lidt [idtr_value]
```

## 5. Linux IRQ Subsystem Architecture

### Overview

Linux has a **generic IRQ subsystem** (since kernel 2.6.18) that abstracts hardware-specific interrupt controllers behind a common interface. This is called the **IRQ domain** framework.

```
LINUX IRQ SUBSYSTEM LAYERED ARCHITECTURE
=========================================

User Space
──────────────────────────────────────────
  Applications, /proc/interrupts, irqbalance

──────────────────────────────────────────
Kernel Space

  ┌─────────────────────────────────────┐
  │         Device Drivers              │
  │  request_irq() / free_irq()         │
  │  irq_handler_t fn → your code       │
  └───────────────┬─────────────────────┘
                  │
  ┌───────────────▼─────────────────────┐
  │      Generic IRQ Layer              │
  │                                     │
  │  struct irq_desc[]  — IRQ descriptors│
  │  irq_chip_ops       — chip callbacks │
  │  irq_domain        — hw↔linux mapping│
  │  irqaction         — handler chain   │
  │  handle_level_irq() / handle_edge() │
  └───────────────┬─────────────────────┘
                  │
  ┌───────────────▼─────────────────────┐
  │   IRQ Controller Driver (irq_chip)  │
  │                                     │
  │  gic_irq_chip (ARM GIC)             │
  │  apic_chip    (x86 APIC)            │
  │  → mask/unmask/ack/eoi operations   │
  └───────────────┬─────────────────────┘
                  │
  ┌───────────────▼─────────────────────┐
  │   Hardware Interrupt Controller     │
  │   (Physical: APIC, GIC, PIC)        │
  └─────────────────────────────────────┘
```

### Key Data Structures

#### `struct irq_desc` — The IRQ Descriptor

This is the **heart** of the Linux IRQ system. Every IRQ has one:

```c
// include/linux/irqdesc.h (simplified)
struct irq_desc {
    struct irq_common_data   irq_common_data;  // shared data
    struct irq_data          irq_data;         // hw chip data
    unsigned int __percpu   *kstat_irqs;       // per-cpu stats
    irq_flow_handler_t       handle_irq;       // flow handler
    struct irqaction        *action;           // IRQ action list
    unsigned int             status_use_accessors; // IRQ status
    unsigned int             core_internal_state__do_not_mess_with_it;
    unsigned int             depth;            // disable nesting count
    unsigned int             wake_depth;       // wakeup enable count
    unsigned int             tot_count;        // total interrupt count
    unsigned int             irq_count;        // for detecting broken IRQs
    unsigned long            last_unhandled;   // time of last unhandled
    unsigned int             irqs_unhandled;   // count of unhandled
    atomic_t                 threads_handled;
    int                      threads_handled_last;
    raw_spinlock_t           lock;             // protects the descriptor
    struct cpumask           *percpu_enabled;
    const struct cpumask     *percpu_affinity;
    const struct cpumask     *affinity_hint;
    // ... more fields
};

// Array of IRQ descriptors:
extern struct irq_desc irq_desc[NR_IRQS]; // legacy
// OR dynamically allocated (sparse IRQ)
```

#### `struct irqaction` — The Action Chain

Each `irq_desc` has a linked list of `irqaction` — one per registered handler:

```c
// include/linux/interrupt.h (simplified)
struct irqaction {
    irq_handler_t       handler;    // your handler function
    void               *dev_id;     // unique ID (for shared IRQs)
    void __percpu      *percpu_dev_id;
    struct irqaction   *next;       // next handler (shared IRQs)
    irq_handler_t       thread_fn;  // threaded handler fn
    struct task_struct *thread;     // kthread for threaded IRQ
    struct irqaction   *secondary;  // secondary action
    unsigned int        irq;        // IRQ number
    unsigned int        flags;      // IRQF_* flags
    unsigned long       thread_flags;
    unsigned long       thread_mask;
    const char         *name;       // shown in /proc/interrupts
    struct proc_dir_entry *dir;
};
```

#### `struct irq_chip` — The Controller Interface

```c
// include/linux/irq.h (simplified)
struct irq_chip {
    struct device   *parent_device;
    const char      *name;
    void (*irq_startup)(struct irq_data *data);
    void (*irq_shutdown)(struct irq_data *data);
    void (*irq_enable)(struct irq_data *data);
    void (*irq_disable)(struct irq_data *data);
    void (*irq_ack)(struct irq_data *data);   // acknowledge
    void (*irq_mask)(struct irq_data *data);  // mask (disable)
    void (*irq_unmask)(struct irq_data *data);// unmask (enable)
    void (*irq_eoi)(struct irq_data *data);   // end-of-interrupt
    void (*irq_set_affinity)(struct irq_data *data,
                             const struct cpumask *dest, bool force);
    // ... more operations
};
```

### IRQ Domain — Hardware to Linux IRQ Mapping

Hardware interrupt controllers use their **own numbering** (hw IRQ numbers). Linux uses its own virtual IRQ numbers (virq / Linux IRQ number). The **IRQ domain** maps between them:

```
HARDWARE IRQ NUMBERS    IRQ DOMAIN       LINUX IRQ NUMBERS
                        (mapping)
  HW IRQ 0  ────────►  Domain Map  ──►  Linux IRQ 16
  HW IRQ 1  ────────►              ──►  Linux IRQ 17
  HW IRQ 2  ────────►              ──►  Linux IRQ 18
  ...                               ...

  irq_domain_add_linear()    → fixed size domain
  irq_domain_add_tree()      → sparse (large) domain
  irq_domain_add_hierarchy() → chained domains (MSI)
```

## 6. IRQ Lifecycle — Complete Flow

### From Hardware Signal to Handler Execution

```
COMPLETE IRQ LIFECYCLE
======================

[HARDWARE DEVICE]
      │  (1) Device generates interrupt signal
      │      (e.g., NIC received a packet)
      ▼
[I/O APIC]
      │  (2) APIC receives signal on IRQ line
      │  (3) Consults Redirection Table entry for this IRQ
      │      → Gets: vector number, destination CPU mask
      │  (4) Sends interrupt message to target CPU's Local APIC
      ▼
[LOCAL APIC (target CPU)]
      │  (5) Receives interrupt message
      │  (6) Checks IRR (Interrupt Request Register)
      │      and TPR (Task Priority Register)
      │  (7) If priority OK, signals CPU core
      ▼
[CPU CORE]
      │  (8) Finishes current instruction
      │  (9) Saves context on kernel stack (SS, RSP, RFLAGS, CS, RIP)
      │  (10) Clears IF flag (disables further hardware IRQs)
      │  (11) Looks up vector in IDT
      │  (12) Jumps to IDT entry's handler address
      ▼
[LINUX ENTRY ASSEMBLY (arch/x86/entry/entry_64.S)]
      │  (13) SWAPGS (switch to kernel GS base)
      │  (14) Push all general-purpose registers (save full context)
      │  (15) Build struct pt_regs on stack
      │  (16) Call common_interrupt() in C
      ▼
[common_interrupt() → handle_irq()]
      │  (17) Get IRQ number from vector
      │  (18) Get irq_desc for this IRQ number
      │  (19) Call irq_desc->handle_irq() [flow handler]
      │       e.g., handle_edge_irq() or handle_level_irq()
      ▼
[FLOW HANDLER: handle_edge_irq() / handle_level_irq()]
      │  (20) Lock irq_desc->lock (raw spinlock)
      │  (21) Call chip->irq_ack() → tell APIC IRQ received
      │  (22) Walk irq_desc->action linked list
      │  (23) Call each handler: action->handler(irq, dev_id)
      ▼
[YOUR DRIVER'S irq_handler_t]
      │  (24) Your handler runs
      │  (25) Returns IRQ_HANDLED or IRQ_NONE
      │       (IRQ_NONE if not your device — shared IRQ)
      ▼
[RETURN FROM HANDLER]
      │  (26) chip->irq_eoi() → send EOI to APIC
      │       (tells APIC this interrupt is done)
      │  (27) Unlock irq_desc->lock
      │  (28) Check for pending softirqs
      │  (29) Return to assembly exit code
      ▼
[LINUX EXIT ASSEMBLY]
      │  (30) Restore all registers from stack (RESTORE_REGS)
      │  (31) SWAPGS back (switch to user GS base)
      │  (32) iret (restores RIP, CS, RFLAGS, RSP, SS)
      │        → RFLAGS restoration re-enables IF flag
      ▼
[INTERRUPTED CODE RESUMES]
      (33) Program continues from exactly where it was

TOTAL LATENCY: typically 1–50 microseconds on modern hardware
```

### Level-Triggered vs Edge-Triggered

This is a fundamental concept in interrupt design:

```
EDGE-TRIGGERED:
  Signal: ____/‾‾‾‾‾‾\____
               ↑
          Interrupt fires here (on RISING EDGE only)

  - One interrupt per transition
  - If missed (CPU was busy), interrupt is LOST
  - Used by: PCI MSI, modern PCIe devices

LEVEL-TRIGGERED:
  Signal: ____/‾‾‾‾‾‾‾‾‾‾‾
               ↑↑↑↑↑↑↑↑↑↑
          Interrupt fires repeatedly while HIGH

  - Interrupt re-asserts if not cleared
  - Cannot be missed (keeps asserting)
  - Hardware must be cleared (de-assert line) in handler
  - Used by: legacy PCI, ISA devices

Decision in irq_desc->handle_irq:
  Edge  → handle_edge_irq()
  Level → handle_level_irq()
```

## 7. Interrupt Context vs Process Context

### Two Worlds in the Kernel

The Linux kernel code runs in two fundamentally different **contexts**:

```
┌─────────────────────────────────────────────────────────┐
│                    KERNEL CONTEXTS                       │
│                                                         │
│  ┌───────────────────┐    ┌───────────────────────┐    │
│  │  PROCESS CONTEXT  │    │  INTERRUPT CONTEXT    │    │
│  │                   │    │                       │    │
│  │ - Syscall handler │    │ - IRQ handler (top ½) │    │
│  │ - Kernel thread   │    │ - Softirq             │    │
│  │ - Workqueue worker│    │ - NMI handler         │    │
│  │                   │    │                       │    │
│  │ CAN:              │    │ CANNOT:               │    │
│  │ ✓ sleep/block     │    │ ✗ sleep or block      │    │
│  │ ✓ kmalloc(GFP_KERNEL)  │ ✗ kmalloc(GFP_KERNEL) │    │
│  │ ✓ mutex_lock()    │    │ ✗ mutex_lock()        │    │
│  │ ✓ schedule()      │    │ ✗ call schedule()     │    │
│  │ ✓ copy_to/from_user    │ ✗ copy_to/from_user   │    │
│  │ ✓ access current  │    │ ✗ access current      │    │
│  │   (meaningful)    │    │   (undefined!)        │    │
│  │                   │    │                       │    │
│  │ CAN USE:          │    │ MUST USE:             │    │
│  │ spin_lock()       │    │ spin_lock_irqsave()   │    │
│  │ mutex_lock()      │    │ atomic operations     │    │
│  │ semaphore         │    │ kmalloc(GFP_ATOMIC)   │    │
│  └───────────────────┘    └───────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### Detecting Interrupt Context

```c
// in_interrupt()   → true if in IRQ or softirq context
// in_irq()         → true if in hardware IRQ handler
// in_softirq()     → true if in softirq
// in_atomic()      → true if cannot sleep (includes interrupt context)

void my_function(void)
{
    if (in_interrupt()) {
        // Must use GFP_ATOMIC, cannot sleep
        ptr = kmalloc(size, GFP_ATOMIC);
    } else {
        // Can use GFP_KERNEL, can sleep
        ptr = kmalloc(size, GFP_KERNEL);
    }
}
```

### Why You Cannot Sleep in Interrupt Context

When the CPU is executing an IRQ handler:
1. It is running on a **specific CPU's interrupt stack** (a small, fixed-size stack)
2. The `current` pointer may point to a **random task** that happened to be running when the IRQ fired
3. Calling `schedule()` would try to save/restore the context of `current` — which is meaningless
4. **There is no "process" to sleep** — you are executing in the context of the interrupt machinery itself

This is one of the most important mental models in Linux kernel programming.

## 8. Top Half and Bottom Half

### The Problem of Long Handlers

An IRQ handler (top half) runs with **interrupts disabled** on that CPU. If it takes too long:
- Other interrupts are delayed (increased latency)
- The system becomes unresponsive
- **Hard real-time deadlines are missed**

The solution: **Split the work** into two halves.

```
TOP HALF vs BOTTOM HALF
=======================

HARDWARE IRQ FIRES
       │
       ▼
┌─────────────────────────────────────────┐
│           TOP HALF (ISR)                │
│                                         │
│  • Runs immediately                     │
│  • Interrupts disabled (on this CPU)    │
│  • MUST be fast (microseconds)          │
│  • Acknowledge the hardware             │
│  • Copy data from hardware registers    │
│  • Schedule bottom half                 │
│  • Return IRQ_HANDLED                   │
└─────────────────────┬───────────────────┘
                      │  "I'll finish later"
                      │  schedule_softirq() / tasklet_schedule()
                      │  queue_work()
                      ▼
┌─────────────────────────────────────────┐
│          BOTTOM HALF (Deferred)         │
│                                         │
│  • Runs "soon" — not immediately        │
│  • Interrupts RE-ENABLED                │
│  • Can take more time                   │
│  • Process the data                     │
│  • Allocate buffers                     │
│  • Notify upper layers                  │
│  • Workqueue: can even sleep!           │
└─────────────────────────────────────────┘

EXAMPLE: Network Card Receives Packet
  TOP HALF:    Copy packet from NIC FIFO → sk_buff
               Acknowledge NIC hardware
               Schedule NET_RX_SOFTIRQ
  BOTTOM HALF: Parse packet headers (IP, TCP)
               Route packet
               Wake up waiting socket
               Deliver to application
```

### Bottom Half Mechanisms — Comparison

```
┌──────────────┬────────────────┬──────────────┬───────────────┐
│   Mechanism  │   Context      │  Can Sleep?  │   Use Case    │
├──────────────┼────────────────┼──────────────┼───────────────┤
│  Softirq     │ Softirq context│     NO       │ High-perf net │
│              │ (interrupt ctx)│              │ timing-critical│
├──────────────┼────────────────┼──────────────┼───────────────┤
│  Tasklet     │ Softirq context│     NO       │ General async │
│              │ (interrupt ctx)│              │ driver work   │
├──────────────┼────────────────┼──────────────┼───────────────┤
│  Workqueue   │ Process context│     YES      │ Block I/O,    │
│              │ (kthread)      │              │ filesystem    │
├──────────────┼────────────────┼──────────────┼───────────────┤
│  Threaded IRQ│ Process context│     YES      │ Modern drivers│
│              │ (kthread)      │              │ (recommended) │
└──────────────┴────────────────┴──────────────┴───────────────┘
```

## 9. Softirqs

### What is a Softirq?

**Softirq** (Software Interrupt) is the fastest bottom-half mechanism. They are **statically defined** at compile time — you cannot add new softirq types without modifying the kernel. There are exactly 10 softirq types:

```c
// include/linux/interrupt.h
enum {
    HI_SOFTIRQ = 0,        // tasklets (high priority)
    TIMER_SOFTIRQ,          // kernel timers
    NET_TX_SOFTIRQ,         // network transmission
    NET_RX_SOFTIRQ,         // network receive (NAPI)
    BLOCK_SOFTIRQ,          // block device
    IRQ_POLL_SOFTIRQ,       // block device polling
    TASKLET_SOFTIRQ,        // tasklets (normal priority)
    SCHED_SOFTIRQ,          // scheduler
    HRTIMER_SOFTIRQ,        // high-resolution timers
    RCU_SOFTIRQ,            // RCU callbacks
    NR_SOFTIRQS             // count (10)
};
```

### Softirq Execution Points

Softirqs are executed at these points:
1. After hardware interrupt return (`irq_exit()`)
2. When `local_bh_enable()` is called
3. In the `ksoftirqd` kernel thread (when load is high)

### Softirq Internals

```c
// Each CPU has a bitmask of pending softirqs:
typedef struct {
    unsigned long __softirq_pending;  // 32-bit bitmask
} irq_cpustat_t;

// To raise a softirq:
raise_softirq(NET_RX_SOFTIRQ);  // sets bit in pending mask

// Softirq processing:
void __do_softirq(void)
{
    // Read pending bitmask
    // For each pending softirq (in priority order):
    //   Clear bit
    //   Enable interrupts (sti)
    //   Call softirq_vec[i].action()
    //   Disable interrupts (cli)
    // If more pending after N iterations → wake ksoftirqd
}
```

### ksoftirqd — The Softirq Kernel Thread

When softirq load is too high, Linux offloads to a **per-CPU kernel thread** called `ksoftirqd/N` (where N is CPU number):

```
$ ps aux | grep ksoftirqd
root         7  0.0  0.0      0     0 ?        S    ...  ksoftirqd/0
root        15  0.0  0.0      0     0 ?        S    ...  ksoftirqd/1
root        21  0.0  0.0      0     0 ?        S    ...  ksoftirqd/2
```

This prevents softirqs from monopolizing CPU time (they run as a schedulable task with low priority).

### Registering a Softirq

```c
// open_softirq() — done at kernel init time only
open_softirq(NET_RX_SOFTIRQ, net_rx_action);

// raise_softirq() — called from IRQ handler
raise_softirq(NET_RX_SOFTIRQ);

// Or from non-IRQ context:
raise_softirq_irqoff(NET_RX_SOFTIRQ); // if IRQs already off
```

**Key constraint:** The same softirq CAN run simultaneously on multiple CPUs. The action function must be **reentrant** or protect shared data with locks.

## 10. Tasklets

### What is a Tasklet?

A **tasklet** is a dynamic, single-CPU-serialized mechanism built ON TOP of softirqs (uses `TASKLET_SOFTIRQ` and `HI_SOFTIRQ`). Unlike softirqs, you can create tasklets dynamically in driver code.

**Critical property:** A given tasklet **never runs concurrently** — it runs on at most ONE CPU at a time. This makes tasklets easier to use than raw softirqs (no need for complex locking).

### Tasklet Data Structure

```c
// include/linux/interrupt.h
struct tasklet_struct {
    struct tasklet_struct *next;  // next in list
    unsigned long          state; // TASKLET_STATE_SCHED,
                                  // TASKLET_STATE_RUN
    atomic_t               count; // 0=enabled, >0=disabled
    bool                   use_callback;
    union {
        void (*func)(unsigned long);    // old API
        void (*callback)(struct tasklet_struct *); // new API
    };
    unsigned long          data;  // passed to func
};
```

### Tasklet API

```c
// DECLARE at compile time:
DECLARE_TASKLET(name, func);        // enabled
DECLARE_TASKLET_DISABLED(name, func); // disabled

// OR initialize at runtime:
void tasklet_init(struct tasklet_struct *t,
                  void (*func)(unsigned long),
                  unsigned long data);

// Schedule (from IRQ handler or anywhere):
tasklet_schedule(&my_tasklet);    // normal priority (TASKLET_SOFTIRQ)
tasklet_hi_schedule(&my_tasklet); // high priority (HI_SOFTIRQ)

// Enable/disable:
tasklet_disable(&my_tasklet); // waits if currently running
tasklet_enable(&my_tasklet);

// Kill (wait for completion, then prevent future runs):
tasklet_kill(&my_tasklet);    // must be called before freeing
```

### Tasklet State Machine

```
                      tasklet_schedule()
                             │
                             ▼
                    ┌────────────────┐
                    │    SCHEDULED   │
                    │  (state: SCHED)│
                    └───────┬────────┘
                            │  CPU picks up softirq
                            ▼
                    ┌────────────────┐
                    │    RUNNING     │
                    │  (state: RUN)  │
                    └───────┬────────┘
                            │  handler completes
                            ▼
                    ┌────────────────┐
                    │     IDLE       │
                    └────────────────┘
                            │
                      (if rescheduled during run)
                            │
                    ┌───────▼────────┐
                    │    SCHEDULED   │  ← queued again
                    └────────────────┘
```

### Tasklet Example in C

```c
#include <linux/interrupt.h>
#include <linux/module.h>

static struct tasklet_struct my_tasklet;

// Tasklet handler — runs in softirq context
static void my_tasklet_handler(unsigned long data)
{
    // 'data' is whatever you passed to tasklet_init
    printk(KERN_INFO "Tasklet running on CPU %d, data=%lu\n",
           smp_processor_id(), data);
    // DO: process data copied in IRQ handler
    // DON'T: sleep, call schedule(), use GFP_KERNEL
}

// IRQ handler — top half
static irqreturn_t my_irq_handler(int irq, void *dev_id)
{
    // Acknowledge hardware, copy minimal data
    // Schedule tasklet for deferred processing
    tasklet_schedule(&my_tasklet);
    return IRQ_HANDLED;
}

static int __init my_init(void)
{
    tasklet_init(&my_tasklet, my_tasklet_handler, 42UL);
    return request_irq(MY_IRQ, my_irq_handler,
                       IRQF_SHARED, "my_driver", &my_tasklet);
}

static void __exit my_exit(void)
{
    free_irq(MY_IRQ, &my_tasklet);
    tasklet_kill(&my_tasklet);  // CRITICAL: always kill before unloading
}
```

> **Note:** As of Linux 5.10+, tasklets are considered **legacy**. The kernel developers have been gradually deprecating them in favor of **threaded IRQs** and workqueues. New drivers should prefer `request_threaded_irq()`.

## 11. Workqueues

### What is a Workqueue?

A **workqueue** allows deferred work to run in **process context** — meaning it can sleep, allocate memory with `GFP_KERNEL`, call blocking operations, and use mutexes. This is implemented via kernel threads called **kworkers**.

```
WORKQUEUE ARCHITECTURE
======================

  IRQ Handler (top half)
       │
       │ queue_work(wq, &my_work)
       ▼
  ┌──────────────────────────────────────┐
  │         Work Queue                  │
  │                                     │
  │  [work_item] → [work_item] → ...    │
  └───────────────────┬──────────────────┘
                      │  kernel picks up work
                      ▼
  ┌──────────────────────────────────────┐
  │         kworker Thread              │
  │   (kernel/workqueue.c)              │
  │                                     │
  │  - Process context (CAN SLEEP!)     │
  │  - Calls work_struct.func()         │
  │  - Has full process capabilities    │
  └──────────────────────────────────────┘
```

### Work Queue Types

```
1. SYSTEM WORKQUEUE (system_wq):
   - Default, shared across kernel
   - use: schedule_work() / queue_work(system_wq, work)
   - Non-reentrant per work item

2. SYSTEM_HIGHPRI WORKQUEUE (system_highpri_wq):
   - Higher priority kworkers
   - For time-sensitive work

3. SYSTEM_LONG WORKQUEUE (system_long_wq):
   - For work that takes a long time
   - Prevents starvation of system_wq

4. SYSTEM_UNBOUND WORKQUEUE (system_unbound_wq):
   - Workers not bound to specific CPU
   - For CPU-agnostic work

5. CUSTOM WORKQUEUE:
   - Created with alloc_workqueue()
   - Your own thread pool
```

### Workqueue API

```c
#include <linux/workqueue.h>

// Define work:
struct work_struct my_work;

// Initialize:
INIT_WORK(&my_work, my_work_handler);

// Handler function:
static void my_work_handler(struct work_struct *work)
{
    // CAN SLEEP HERE
    // CAN USE GFP_KERNEL
    // CAN USE mutex_lock()
    struct my_device *dev = container_of(work,
                                         struct my_device,
                                         my_work);
    // process...
}

// Queue to system workqueue:
schedule_work(&my_work);                    // from any context
queue_work(system_wq, &my_work);            // explicit queue
queue_work_on(cpu, system_wq, &my_work);    // on specific CPU

// Queue with delay:
INIT_DELAYED_WORK(&my_delayed_work, handler);
schedule_delayed_work(&my_delayed_work, HZ); // after 1 second

// Cancel:
cancel_work_sync(&my_work);       // cancel and wait
cancel_delayed_work_sync(&my_delayed_work);

// Custom workqueue:
struct workqueue_struct *my_wq;
my_wq = alloc_workqueue("my_wq", WQ_MEM_RECLAIM, 1);
queue_work(my_wq, &my_work);
destroy_workqueue(my_wq);

// Flags for alloc_workqueue:
// WQ_HIGHPRI       - high priority workers
// WQ_CPU_INTENSIVE - CPU-bound work
// WQ_UNBOUND       - not bound to specific CPU
// WQ_MEM_RECLAIM   - can be used in memory reclaim path
// WQ_FREEZABLE     - freezable (suspend)
```

### `container_of` — Essential Kernel Pattern

In the workqueue handler, you often need to recover your device structure. The `container_of` macro does this:

```c
// container_of(ptr, type, member)
// Given ptr to a MEMBER of a struct,
// returns ptr to the CONTAINING struct

struct my_device {
    int id;
    struct work_struct work;   // embedded work_struct
    char *buffer;
};

static void handler(struct work_struct *work)
{
    // work points to my_device.work
    // container_of recovers my_device *
    struct my_device *dev = container_of(work,
                                          struct my_device,
                                          work);
    // Now use dev->id, dev->buffer, etc.
}

// How container_of works:
// Given: ptr to member
// Subtract the offset of member within struct
// Result: ptr to struct start

#define container_of(ptr, type, member) \
    (type *)((char *)(ptr) - offsetof(type, member))
```

## 12. Threaded IRQs

### Modern Preferred Approach

**Threaded IRQs** (introduced in Linux 2.6.30) are the **recommended modern mechanism** for drivers. They automatically split IRQ handling into:
- A **primary handler** (runs in hard IRQ context, very fast)
- A **thread handler** (runs in a dedicated kthread, can sleep)

```
request_threaded_irq() creates:
  ┌─────────────────────────────────────┐
  │   Hard IRQ Handler (primary)        │
  │                                     │
  │   - Runs in interrupt context       │
  │   - Just checks: is this my device? │
  │   - Returns IRQ_WAKE_THREAD         │
  └────────────────┬────────────────────┘
                   │  wakes up kthread
                   ▼
  ┌─────────────────────────────────────┐
  │   Thread Handler (irq/N-name)       │
  │                                     │
  │   - Runs as kthread (process ctx)   │
  │   - CAN SLEEP                       │
  │   - Full processing here            │
  └─────────────────────────────────────┘

  kthread name: irq/IRQ_NUM-DRIVER_NAME
  $ ps aux | grep "irq/"
    root  ... [irq/27-eth0]
    root  ... [irq/28-nvme0]
```

### Threaded IRQ API

```c
int request_threaded_irq(
    unsigned int irq,           // IRQ number
    irq_handler_t handler,      // primary handler (hard IRQ)
    irq_handler_t thread_fn,    // thread handler
    unsigned long irqflags,     // IRQF_* flags
    const char *devname,        // name for /proc/interrupts
    void *dev_id                // unique ID for free_irq
);

// Return values for primary handler:
// IRQ_NONE       → not my interrupt
// IRQ_HANDLED    → handled, no threading needed
// IRQ_WAKE_THREAD → wake the thread handler

// Return values for thread handler:
// IRQ_HANDLED    → done
// IRQ_NONE       → not handled

// If primary handler is NULL:
//   Linux uses default_primary_handler
//   which just returns IRQ_WAKE_THREAD always
```

### Example: Full Threaded IRQ Driver

```c
#include <linux/interrupt.h>
#include <linux/module.h>

struct my_device {
    int irq;
    void *dev_id;
    spinlock_t lock;
    u32 status_reg;
};

// Primary handler: fast, in hard IRQ context
static irqreturn_t primary_handler(int irq, void *dev_id)
{
    struct my_device *dev = dev_id;

    // Quick check: is this our interrupt?
    // (reads a status register)
    if (!(readl(dev->status_reg) & IRQ_STATUS_BIT))
        return IRQ_NONE;  // not ours (shared IRQ)

    // Acknowledge hardware immediately
    writel(IRQ_ACK, dev->status_reg);

    // Wake thread for full processing
    return IRQ_WAKE_THREAD;
}

// Thread handler: can sleep, in process context
static irqreturn_t thread_handler(int irq, void *dev_id)
{
    struct my_device *dev = dev_id;

    // CAN SLEEP — allocate memory, use mutexes
    u8 *buf = kmalloc(BIG_SIZE, GFP_KERNEL);
    if (!buf)
        return IRQ_HANDLED;

    // Do heavy processing
    process_data(dev, buf);
    kfree(buf);

    return IRQ_HANDLED;
}

static int my_probe(struct platform_device *pdev)
{
    struct my_device *dev = /* allocated */;

    return request_threaded_irq(
        dev->irq,
        primary_handler,    // fast handler
        thread_handler,     // thread handler
        IRQF_SHARED,        // flags
        "my_driver",        // name
        dev                 // dev_id
    );
}

static int my_remove(struct platform_device *pdev)
{
    struct my_device *dev = platform_get_drvdata(pdev);
    free_irq(dev->irq, dev);  // dev_id must match request_threaded_irq
    return 0;
}
```

## 13. IRQ Affinity — Binding Interrupts to CPUs

### What is IRQ Affinity?

By default, Linux tries to balance IRQs across CPUs. But you can **pin (bind) an IRQ to specific CPUs** for:
- **Performance** (cache locality — data stays warm on one CPU)
- **Real-time** (isolate latency-sensitive IRQs)
- **NUMA optimization** (process NIC IRQs on the NUMA node closest to the NIC)

### IRQ Affinity in /proc

```bash
# View affinity of IRQ 27 (hexadecimal CPU bitmask):
$ cat /proc/irq/27/smp_affinity
ff

# ff hex = 11111111 binary = CPUs 0,1,2,3,4,5,6,7 (all 8 CPUs)

# Set IRQ 27 to run ONLY on CPU 0:
$ echo 1 > /proc/irq/27/smp_affinity
# 1 hex = 00000001 binary = CPU 0 only

# Set IRQ 27 to run on CPUs 2 and 3:
$ echo c > /proc/irq/27/smp_affinity
# c hex = 00001100 binary = CPUs 2,3

# /proc/irq/N/smp_affinity_list — CPU list format (easier):
$ cat /proc/irq/27/smp_affinity_list
0-7

$ echo 2-3 > /proc/irq/27/smp_affinity_list  # CPUs 2,3
```

### Programmatic IRQ Affinity

```c
#include <linux/interrupt.h>
#include <linux/cpumask.h>

// Set affinity from kernel driver:
cpumask_t mask;
cpumask_clear(&mask);
cpumask_set_cpu(0, &mask);  // CPU 0 only

irq_set_affinity_hint(irq, &mask);
// or:
irq_set_affinity(irq, &mask);
```

### irqbalance Daemon

The **irqbalance** userspace daemon automatically distributes IRQs across CPUs based on load:

```bash
# Check if running:
$ systemctl status irqbalance

# View its decisions:
$ irqbalance --foreground --debug

# irqbalance policy hints (via /proc/irq/N/node):
# HINT_POLICY_NONE    — no hint, irqbalance decides
# HINT_POLICY_EXACT   — strict: must use affinity hint
# HINT_POLICY_SUBSET  — use hint as subset
```

## 14. Shared Interrupts

### What is a Shared IRQ?

Multiple devices can share the **same IRQ line**. When the interrupt fires, Linux calls **all registered handlers** in sequence. Each handler must determine if the interrupt was from **its device**.

```
IRQ LINE 10  ←── NIC
IRQ LINE 10  ←── Sound Card
IRQ LINE 10  ←── USB Controller
(all share IRQ 10 — legacy PCI)

When IRQ 10 fires:
  1. Call nic_handler()     → "Is it mine?" → NO  → IRQ_NONE
  2. Call sound_handler()   → "Is it mine?" → NO  → IRQ_NONE
  3. Call usb_handler()     → "Is it mine?" → YES → IRQ_HANDLED
  → IRQ was handled

If ALL handlers return IRQ_NONE → SPURIOUS IRQ
  → kernel logs: "irq 10: nobody cared"
  → after many spurious IRQs: kernel disables the line
```

### Shared IRQ Registration

```c
// IRQF_SHARED flag is REQUIRED for shared IRQs
// dev_id MUST be unique (non-NULL) — used to identify your handler

request_irq(IRQ_NUMBER,
            my_handler,
            IRQF_SHARED,   // REQUIRED
            "my_device",
            my_unique_dev_id);  // CANNOT be NULL for shared

// In handler, check hardware status to determine if it's yours:
static irqreturn_t my_handler(int irq, void *dev_id)
{
    struct my_dev *dev = dev_id;
    u32 status;

    status = ioread32(dev->base + STATUS_REG);
    if (!(status & MY_INTERRUPT_BIT))
        return IRQ_NONE;   // Not my interrupt

    // Handle interrupt...
    return IRQ_HANDLED;
}

// free_irq uses dev_id to find and remove YOUR handler only:
free_irq(IRQ_NUMBER, my_unique_dev_id);
```

## 16. Writing Interrupt Handlers in C

### Complete Driver Example

This is a **full, production-quality** Linux kernel module demonstrating IRQ handling:

```c
// my_irq_driver.c — Complete IRQ handling example
// Build: make -C /lib/modules/$(uname -r)/build M=$(pwd) modules

#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/init.h>
#include <linux/interrupt.h>
#include <linux/platform_device.h>
#include <linux/io.h>
#include <linux/workqueue.h>
#include <linux/slab.h>
#include <linux/spinlock.h>
#include <linux/atomic.h>

#define DRIVER_NAME     "my_irq_driver"
#define MY_IRQ_NUM      10          // IRQ number (example)
#define REG_STATUS      0x00        // Status register offset
#define REG_DATA        0x04        // Data register offset
#define REG_CONTROL     0x08        // Control register offset
#define STATUS_IRQ_BIT  BIT(0)      // Bit 0 = interrupt pending
#define CTRL_IRQ_ENABLE BIT(0)      // Bit 0 = enable IRQ

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("IRQ handling demonstration driver");
MODULE_VERSION("1.0");

// Per-device structure — holds all state
struct my_device {
    void __iomem       *base_addr;    // mapped register base
    int                 irq;          // IRQ number
    spinlock_t          lock;         // protects hw_buf, data_len
    u8                 *hw_buf;       // buffer for data from hardware
    size_t              data_len;     // how much data is in hw_buf
    struct work_struct  work;         // work item for bottom half
    struct workqueue_struct *wq;      // dedicated workqueue
    atomic_t            irq_count;   // total interrupts handled
    atomic_t            spurious_count; // spurious interrupts
    bool                irq_enabled;
};

// Forward declarations
static irqreturn_t my_irq_handler(int irq, void *dev_id);
static irqreturn_t my_thread_handler(int irq, void *dev_id);
static void my_work_fn(struct work_struct *work);

// ─────────────────────────────────────────────────────────
// HARDWARE REGISTER ACCESS HELPERS
// ─────────────────────────────────────────────────────────

static inline u32 hw_read(struct my_device *dev, u32 offset)
{
    return ioread32(dev->base_addr + offset);
}

static inline void hw_write(struct my_device *dev,
                             u32 offset, u32 value)
{
    iowrite32(value, dev->base_addr + offset);
}

// ─────────────────────────────────────────────────────────
// TOP HALF: Primary IRQ Handler
// Runs in hard interrupt context — CANNOT SLEEP
// Must be extremely fast
// ─────────────────────────────────────────────────────────

static irqreturn_t my_irq_handler(int irq, void *dev_id)
{
    struct my_device *dev = dev_id;
    u32 status;
    unsigned long flags;

    /*
     * Step 1: Check if this interrupt is from OUR hardware.
     * Critical for shared IRQs — return IRQ_NONE if not ours.
     */
    status = hw_read(dev, REG_STATUS);
    if (!(status & STATUS_IRQ_BIT)) {
        atomic_inc(&dev->spurious_count);
        return IRQ_NONE;  // Not our interrupt
    }

    /*
     * Step 2: Acknowledge the interrupt at hardware level.
     * This de-asserts the IRQ line BEFORE we process data.
     * Critical for level-triggered IRQs — must clear before return.
     */
    hw_write(dev, REG_STATUS, STATUS_IRQ_BIT);  // write-to-clear

    /*
     * Step 3: Copy minimal data from hardware registers.
     * Hardware registers may be overwritten by next interrupt!
     * Use spinlock to protect shared buffer.
     */
    spin_lock_irqsave(&dev->lock, flags);
    dev->data_len = hw_read(dev, REG_DATA);  // e.g., data length
    // In real driver: copy actual data from DMA buffer or FIFO
    spin_unlock_irqrestore(&dev->lock, flags);

    atomic_inc(&dev->irq_count);

    /*
     * Step 4: Schedule deferred work.
     * Return IRQ_WAKE_THREAD → kernel wakes thread_handler kthread.
     * OR: queue_work(dev->wq, &dev->work) and return IRQ_HANDLED.
     */
    return IRQ_WAKE_THREAD;  // wake thread handler
    /* Alternative:
     * queue_work(dev->wq, &dev->work);
     * return IRQ_HANDLED;
     */
}

// ─────────────────────────────────────────────────────────
// THREADED HANDLER: Runs in process context (kthread)
// CAN SLEEP, CAN USE GFP_KERNEL, CAN USE MUTEXES
// ─────────────────────────────────────────────────────────

static irqreturn_t my_thread_handler(int irq, void *dev_id)
{
    struct my_device *dev = dev_id;
    u8 *process_buf;
    size_t len;
    unsigned long flags;

    // Retrieve the data length saved by primary handler
    spin_lock_irqsave(&dev->lock, flags);
    len = dev->data_len;
    spin_unlock_irqrestore(&dev->lock, flags);

    if (!len)
        return IRQ_HANDLED;

    // CAN USE GFP_KERNEL here (can sleep)
    process_buf = kmalloc(len, GFP_KERNEL);
    if (!process_buf) {
        dev_err(NULL, "Failed to allocate process buffer\n");
        return IRQ_HANDLED;
    }

    // CAN call blocking functions here
    // Example: notify upper layers, write to ring buffer, etc.
    // msleep(10);  // would be OK here but not in irq handler!

    pr_info("Threaded handler: processed %zu bytes, "
            "total IRQs: %d\n",
            len, atomic_read(&dev->irq_count));

    kfree(process_buf);
    return IRQ_HANDLED;
}

// ─────────────────────────────────────────────────────────
// WORKQUEUE HANDLER (alternative to threaded IRQ)
// Also runs in process context — CAN SLEEP
// ─────────────────────────────────────────────────────────

static void my_work_fn(struct work_struct *work)
{
    struct my_device *dev = container_of(work,
                                          struct my_device,
                                          work);
    pr_info("Work handler running on CPU %d\n",
            smp_processor_id());
    // Heavy processing here
}

// ─────────────────────────────────────────────────────────
// DRIVER PROBE: Initialize and register IRQ
// ─────────────────────────────────────────────────────────

static int my_probe(struct platform_device *pdev)
{
    struct my_device *dev;
    struct resource *res;
    int ret;

    // Allocate device structure (managed — freed on unbind)
    dev = devm_kzalloc(&pdev->dev, sizeof(*dev), GFP_KERNEL);
    if (!dev)
        return -ENOMEM;

    platform_set_drvdata(pdev, dev);

    // Map hardware registers
    res = platform_get_resource(pdev, IORESOURCE_MEM, 0);
    dev->base_addr = devm_ioremap_resource(&pdev->dev, res);
    if (IS_ERR(dev->base_addr))
        return PTR_ERR(dev->base_addr);

    // Get IRQ number from platform data / device tree
    dev->irq = platform_get_irq(pdev, 0);
    if (dev->irq < 0)
        return dev->irq;

    // Initialize synchronization primitives
    spin_lock_init(&dev->lock);
    atomic_set(&dev->irq_count, 0);
    atomic_set(&dev->spurious_count, 0);

    // Initialize workqueue (if using WQ instead of threaded IRQ)
    INIT_WORK(&dev->work, my_work_fn);
    dev->wq = alloc_workqueue("my_irq_wq",
                               WQ_MEM_RECLAIM | WQ_HIGHPRI,
                               1);  // max 1 active worker
    if (!dev->wq)
        return -ENOMEM;

    // Register threaded IRQ handler
    ret = request_threaded_irq(
        dev->irq,
        my_irq_handler,       // primary: hard IRQ context
        my_thread_handler,    // thread: process context
        IRQF_SHARED |         // shared IRQ line
        IRQF_TRIGGER_RISING,  // edge triggered, rising edge
        DRIVER_NAME,          // name in /proc/interrupts
        dev                   // dev_id (must be unique, non-NULL)
    );
    if (ret) {
        dev_err(&pdev->dev, "Failed to request IRQ %d: %d\n",
                dev->irq, ret);
        destroy_workqueue(dev->wq);
        return ret;
    }

    // Enable IRQ in hardware
    hw_write(dev, REG_CONTROL, CTRL_IRQ_ENABLE);
    dev->irq_enabled = true;

    dev_info(&pdev->dev,
             "IRQ %d registered, handler at %p\n",
             dev->irq, my_irq_handler);
    return 0;
}

// ─────────────────────────────────────────────────────────
// DRIVER REMOVE: Clean up
// ─────────────────────────────────────────────────────────

static int my_remove(struct platform_device *pdev)
{
    struct my_device *dev = platform_get_drvdata(pdev);

    // Disable IRQ in hardware FIRST
    hw_write(dev, REG_CONTROL, 0);

    // Disable IRQ at kernel level
    disable_irq(dev->irq);

    // Free the IRQ handler
    // IMPORTANT: dev_id must match what was passed to request_*irq
    free_irq(dev->irq, dev);

    // Cancel any pending work and wait for it to complete
    cancel_work_sync(&dev->work);
    destroy_workqueue(dev->wq);

    pr_info("Total IRQs: %d, Spurious: %d\n",
            atomic_read(&dev->irq_count),
            atomic_read(&dev->spurious_count));

    return 0;
}

// ─────────────────────────────────────────────────────────
// IRQ ENABLE/DISABLE — Runtime Control
// ─────────────────────────────────────────────────────────

/*
 * disable_irq(irq)      — synchronously disable IRQ
 *                         waits for any running handler to complete
 * disable_irq_nosync(irq) — disable without waiting
 * enable_irq(irq)       — re-enable the IRQ
 *
 * local_irq_disable()   — disable ALL IRQs on THIS CPU (cli)
 * local_irq_enable()    — enable ALL IRQs on THIS CPU (sti)
 * local_irq_save(flags) — save flags and disable
 * local_irq_restore(flags) — restore from saved flags
 *
 * disable_irq()/enable_irq() are reference-counted:
 * disable_irq() increments depth
 * enable_irq()  decrements depth
 * IRQ fires only when depth == 0
 */

// Platform driver registration
static const struct of_device_id my_of_match[] = {
    { .compatible = "myvendor,my-irq-device" },
    { }
};
MODULE_DEVICE_TABLE(of, my_of_match);

static struct platform_driver my_driver = {
    .probe  = my_probe,
    .remove = my_remove,
    .driver = {
        .name           = DRIVER_NAME,
        .of_match_table = my_of_match,
    },
};

module_platform_driver(my_driver);
```

## 17. Writing Interrupt Handlers in Rust

### Rust in the Linux Kernel

As of Linux 6.1+, Rust is officially supported in the kernel. The `rust/kernel/` directory provides safe abstractions over kernel APIs.

> **Note:** Rust kernel development requires a kernel built with `CONFIG_RUST=y`. The APIs shown here reflect the Rust-for-Linux project abstractions.

### Key Concepts Before Writing Rust IRQ Code

**Concept: `irq::Handler` Trait**
Rust's kernel bindings use traits to represent the IRQ handler, providing type safety and preventing common bugs (like forgetting to return IRQ_HANDLED).

**Concept: `Registration` — RAII for IRQ**
Rust uses RAII (Resource Acquisition Is Initialization) — the IRQ is automatically freed when the `Registration` object is dropped. This eliminates a common C bug of forgetting to call `free_irq()`.

### Rust IRQ Handler Implementation

```rust
// my_irq_driver.rs — Rust IRQ handling for Linux kernel
// Requires: Linux kernel built with CONFIG_RUST=y

use kernel::prelude::*;
use kernel::{
    irq,
    sync::{Arc, Mutex, SpinLock},
    platform,
    io_mem::IoMem,
};

// ─────────────────────────────────────────────────────────
// MODULE DECLARATION
// ─────────────────────────────────────────────────────────

module! {
    type: MyIrqDriver,
    name: "my_irq_driver",
    author: "Your Name",
    description: "IRQ handling demonstration in Rust",
    license: "GPL",
}

// ─────────────────────────────────────────────────────────
// DEVICE STATE
// ─────────────────────────────────────────────────────────

/// Register offsets
const REG_STATUS: usize  = 0x00;
const REG_DATA: usize    = 0x04;
const REG_CONTROL: usize = 0x08;

/// Status register bits
const STATUS_IRQ_BIT: u32  = 1 << 0;
const CTRL_IRQ_ENABLE: u32 = 1 << 0;

/// Per-device state — shared between IRQ handler and main code
struct DeviceData {
    irq_count: u64,
    spurious_count: u64,
    data_len: u32,
}

/// The device structure
struct MyDevice {
    // SpinLock wraps data that needs protection in IRQ context
    data: SpinLock<DeviceData>,
    // IoMem provides safe, bounds-checked MMIO access
    regs: IoMem<0x100>,   // 256 bytes of registers
}

// ─────────────────────────────────────────────────────────
// IRQ HANDLER — implements irq::Handler trait
// ─────────────────────────────────────────────────────────

/// Implements the IRQ handler for MyDevice.
/// The trait ensures we return the correct type (irq::Return).
impl irq::Handler for MyDevice {
    type Data = Arc<MyDevice>;

    fn handle_irq(data: &Arc<MyDevice>) -> irq::Return {
        // SAFETY: We're in hard IRQ context.
        // SpinLock::lock() is safe here (it's IRQ-safe via
        // spin_lock_irqsave internally).

        // Step 1: Read status register — check if our interrupt
        let status = unsafe { data.regs.readl(REG_STATUS) };

        if status & STATUS_IRQ_BIT == 0 {
            // Not our interrupt (shared IRQ line)
            let mut d = data.data.lock();
            d.spurious_count += 1;
            return irq::Return::None;
        }

        // Step 2: Acknowledge hardware (write-to-clear)
        unsafe { data.regs.writel(STATUS_IRQ_BIT, REG_STATUS) };

        // Step 3: Copy minimal data
        let data_len = unsafe { data.regs.readl(REG_DATA) };

        {
            let mut d = data.data.lock();
            d.irq_count += 1;
            d.data_len = data_len;
        }

        // Step 4: Return Handled (or WakeThread for threaded)
        irq::Return::Handled
    }
}

// ─────────────────────────────────────────────────────────
// DRIVER STRUCT — module-level state
// ─────────────────────────────────────────────────────────

struct MyIrqDriver {
    // _irq_reg is the RAII guard — drop it to free the IRQ.
    // As long as this struct lives, the IRQ is registered.
    _irq_reg: irq::Registration<MyDevice>,
}

// ─────────────────────────────────────────────────────────
// MODULE INIT and EXIT
// ─────────────────────────────────────────────────────────

impl kernel::Module for MyIrqDriver {
    fn init(_module: &'static ThisModule) -> Result<Self> {
        pr_info!("my_irq_driver: initializing\n");

        // In a real driver, get IRQ number from device tree
        // or platform resources. Using 10 as example.
        let irq_num: u32 = 10;

        // Initialize device state
        let device_state = DeviceData {
            irq_count: 0,
            spurious_count: 0,
            data_len: 0,
        };

        // Create the device — Arc allows sharing with IRQ handler
        // SpinLock::new() initializes the spinlock
        let device = Arc::try_new(MyDevice {
            data: SpinLock::new(device_state),
            regs: unsafe {
                // SAFETY: caller must ensure this MMIO region is valid
                IoMem::try_new(0xDEAD_0000)?
            },
        })?;

        // Register IRQ handler
        // This is the RAII registration — freed when dropped
        let irq_reg = irq::Registration::try_new(
            irq_num,
            device.clone(),      // Arc clone — shared ownership
            irq::flags::SHARED,  // IRQF_SHARED equivalent
            c_str!("my_irq_driver"), // name for /proc/interrupts
        )?;

        pr_info!("my_irq_driver: IRQ {} registered\n", irq_num);

        Ok(MyIrqDriver {
            _irq_reg: irq_reg,
        })
    }
}

// When MyIrqDriver is dropped (module unload):
// _irq_reg.drop() → automatically calls free_irq()
// Rust's Drop trait handles this — you CANNOT forget to free!
impl Drop for MyIrqDriver {
    fn drop(&mut self) {
        pr_info!("my_irq_driver: unloading\n");
        // _irq_reg is dropped automatically → free_irq() called
        // No manual cleanup needed!
    }
}
```

### Rust Advantages in IRQ Code

```
SAFETY GUARANTEES FROM RUST'S TYPE SYSTEM:
============================================

1. CANNOT FORGET free_irq():
   Registration<T> implements Drop → auto-freed
   C: Easy to forget free_irq() in error paths

2. CANNOT USE WRONG dev_id:
   Type parameter <MyDevice> enforces correct type
   C: void* — any pointer accepted, easy to mismatch

3. CANNOT SLEEP IN IRQ CONTEXT:
   SpinLock in Rust disables preemption → prevents sleeping
   Rust's type system can encode "cannot block" via marker traits

4. DATA RACE PREVENTION:
   Arc<SpinLock<T>> → compiler enforces locking before access
   C: Manual discipline — easy to forget spin_lock()

5. MEMORY SAFETY:
   Arc reference counting → no use-after-free
   C: Manual lifetime management → frequent source of CVEs

EXAMPLE: The Rust compiler REJECTS this mistake:
  let data = device_data.lock();
  drop(data);  // explicit drop
  // Compiler error if you use 'data' after this line!
  // In C, you'd just have a use-after-free bug.
```

### Rust Atomic Operations for IRQ Counters

```rust

struct SafeCounters {
```

## 18. IRQ Flags and Spinlocks — Concurrency in Interrupt Context

### The Concurrency Problem

In an SMP (multi-CPU) system, multiple CPUs can run code simultaneously. When IRQ handlers access shared data, we need synchronization:

```
SCENARIO: CPU0 is updating shared_buffer
          CPU1 receives same IRQ, also tries to update shared_buffer
          → DATA CORRUPTION if not synchronized

SOLUTION: Spinlock
```

### Spinlock vs Mutex — Which to Use?

```
┌──────────────────┬──────────────────────┬────────────────────┐
│                  │    SPINLOCK          │    MUTEX           │
├──────────────────┼──────────────────────┼────────────────────┤
│ Waiting behavior │ Spin (busy-wait)     │ Sleep (block)      │
│ Interrupt context│ YES — safe to use    │ NO — cannot sleep  │
│ Hold duration    │ Very short (μs)      │ Can hold for ms    │
│ Overhead         │ Low (atomic ops)     │ Higher (scheduling)│
│ Preemption       │ Disabled while held  │ Enabled            │
└──────────────────┴──────────────────────┴────────────────────┘
```

### Spinlock Variants and When to Use Each

```c
// 1. Basic spinlock (process context only, no IRQ):
spin_lock(&lock);
// critical section
spin_unlock(&lock);

// 2. IRQ-disabling spinlock (process context + IRQ handler share data):
unsigned long flags;
spin_lock_irqsave(&lock, flags);    // saves and disables IRQs
// critical section — safe against IRQ preemption on THIS CPU
spin_unlock_irqrestore(&lock, flags); // restores IRQ state

// 3. Bottom-half disabling (process context + softirq share data):
spin_lock_bh(&lock);     // disables softirqs on this CPU
// critical section
spin_unlock_bh(&lock);

// WHEN TO USE WHICH:
//
// Process context only (never accessed from IRQ):
//   → mutex or spin_lock()
//
// Process context AND IRQ handler:
//   → spin_lock_irqsave() in BOTH places
//
// Process context AND softirq/tasklet:
//   → spin_lock_bh() in process context
//   → spin_lock() in softirq (softirq can't be preempted by softirq
//      on same CPU, but SMP still needs locking)
//
// IRQ handler AND softirq:
//   → spin_lock_irqsave() in IRQ handler
//   → spin_lock() in softirq (IRQ will disable if needed)
```

### Why `irqsave` and not just `irq_disable`?

```c
// WRONG — disables IRQs without saving previous state:
local_irq_disable();
spin_lock(&lock);
spin_unlock(&lock);
local_irq_enable();  // What if IRQs were already disabled before?
                     // This incorrectly RE-ENABLES them!

// CORRECT — saves and restores:
unsigned long flags;
spin_lock_irqsave(&lock, flags);
// ...
spin_unlock_irqrestore(&lock, flags);
// IRQ state restored to EXACTLY what it was before
// If they were disabled before → still disabled after
// If they were enabled before → re-enabled after
```

### Read-Write Spinlocks

When data is read frequently but written rarely:

```c
rwlock_t my_rwlock;
rwlock_init(&my_rwlock);

// Multiple readers simultaneously:
read_lock(&my_rwlock);       // or read_lock_irqsave()
// ... read shared data ...
read_unlock(&my_rwlock);

// Exclusive writer:
write_lock_irqsave(&my_rwlock, flags);
// ... modify shared data ...
write_unlock_irqrestore(&my_rwlock, flags);

// RCU (Read-Copy-Update) is even better for read-heavy workloads:
// rcu_read_lock() / rcu_read_unlock() — no locking, just memory barriers!
```

## 19. MSI and MSI-X — Message Signaled Interrupts

### What are MSIs?

**MSI (Message Signaled Interrupts)** is the modern interrupt mechanism for PCIe devices. Instead of using dedicated IRQ lines (wires), the device writes a **message** (a specific value to a specific memory address) to signal an interrupt.

```
LEGACY INTERRUPTS:
  Device → [IRQ wire] → APIC → CPU
  (shared wire, limited to ~16 IRQs, edge/level issues)

MSI INTERRUPTS:
  Device → [PCIe bus] → CPU writes message → Local APIC
  (no wire needed — message itself IS the interrupt)

  Message = Write to Local APIC address with vector number
  APIC sees this write → processes as interrupt

ADVANTAGES:
  ✓ No IRQ line sharing (each device gets unique vector)
  ✓ No spurious interrupts (write is atomic)
  ✓ Up to 32 vectors per device (MSI)
  ✓ Up to 2048 vectors per device (MSI-X)
  ✓ Per-vector CPU affinity
  ✓ Multi-queue NICs can use one vector per CPU → no lock contention
```

### MSI-X Table Structure

```
MSI-X CONFIGURATION:
====================

PCIe Configuration Space:
  ┌─────────────────────────────────────────┐
  │  MSI-X Capability:                      │
  │  - Table Size: N entries                │
  │  - Table Offset in BAR                  │
  │  - PBA (Pending Bit Array) Offset       │
  └──────────────────┬──────────────────────┘
                     │
  MSI-X Table (in BAR memory):
  ┌──────────────┬────────────┬──────────┬──────────┐
  │ Entry 0:     │            │          │          │
  │  Msg Addr   │  Msg Data  │  Masked? │  Vector 0 │
  ├──────────────┼────────────┼──────────┼──────────┤
  │ Entry 1:     │            │          │          │
  │  Msg Addr   │  Msg Data  │  Masked? │  Vector 1 │
  ├──────────────┼────────────┼──────────┼──────────┤
  │ ...          │            │          │          │
  └──────────────┴────────────┴──────────┴──────────┘

  Msg Addr = Local APIC address of target CPU
  Msg Data = Interrupt vector number
  → Device writes Data to Addr → CPU receives interrupt on that vector
```

### MSI-X API in Linux

```c
#include <linux/pci.h>

// Request MSI-X vectors for a PCI device:
struct msix_entry {
    u32 vector;   // filled by kernel (Linux IRQ number)
    u16 entry;    // user sets this (0..N-1)
};

struct msix_entry entries[4] = {
    { .entry = 0 },
    { .entry = 1 },
    { .entry = 2 },
    { .entry = 3 },
};

// Request 4 MSI-X vectors:
int nvec = pci_enable_msix_range(pdev, entries, 1, 4);
if (nvec < 0) {
    // Fall back to MSI or legacy IRQ
}

// Now entries[i].vector contains the Linux IRQ number
for (i = 0; i < nvec; i++) {
    request_irq(entries[i].vector, handler, 0,
                "my_nic_queue", &queues[i]);
}

// Multi-queue NIC example: one IRQ per CPU
// IRQ 0 → CPU 0 → RX queue 0
// IRQ 1 → CPU 1 → RX queue 1
// IRQ 2 → CPU 2 → RX queue 2
// → No cross-CPU locking needed for queue access!

// Clean up:
pci_disable_msix(pdev);
```

## 20. NAPI — Network Adaptive Polling

### The IRQ Storm Problem

A high-speed 10Gbps NIC can receive millions of packets per second. If each packet generates an IRQ → **IRQ storm** → CPU spends all time handling interrupts, no time for actual work.

### NAPI's Hybrid Approach

**NAPI (New API)** combines interrupt-driven and polling modes:

```
NAPI STATE MACHINE:
===================

          IDLE (interrupt mode)
               │
               │  Packet arrives → IRQ fires
               ▼
    ┌─────────────────────┐
    │  IRQ Handler runs:  │
    │  1. Disable IRQ     │  ← Stop more IRQs
    │  2. Schedule NAPI   │  ← Queue for softirq
    │  3. Return          │
    └──────────┬──────────┘
               │
               ▼
    ┌─────────────────────┐
    │  NET_RX_SOFTIRQ:    │
    │  Poll loop:         │
    │   while budget > 0: │
    │     rx = poll_func()│  ← napi->poll()
    │     budget -= rx    │
    │     if rx == 0:     │
    │       Re-enable IRQ │ ← Nothing left, back to interrupt mode
    │       return        │
    │   if budget == 0:   │
    │     Reschedule NAPI │ ← More work, keep polling
    └─────────────────────┘

KEY INSIGHT:
  When busy: POLL mode (no IRQ overhead, process many packets)
  When idle: INTERRUPT mode (instant notification, no wasted polling)
  → Best of both worlds!
```

### NAPI API

```c
// Register NAPI:
netif_napi_add(netdev, &nic->napi, my_poll_fn, NAPI_POLL_WEIGHT);
// NAPI_POLL_WEIGHT = 64 (default budget per poll call)

// In IRQ handler: disable IRQ and schedule NAPI
static irqreturn_t nic_irq_handler(int irq, void *dev_id)
{
    struct nic *nic = dev_id;

    // Disable further receive IRQs from this NIC
    nic_disable_rx_irq(nic);

    // Schedule NAPI poll in NET_RX_SOFTIRQ
    napi_schedule(&nic->napi);

    return IRQ_HANDLED;
}

// NAPI poll function — called by softirq with budget:
static int my_poll_fn(struct napi_struct *napi, int budget)
{
    struct nic *nic = container_of(napi, struct nic, napi);
    int work_done = 0;

    while (work_done < budget) {
        struct sk_buff *skb = nic_receive_packet(nic);
        if (!skb)
            break;  // no more packets

        netif_receive_skb(skb);  // deliver to network stack
        work_done++;
    }

    if (work_done < budget) {
        // We exhausted the packet queue before the budget
        // → Go back to interrupt mode
        napi_complete_done(napi, work_done);
        nic_enable_rx_irq(nic);  // re-enable NIC interrupt
    }
    // else: budget exhausted → will be rescheduled automatically

    return work_done;
}
```

## 23. Common Bugs and Pitfalls

### Bug 1: Sleeping in Interrupt Context

```c
// WRONG — mutex_lock can sleep!
static irqreturn_t bad_handler(int irq, void *dev_id)
{
    mutex_lock(&my_mutex);    // BUG! Can sleep in IRQ context
    do_something();
    mutex_unlock(&my_mutex);
    return IRQ_HANDLED;
}

// CORRECT — use spinlock
static irqreturn_t good_handler(int irq, void *dev_id)
{
    unsigned long flags;
    spin_lock_irqsave(&my_spinlock, flags);
    do_something_fast();
    spin_unlock_irqrestore(&my_spinlock, flags);
    return IRQ_HANDLED;
}
```

### Bug 2: Forgetting free_irq on Error Path

```c
// WRONG — resource leak if later init fails
static int init(void)
{
    request_irq(irq, handler, 0, "name", dev);
    ret = initialize_hardware();  // might fail!
    if (ret)
        return ret;  // IRQ was never freed! LEAK!
}

// CORRECT — cleanup on every error path
static int init(void)
{
    ret = request_irq(irq, handler, 0, "name", dev);
    if (ret)
        return ret;

    ret = initialize_hardware();
    if (ret) {
        free_irq(irq, dev);  // cleanup before returning error
        return ret;
    }
    return 0;
}
```

### Bug 3: NULL dev_id for Shared IRQ

```c
// WRONG — NULL dev_id with IRQF_SHARED
request_irq(irq, handler, IRQF_SHARED, "dev", NULL);
// Kernel will REJECT this! dev_id must be unique for shared IRQs

// CORRECT
request_irq(irq, handler, IRQF_SHARED, "dev", &my_device_struct);
```

### Bug 4: Accessing `current` in IRQ Context

```c
// WRONG — current is meaningless in IRQ context
static irqreturn_t bad_handler(int irq, void *dev_id)
{
    printk("IRQ for pid %d\n", current->pid);  // BUG! Random process!
    return IRQ_HANDLED;
}
// current points to whatever task was running when IRQ fired
// This is a random, unrelated process — not "your" process
```

### Bug 5: Not Using irqsave When Sharing Data with IRQ Handler

```c
struct my_data { int value; };
struct my_data shared;
spinlock_t lock;

// Process context:
void process_fn(void)
{
    spin_lock(&lock);        // WRONG if IRQ handler also uses this lock!
    shared.value = 42;       // IRQ can fire here, tries to acquire lock
    spin_unlock(&lock);      // → DEADLOCK on same CPU!
}

// CORRECT in process context (when sharing with IRQ):
void process_fn(void)
{
    unsigned long flags;
    spin_lock_irqsave(&lock, flags);   // disables IRQs on this CPU
    shared.value = 42;                 // IRQ cannot fire now
    spin_unlock_irqrestore(&lock, flags);
}
```

### Bug 6: tasklet_kill Not Called on Module Unload

```c
// WRONG — tasklet may still be scheduled when module code is gone!
static void __exit my_exit(void)
{
    free_irq(irq, dev);
    // tasklet not killed! If it runs now → BUG (code is gone!)
}

// CORRECT:
static void __exit my_exit(void)
{
    free_irq(irq, dev);     // stop new tasklets from being scheduled
    tasklet_kill(&my_tasklet); // wait for running tasklet, prevent future
}
```

### Bug 7: Interrupt Handler Too Slow → Watchdog

```c
// If your handler takes > softlockup_thresh seconds (default 20s):
// Kernel prints: "BUG: soft lockup - CPU#0 stuck for 22s!"
// Stack trace of the stuck IRQ handler is printed

// Solutions:
// 1. Move work to bottom half (tasklet/workqueue/threaded IRQ)
// 2. Split long loops into chunks
// 3. Use NAPI for network drivers
// 4. Profile with ftrace to find the bottleneck
```

## 24. Mental Models and Expert Intuition

### The Interrupt Mental Model

Think of the CPU as a **surgeon** and interrupts as **emergencies**:

```
SURGEON (CPU) performing operation (running user process):
  1. Normal work: cutting, suturing (executing instructions)
  2. Emergency signal (IRQ): nurse rushes in
  3. Surgeon finishes current stitch (current instruction)
  4. Surgeon saves position in operation log (saves RIP, RFLAGS)
  5. Surgeon handles emergency (ISR runs)
  6. Surgeon returns to exact saved position (IRET)
  7. Operation resumes

KEY INSIGHT: The patient (user process) never knows this happened.
Everything is saved and restored perfectly.
```

### The Stack Mental Model

```
STACK DURING IRQ HANDLING:

Before IRQ:
  [User Stack / Process Kernel Stack]
  ┌────────────────┐
  │   ...task...   │
  └────────────────┘

During IRQ (x86 switches to interrupt stack):
  [Interrupt Stack per CPU]
  ┌────────────────┐
  │    SS (old)    │  ─┐
  │    RSP (old)   │   │ Saved by CPU hardware
  │    RFLAGS      │   │ (IRET restores these)
  │    CS          │   │
  │    RIP (old)   │  ─┘
  │   [pt_regs]    │  ← Assembly saves all registers
  │  handler local │  ← ISR's local variables
  │    variables   │
  └────────────────┘

After IRET:
  [Back to User Stack / Process Kernel Stack]
  (interrupt stack is "unwound" — nothing left)

DANGER: Interrupt stack is SMALL (typically 4KB or 16KB)
  → Never do deep recursion in IRQ handlers
  → Never allocate large arrays on the IRQ stack
```

### Expert Pattern Recognition

When you see an IRQ-related performance problem, think in this order:

```
DIAGNOSTIC DECISION TREE:
==========================

Is IRQ rate too high?
├── YES: IRQ Storm
│   ├── Network? → Enable NAPI, set coalescing
│   ├── Timer? → Check if timer frequency is correct
│   └── Hardware fault? → Check spurious IRQ count
│
└── NO: Is IRQ latency too high?
    ├── Handler doing too much work?
    │   └── Move to bottom half (threaded IRQ/workqueue)
    ├── IRQ on wrong CPU? (cache miss)
    │   └── Set smp_affinity to NUMA-local CPU
    ├── IRQs disabled too long?
    │   └── Find long critical sections with ftrace
    └── Scheduler interference?
        └── Use isolcpus + set affinity
```

### Cognitive Principles for Mastering IRQs

**1. Chunking — Build Mental Layers**
```
Layer 1: Hardware signal → APIC → CPU
Layer 2: CPU saves state → IDT lookup → ISR
Layer 3: Linux IRQ subsystem (irq_desc, irqaction)
Layer 4: Top half/bottom half split
Layer 5: Performance optimization (affinity, NAPI, MSI-X)

Master each layer completely before adding the next.
Do not skip layers — each builds on the previous.
```

**2. Deliberate Practice — Active Debugging**
```
Don't just read — write drivers, break them, fix them.
Exercises:
  → Write a module that registers an IRQ handler and
    counts interrupts — verify with /proc/interrupts
  → Intentionally sleep in an IRQ handler → observe the BUG
  → Add a long loop in an IRQ handler → observe soft lockup
  → Trace with ftrace — see the exact call flow
```

**3. Systems Thinking — Understand Interactions**
```
IRQs don't exist in isolation. They interact with:
  → Scheduler (preemption, scheduling latency)
  → Memory allocator (GFP_ATOMIC limits)
  → Network stack (NAPI, sk_buff lifecycle)
  → Block layer (bio, request queues)
  → RCU (grace periods in softirq context)

Ask always: "What else is affected when an IRQ fires?"
```

**4. The Performance Mental Model**
```
Every interrupt has a COST:
  1. Context save: ~50–100 cycles
  2. IDT lookup + mode switch: ~100–200 cycles
  3. Cache pollution: 100–1000 cycles (most expensive!)
     (IRQ code and data evict your working set)
  4. Context restore + IRET: ~50–100 cycles

Total: ~300–1500 cycles per interrupt

At 10Gbps NIC: 14.88M small packets/sec
  × 1000 cycles = 14.88 BILLION cycles/sec
  = 14.88 GHz equivalent on one core!
  → Impossible without NAPI and interrupt coalescing
```

**5. The Safety Mental Model (for Rust)**
```
In C, IRQ bugs are among the most dangerous:
  - Use-after-free (handler fires after free_irq in wrong order)
  - Double-free (error paths skip free_irq)
  - Data races (spinlock not used consistently)
  - Priority inversion (mutex held across IRQ)

In Rust:
  - Registration<T> Drop prevents use-after-free
  - Arc prevents double-free
  - SpinLock<T> prevents data races at TYPE LEVEL
  - Compiler rejects incorrect usage at compile time

This is why Rust kernel drivers have fewer CVEs.
The type system encodes the safety invariants.
```
